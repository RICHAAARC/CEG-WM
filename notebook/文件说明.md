# notebook 组构建摘要

## 一、总体目标

拟构建一套面向论文级实验与最终发布的 `notebook/` 体系。该体系不应将所有任务堆叠在单一 notebook 中，而应按**任务语义与数据依赖关系**进行分层组织，以适配下列实际需求：

1. Colab 多会话执行；
2. 单次会话时长有限，需支持断点续跑；
3. 论文结果生产、机制验证、消融实验与最终发布应彼此解耦；
4. notebook 只负责流程编排，核心算法与正式逻辑尽量通过项目脚本或 CLI 调用；
5. 各阶段中间产物应标准化保存，以便不同 notebook 之间复用。

---

## 二、总体分层

建议将 `notebook/` 分为四层：

1. `core/`：项目论文机制主链路，`Full model` 作为正式基线；
2. `ablation/`：论文级消融实验层，细分为主结构消融与修复机制消融；
3. `eval/`：论文数字生产层，负责质量指标、攻击矩阵、`TPR@FPR` 与最终表格；
4. `release/`：最终可复现发布层，负责 repro bundle、审计与 signoff。

该分层的核心思想是：

* `core/` 负责证明“方法机制真实闭环成立”；
* `ablation/` 负责回答“各机制组件分别贡献了什么”；
* `eval/` 负责稳定、批量、可复用地产出论文主表与图；
* `release/` 负责将正式结果收口为可复现发布产物。

## 四、各层构建思路

### （一）`core/` 的构建思路

`core/` 是唯一的**机制主链路层**。
其职责不是生产全部论文数字，而是给出正式的 `Full model baseline` 与主流程闭环结果。

建议包含：

1. `00_Mini_Real_Validation.ipynb`
   用于在大规模正式运行前做小规模真实样本验证，确认正式路径前置条件成立。

2. `01_Paper_Full_Cuda.ipynb`
   作为 `Full model` 的正式主机制 notebook，负责执行完整链路并产出正式基线 run。

3. `02_Parallel_Attestation_Statistics.ipynb`
   基于 `01` 的正式输出，独立完成 attestation 统计，不重新跑主链。

4. `03_Experiment_Matrix_Full.ipynb`
   基于 `01` 的正式基线，独立执行攻击矩阵或 experiment matrix，不与主 notebook 耦合。

设计原则是：
**主链只保留真正必要的完整流程，其余 detached branch 独立拆出。**

---

### （二）`ablation/` 的构建思路

`ablation/` 的任务不是简单“关闭若干开关”，而是要严格区分**两类语义不同的消融**：

1. **主结构消融**：评估结构组件本身的贡献；
2. **修复机制消融**：评估 detect-side repair 或补偿机制的贡献。

因此建议：

#### 1. `00_Ablation_Manifest_And_Baseline.ipynb`

作为整个 `ablation/` 层的统一上游，负责：

* 固定消融用 prompt 与 seed；
* 冻结样本清单；
* 生成 ablation 专用 baseline；
* 保存可复用的基础产物。

#### 2. `structural/`

用于回答：

* `w/o Geometry`
* `w/o Anchor`
* `w/o HF`
* `w/o LF`
* 可选：`w/o Content`

其中应区分两类变量：

* **detect rerun 型变量**：可复用同一基础嵌入，仅改变 detect 侧配置；
* **re-embed 型变量**：会改变嵌入阶段，必须重新 embed 再 detect。

因此，建议拆为：

* `01_Geometry_And_Anchor_Ablation.ipynb`
* `02_LF_HF_Content_Ablation.ipynb`
* `03_Structural_Ablation_Tables.ipynb`

#### 3. `repair/`

用于回答：

* `GEO repair on/off`
* `LF exact repair on/off`

这类消融通常是 detect-side rerun，最适合在同一份基础产物上比较，因此建议拆为：

* `01_GEO_Repair_Ablation.ipynb`
* `02_LF_Exact_Repair_Ablation.ipynb`
* `03_Repair_Ablation_Tables.ipynb`

设计原则是：
**不要把“主结构移除”和“修复开关 on/off”混在同一张表里解释。**

---

### （三）`eval/` 的构建思路

`eval/` 不是机制验证层，而是**论文数字生产层**。
其目标是：在口径冻结的前提下，分阶段稳定产出论文中的质量指标、鲁棒性指标、攻击矩阵与最终表格。

建议包含七个 notebook：

1. `00_Manifest_And_Splits.ipynb`
   冻结评测用 prompt、seed、sample_id、split 与 attack plan。

2. `01_Clean_Generation.ipynb`
   生成 `baseline clean images` 与 `watermarked clean images`。

3. `02_Readonly_Calibration.ipynb`
   基于 calibration split 做 detect 打分并生成只读 threshold。

4. `03_Quality_Metrics.ipynb`
   计算 LPIPS、SSIM、FID 等质量指标。

5. `04_Attack_Generation.ipynb`
   对 clean watermarked images 施加攻击并生成 attacked images。

6. `05_Detect_And_TPR.ipynb`
   对 clean 与 attacked images 做 detect，并在只读阈值下计算 `TPR@FPR`。

7. `06_Tables_And_Figures.ipynb`
   汇总质量指标与检测指标，生成论文主表、攻击矩阵表、图与 LaTeX 表。

设计原则是：
**按数据生命周期拆 notebook，而不是按指标名称逐个切碎。**

---

### （四）`release/` 的构建思路

`release/` 的目标是：将前述正式结果组织成最终可复现发布产物。

建议包含：

1. `00_Repro_Bundle.ipynb`
   从正式主 run 出发，收集 configs、records、artifacts，生成 repro bundle。

2. `01_All_Audits.ipynb`
   对 repro bundle 与正式输出执行审计与一致性检查。

3. `02_Publish_And_Signoff.ipynb`
   完成最终 signoff、发布摘要与最终可交付包。

设计原则是：
**发布链不参与调参与实验探索，只服务于最终结果封装。**

## 五、关键依赖关系

### （一）`core/` 与其他层

* `core/01_Paper_Full_Cuda.ipynb` 的正式基线输出，是整个体系中最重要的项目级上游。
* `core/02` 与 `core/03` 均应消费 `core/01` 的产物，而不是重新跑完整链路。
* `release/` 层应以 `core/01` 的正式 run 为主要输入。
* `ablation/` 与 `eval/` 不应直接依赖 `core/01` 的全部过程，而应各自维护自己的 manifest 与评测口径；但其汇总表中可以引用 `core/01` 的 full baseline 行。

---

### （二）`ablation/` 内部

* `ablation/00_Ablation_Manifest_And_Baseline.ipynb` 是整个 ablation 层的共享上游。
* `structural/01` 与 `repair/01`、`repair/02` 可以直接复用 `ablation/00` 的基础嵌入产物。
* `structural/02` 对于会影响 embed 的变量，不应直接复用相同嵌入，而应在固定 manifest 下重新 embed。
* `structural/03` 与 `repair/03` 只做汇总，不做重型生成。

---

### （三）`eval/` 内部

建议的依赖链如下：

1. `eval/00` → `eval/01`
2. `eval/00` → `eval/02`
3. `eval/01` → `eval/03`
4. `eval/01` → `eval/04`
5. `eval/02 + eval/01 + eval/04` → `eval/05`
6. `eval/03 + eval/05` → `eval/06`

其中必须强调：

* `eval/05` 只能读取 `eval/02` 的只读阈值，不允许重估阈值；
* `eval/04` 的攻击图像必须来自 `eval/01` 的 clean watermarked images；
* `eval/03` 与 `eval/04` 在 `eval/01` 完成后可以并行。

---

### （四）`release/` 层

* `release/00` 以正式主结果为输入，生成 repro bundle；
* `release/01` 对 bundle 与正式产物做审计；
* `release/02` 基于前两者完成最终 signoff 与发布。

### 第一阶段：机制主链

## 七、构建 notebook 组时的统一原则

为便于新会话继续讨论，可将以下原则作为 notebook 组构建的固定前提：

1. notebook 只负责流程编排，不承载项目核心机制实现；
2. 所有 notebook 都应支持断点续跑与中间产物落盘；
3. 每一层都应有自己的 manifest 或 baseline，不依赖隐式状态；
4. `core/` 解决“机制是否成立”，`eval/` 解决“论文数字如何产出”，二者不可混淆；
5. `ablation/` 必须区分 detect-rerun 型变量与 re-embed 型变量；
6. `release/` 只处理正式结果封装，不参与探索性实验；
7. 同一指标体系下，应共享同一组样本定义、配置摘要与阈值来源；
8. 不建议“一个指标一个 notebook”，应按数据依赖阶段拆分 notebook。

---

## 八、一句话总结

这套 `notebook/` 的构建思路可以概括为：

**以 `core/` 固定 full baseline 与正式机制闭环，以 `ablation/` 拆解结构贡献与修复贡献，以 `eval/` 批量稳定地产出论文表格与图，以 `release/` 完成最终可复现发布；整个体系通过标准化 manifest、阶段化中间产物和只读阈值机制实现多会话 Colab 下的可复用编排。**

如果你愿意，在新会话中可以直接基于这份摘要，继续细化以下任一方向：
一是 `notebook/README.md` 的完整文本；二是每个 notebook 的 cell 级模板；三是各 notebook 的输入输出文件契约。
