# 2030修复方案

**版本：** 2026-03-06-2030-final  
**适用范围：** `configs/paper_full_cuda.yaml` + `main/`（核心发布范围）；`scripts/` 仅用于 workflow 补强，不作为核心判决链事实源。  
**目标：** 在不改变既有冻结语义、不引入新写盘旁路、不破坏现有 workflow 顺序的前提下，补齐 detect 侧稳定输出口径、LF 证据写盘口径与 BP 收敛诊断可追溯性，并为攻击覆盖与可移植性提供可执行补强方案。

---

## 一、修复目标

本次修复聚焦以下 6 项目标。

1. 为 detect record 提供稳定、可审计的最终判决快照 `final_decision`。
2. 补齐 `content_evidence_payload.score_parts.lf_status` 顶层口径，消除 LF/HF 写盘不对称。
3. 为 LF PRC 检测链补充 BP 收敛诊断字段 `bp_converge_status`，并保证 `lf_trace_digest` 可复算性不受影响。
4. 为 surrogate 模式下的 `detect_hf_score` 缺失补充显式原因字段。
5. 在 schema 中完成新增字段登记，保持 append-only 与 `absent_ok` 向后兼容。
6. 在脚本层补齐攻击覆盖与运行环境可移植性补强，但不改变核心判决链的既有结构。

---

## 二、执行原则

### 2.1 冻结语义保持不变

以下语义面保持不变：

1. `FusionDecision.decision_status` 仍为 `decided / abstain / error`。
2. `FusionDecision.is_watermarked` 仍遵循现有约束：
   - `decision_status=decided` 时必须为布尔值。
   - `decision_status=abstain` 或 `error` 时必须为 `null`。
3. `fusion_result` 原结构保持原样保留，不做替换、不做裁剪。
4. `low_freq_coder` 现有 `trace["status"]` 保持不变，不直接改成 `degraded`。
5. 不修改当前 onefile / repro workflow 的主顺序：`embed -> detect -> calibrate -> evaluate`。

### 2.2 写盘路径保持单一收口

所有新增字段均通过现有 orchestrator record 组装路径写入，不新增独立写盘入口，不绕过 `records_io`、`path_policy`、`freeze_gate`。

### 2.3 新增字段全部 append-only

本次新增字段全部采用 append-only 策略，旧 records 缺失这些字段时视为 `absent_ok`，不得影响旧产物读取与审计。

## 三、核心修复项

### 3.1 P0：detect 顶层稳定判决快照

**修改文件：** `main/watermarking/detect/orchestrator.py`

在 `run_detect_orchestrator()` 末尾组装 detect record 时，新增顶层字段 `final_decision`。

#### 字段结构

`final_decision` 固定包含以下字段：

1. `decision_status`
2. `is_watermarked`
3. `routing_decisions`
4. `threshold_source`

#### 字段来源

全部字段均从 `fusion_result` 只读投影获得：

1. `decision_status` 来自 `getattr(fusion_result, "decision_status", None)`。
2. `is_watermarked` 来自 `getattr(fusion_result, "is_watermarked", None)`。
3. `routing_decisions` 来自 `getattr(fusion_result, "routing_decisions", None)`。
4. `threshold_source` 来自 `getattr(fusion_result, "audit", {}).get("threshold_source")`。

#### 约束

1. `fusion_result` 顶层字段继续保留，不得用 `final_decision` 替换。
2. `final_decision` 仅作为稳定快照，不承载新的判决逻辑。
3. 若 `fusion_result` 缺失或字段不可读，`final_decision` 可写为 `null`，但不得抛出异常阻断写盘。

#### 预期产物结果

detect record 将新增：

- `final_decision.decision_status`
- `final_decision.is_watermarked`
- `final_decision.routing_decisions`
- `final_decision.threshold_source`

### 3.2 P0：补齐 LF 顶层状态口径

**修改文件：** `main/watermarking/detect/orchestrator.py`

在 `_bind_raw_scores_to_content_payload()` 中，保留现有：

1. `score_parts["lf_detect_trace"]`
2. `score_parts["prc_latent_status"]`

并追加补齐：

1. 当 `score_parts` 中不存在 `lf_status` 时，若 `prc_latent_status` 为非空字符串，则写入 `score_parts["lf_status"] = prc_latent_status`。

#### 约束

1. **必须使用 if-not-in 守卫**，不得覆写 `UnifiedContentExtractor` 已写入的 `lf_status`。
2. `score_parts["lf_status"]` 的值来源仅为当前 LF trace 的 `lf_status` 镜像，不新增独立来源。
3. 现有 `prc_latent_status` 与 `lf_detect_trace` 保持不变。

#### 预期产物结果

在当前生产路径下，detect record 的 `content_evidence_payload.score_parts.lf_status` 将从缺失变为存在，且与 `prc_latent_status` 保持一致。

---

### 3.3 P1：BP 收敛状态诊断字段

**修改文件：** `main/watermarking/content_chain/low_freq_coder.py`

在 `LFCoderPRC.detect_score()` 生成 LF trace 时，新增字段：

1. `bp_converge_status`

#### 字段语义

1. `bp_converged=True` 时，`bp_converge_status="ok"`。
2. `bp_converged=False` 时，`bp_converge_status="degraded"`。

#### 写入顺序要求

必须先计算：

1. `trace["lf_trace_digest"] = digests.canonical_sha256(trace)`

再追加：

1. `trace["bp_converge_status"] = ...`

#### 约束

1. 不修改现有 `trace["status"]`。
2. 不修改现有 `bp_converged`、`bp_iteration_count`、`parity_check_digest` 等字段。
3. `bp_converge_status` 不参与 `lf_trace_digest` 输入域。
4. 该字段仅用于诊断与审计，不参与内容判决逻辑。

#### 预期产物结果

LF trace 将具备显式的 BP 收敛语义，可在审计时区分：

1. PRC 检测链运行成功但 BP 未收敛。
2. PRC 检测链运行成功且 BP 已收敛。

---

### 3.4 P1：orchestrator 透传 BP 收敛状态

**修改文件：** `main/watermarking/detect/orchestrator.py`

在 LF PRC 分支将 `prc_trace_payload` 投影到 `lf_trace` 时，显式补入：

1. `bp_converge_status`

#### 约束

1. 该字段为 append-only 诊断字段，允许缺失为 `null`。
2. 若 coder 尚未输出该字段，`orchestrator` 必须通过 `.get()` 安全读取，不得 raise。

#### 预期产物结果

`content_evidence_payload.score_parts.lf_detect_trace.bp_converge_status` 将稳定出现在 detect record 中，不再被中间投影静默丢弃。

---

### 3.5 P1：HF 缺失原因显式化

**修改文件：** `main/watermarking/detect/orchestrator.py`

在 detect 侧同构评分写入段，保留现有：

1. `content_evidence_payload["detect_lf_score"]`
2. `content_evidence_payload["detect_hf_score"]`

并新增：

1. 当 `hf_basis is None`（即 `_plan_inner_dict.get("hf_basis")` 返回 `None`，导致 `extract_hf_score_from_detect_latents` 返回 `(None, "absent")`）时，写入 `content_evidence_payload["detect_hf_score_absent_reason"] = "hf_basis_not_computed_in_surrogate_mode"`。

> **说明：** `hf_basis is None` 是 surrogate 路径的充分判断条件，代码中无独立的"surrogate 路径标志"变量；以 `hf_basis is None` 作为写入守卫，与 `orchestrator.py` 实际变量名一一对应。`detect_hf_score is None` 是 `hf_basis is None` 的结果，两者在逻辑上等价，但守卫应置于 `hf_basis` 判断处，不对函数返回值做二次推断。

#### 约束

1. 该字段仅用于诊断，不改变 `detect_hf_score` 现有语义。
2. 不改变 `hf_score` 或 `score_parts.hf_status` 的现有生成逻辑。
3. 写入守卫为 `if hf_basis is None`，不依赖 `detect_hf_score is None` 的二次判断。

## 五、脚本层补强项

### 5.1 攻击覆盖补强

**修改文件：** `scripts/run_onefile_workflow.py`

目标是在 workflow 的 detect 阶段前，对同一水印图像生成：

1. identity baseline
2. attack_protocol 中定义的真实攻击变体

并确保 detect record 中的攻击条件字段不再退化为 `unknown_attack::unknown_params`。

#### 实施要求

1. 攻击参数必须来自 `configs/attack_protocol.yaml`，不得硬编码。
2. 攻击必须在 detect 前应用到图像域输入，不允许在 detect record 写出后补写标签。
3. 每个攻击样本必须携带明确的 `attack_family` 与 `attack_params_version`。
4. identity baseline 也必须使用显式标签，不得继续使用 unknown 占位。

#### 输出目标

`evaluate_record.metrics_by_attack_condition[*].group_key` 应出现真实攻击分组，例如：

1. `none::v1`
2. `rotate::v1`
3. `jpeg::v1`
4. `gaussian_noise::v1`

### 5.2 模型路径可移植性补强

**修改文件：**

1. `configs/paper_full_cuda.yaml`
2. `main/watermarking/content_chain/semantic_mask_provider.py`

#### 目标

将 `mask.semantic_model_path` 的解析语义统一到工程根目录或受控绝对路径，消除对调用时 CWD 的隐式依赖。

#### 实施方案

**`main/watermarking/content_chain/semantic_mask_provider.py` 改动：**

在 `_probe_model_v2_availability()` 中，对 `semantic_model_path` 进行路径解析时，若路径为非绝对路径，则以 `semantic_mask_provider.py` 文件所在目录向上数 3 级（`Path(__file__).resolve().parents[2]`，即工程根 `CEG-WM/`）作为基准目录进行拼接，而非依赖调用时的 CWD：

```python
raw_path = Path(mask_params["semantic_model_path"])
if not raw_path.is_absolute():
    project_root = Path(__file__).resolve().parents[2]
    model_path = project_root / raw_path
else:
    model_path = raw_path
```

**`configs/paper_full_cuda.yaml` 改动：**

将 `mask.semantic_model_path` 从 `"../models/inspyrenet/ckpt_base.pth"` 改为 `"models/inspyrenet/ckpt_base.pth"`（相对工程根，去除 `../` 前缀），与上述解析基准对齐。

#### 约束

1. 配置中不得继续依赖不稳定的 `../` 相对路径解释。
2. 路径解析基准固定为**工程根目录**（`semantic_mask_provider.py` 的 `parents[2]`），不依赖 CWD。
3. 绝对路径不受影响，解析逻辑仅对相对路径生效。
4. 不在 `_probe_model_v2_availability()` 以外的业务逻辑中散落路径拼接。

## 六、本版不纳入的改动

以下内容不纳入本次核心修复面。

1. 不直接将 detect 阶段的 fallback 判决改造为 NP canonical 判决。当前 workflow 顺序决定 detect 阶段在 calibrate 前运行；若要彻底消除 detect fallback，需要单独设计 detect-after-calibrate 路径或调整主 workflow 顺序。
2. 不直接把 LF trace 的 `status` 改成 `degraded`。当前 `UnifiedContentExtractor` 对非 `ok` LF 状态有明确传播逻辑，直接改状态会造成连锁语义变化。
3. 不把 FPR 收敛问题作为代码层修复项。`target_fpr=0.01` 下要获得更稳健的经验 FPR 控制，负样本规模需提升到至少 100；这属于实验规模与运行成本问题。
4. 不扩展 `FusionDecision.decision_status` 枚举，不新增 `rejected_insufficient_evidence`。

## 九、验收标准

本次修复完成后，应满足以下验收断言。

### 9.1 detect 判决快照

1. detect record 顶层存在 `final_decision`。
2. `final_decision.decision_status` 与 `fusion_result.decision_status` 一致。
3. `final_decision.is_watermarked` 在 `decided` 时为布尔值，在 `abstain/error` 时为 `null`。
4. `final_decision.threshold_source` 与 `fusion_result.audit.threshold_source` 一致。

### 9.2 LF 顶层口径

1. `content_evidence_payload.score_parts.lf_status` 在当前生产路径下存在。
2. 若 `score_parts` 已有 `lf_status`，新逻辑不得覆写原值。
3. `score_parts.prc_latent_status` 继续存在。

### 9.3 BP 收敛诊断

1. `content_evidence_payload.score_parts.lf_detect_trace.bp_converge_status` 存在。
2. 当 `bp_converged=False` 时，该字段为 `degraded`。
3. 当 `bp_converged=True` 时，该字段为 `ok`。
4. `lf_trace_digest` 的输入域不包含该字段。

### 9.4 HF 缺失原因

1. surrogate 模式下若 `detect_hf_score=None` 且原因为 `hf_basis` 未生成，则存在 `detect_hf_score_absent_reason`。
2. 该字段不会改变原有 `detect_hf_score`、`hf_status`、`hf_score` 逻辑。

### 9.5 Schema 与兼容性

1. 新增字段全部完成 schema 登记。
2. 旧 records 缺失这些字段时仍按 `absent_ok` 兼容。
3. 不新增 schema drift 或 append-only 违规。

### 9.6 脚本层攻击覆盖

1. workflow 生成的 detect/evaluate 产物中不再只有 `unknown_attack::unknown_params`。
2. `metrics_by_attack_condition` 中出现真实攻击分组。

## 十、最终交付口径

本次修复完成后，系统将具备以下交付特征。

1. detect 侧存在稳定、可审计的最终判决快照，便于后续冻结与审查。
2. LF 状态写盘口径与 HF 保持对称，消除目标字段缺失问题。
3. LF PRC 检测链的 BP 非收敛情况可被直接观测，且不破坏摘要可复算性。
4. surrogate 模式下的 HF 缺失原因具备显式诊断表达。
5. 所有新增字段均完成 append-only schema 登记，旧产物兼容性保持不变。
6. 攻击覆盖与路径可移植性具备清晰、可执行的补强路径，但不会冲击核心判决链的稳定性。
