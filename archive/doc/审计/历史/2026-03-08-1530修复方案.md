## 3. 修复方案整体思路

### 主链阻断点分析（目标字段反向溯源）

```
目标字段：evaluate_record.fpr_empirical ≤ 0.01
         evaluate_record.tpr_at_fpr_primary != null

↓ 依赖

统计样本独立性：100 对正负样本必须来自独立推理运行（STATIC-8 阻断）
↓ 依赖

内容分数判别性：watermarked hf_score > non-watermarked hf_score（STATIC-2 阻断）
↓ 依赖

embed 端 mask/plan 有效性：mask_digest != null（STATIC-1 阻断）
↓ 依赖

embed 语义掩码输入有效：预览生成提供语义一致的图像输入（当前为合成梯度图，语义失真）
```

次要依赖链：
```
detect 最终判决口径正确性：threshold_source = "np_canonical"（STATIC-4 阻断）
↓ 依赖

thresholds_artifact 在 detect 阶段被注入 cfg（机制已存在，编排层未接通）
↓ 依赖

calibrate 阶段产出 thresholds_artifact.json（需 detect_records_glob 指向有效样本集）
```

### 为什么这是最小风险方案

1. **Preview Generation 优先**：STATIC-1 的修复采用 Preview Generation 策略，在 `scripts/` 层将合成梯度图生成替换为无注入 SD3 推理，无需修改 YAML 静态字段，语义有效性得到保证（预览图与水印图使用相同生成条件 $(p, s)$）。
2. **最小化代码改动边界**：STATIC-2 修复限定于 `unified_content_extractor.py` 中 `compute_hf_score()` 单函数，不触及冻结语义枚举、字段路径、合同结构。
3. **不引入新架构**：STATIC-4 的修复利用 `run_detect.py` 已有的 `thresholds_path` 参数机制，仅修改 `scripts/` 编排层调用方式。
4. **统计样本修复在 scripts 层**：STATIC-8 修复限定于 `scripts/run_onefile_workflow.py:_run_dual_branch_embedding_and_detection()`，不改变 `main/` 接口语义。
5. **冻结面零改动**：所有修复不涉及 `frozen_contracts.yaml`、`runtime_whitelist.yaml`、`policy_path_semantics.yaml`、枚举语义、字段路径 registry。

#### P0-B：重新设计 compute_hf_score() 的评分信号 [STATIC-2]

**问题根因（代码证据）：**  
`main/watermarking/content_chain/unified_content_extractor.py:555-590`，`compute_hf_score()` 实现：

```
observed = mean_l2 + std_l2          # 来自 SD3 推理轨迹 L2 范数统计
expected_delta = embed 侧 hf_delta_norm_mean  # 同样基于 SD3 轨迹
alignment = 1/(1 + |observed - expected_delta|)
boost = 1 + max(0, embed_hint)       # embed_hint = embed_record.hf_score = None → boost = 1.0
return observed * alignment * boost
```

**根本缺陷**：`expected_delta` 来自 embed 侧 SD3 推理轨迹的 HF 统计，`observed` 来自 detect 侧 SD3 推理轨迹的 HF 统计。由于两次推理使用同一 SD3 模型，无论图像是否含水印，`mean_l2 + std_l2` 在模型降噪过程中均趋近 ≈262，导致 `|observed - expected_delta| ≈ 0`，`alignment ≈ 1.0`，`hf_score ≈ 262` 对所有 SD3 图像成立。此分数对水印存在与否无判别力。

**修复文件：** `main/watermarking/content_chain/unified_content_extractor.py`

**修复位置：** `UnifiedContentExtractor.extract()` 方法中，`_extract_channel()` 返回 `hf_score=None` 之后、`compute_hf_score()` 调用之前（第 196–210 行附近）。

**【审计修正 2026-03-08】修复入口错误更正：**

原方案描述"修改 `compute_hf_score()` 并通过 `hf_evidence` 传入 `hf_score_raw`"为错误。代码核实后确认实际流程如下：

- `hf_evidence`（由 `_build_hf_detect_evidence()` 构造的 embed 侧证据 dict）不含 `hf_score` 键
- `_extract_channel(evidence=hf_evidence_dict, score_key="hf_score", ...)` 进入 dict 分支，`score = evidence_dict.get("hf_score") = None`，返回 `hf_score=None`
- `hf_score_raw`（图像域旁路值，通过 `detector_inputs["hf_score"]` 传入）在 `evidence is not None` 时被 `_extract_channel()` 的 dict 分支跳过，**不被使用**
- `hf_score is None` → `hf_statistics.status == "ok"` 成立 → `compute_hf_score()` 被调用 → 返回 ≈262（对所有 SD3 图像）
- `compute_hf_score()` 的三个参数（`hf_statistics`、`injection_evidence`、`hf_evidence`）均不携带 `hf_score_raw`，在该函数内部进行任何修改对 `hf_score` 输出**无效**

**正确修复位置（代码证据）：**

在 `UnifiedContentExtractor.extract()` 中，`_extract_channel()` 调用之后，`extract_high_freq_statistics()` + `compute_hf_score()` 调用之前，插入图像域分数优先逻辑：

```python
hf_status, hf_score, hf_failure_reason = _extract_channel(
    evidence=normalized_inputs.get("hf_evidence"),
    score_key="hf_score",
    default_score=normalized_inputs.get("hf_score"),
    channel_name="hf"
)
# 【新增】若 _extract_channel() 因 hf_evidence dict 不含 hf_score 键返回 None，
# 且图像域旁路分数（detector_inputs["hf_score"] = hf_score_raw）可用，
# 则以 max(0.0, hf_score_raw) 作为主评分，绕过 compute_hf_score() 轨迹路径。
if hf_score is None:
    hf_raw = normalized_inputs.get("hf_score")
    if isinstance(hf_raw, (int, float)) and not isinstance(hf_raw, bool):
        hf_score = max(0.0, float(hf_raw))
        hf_status = "ok"
        hf_failure_reason = None
# 既有逻辑（成为 hf_score_raw 也缺失时的备用路径，不变）：
hf_statistics = extract_high_freq_statistics(normalized_inputs)
if hf_score is None and isinstance(hf_statistics, dict) and hf_statistics.get("status") == "ok":
    hf_score = compute_hf_score(...)
```

**修复目的：** 使 `hf_score` 反映图像域 HF 模式相关性（`hf_score_raw` clip 后），而非 SD3 轨迹 L2 范数统计，使其对水印图像和非水印图像具有统计可分离性。

**【必须在代码修改前完成的前置验证】（审计新增）：**

在 Colab 环境中运行 3–5 对 watermarked / non-watermarked 样本，记录 `hf_score_raw`（位于 `detect_record.content_evidence_payload.hf_detect_trace` 或 `score_parts.hf_detect_trace.hf_score_raw`）实测值：
- watermarked 图像的 `hf_score_raw` 需显著高于 non-watermarked 图像
- 若两者均趋近 0（watermarked 图像 `hf_score_raw ≈ -0.0006`，与已知产物值相同），则 clip 后 hf_score=0，判别性仍为零；此时需先检视 embed 侧 HF 注入强度（`watermark.hf.tau`、`tail_truncation_ratio`）后再决定修复路径

**修复原则：**

1. 修改 `UnifiedContentExtractor.extract()` 中 `_extract_channel()` 调用后的处理逻辑，使 `hf_score_raw` clip 结果直接作为 HF 主评分，跳过 `compute_hf_score()`。
2. `compute_hf_score()` 函数签名和实现**保持不变**（降级为 `hf_score_raw` 缺失时的备用路径）。
3. SD3 轨迹 L2 统计保留在 `hf_detect_trace` 诊断路径，不影响主评分链。
4. 评分方向保持：`hf_score_raw` 越大（图像域 HF 相关性越高）表示水印存在证据越强。

**输入变化：**

- `compute_hf_score()` 函数签名**不变**
- `UnifiedContentExtractor.extract()` 方法签名不变
- `detector_inputs["hf_score"]`（= `hf_score_raw`）原已传入，新增逻辑在 `_extract_channel()` 返回 None 后将其 clip 取用

**输出变化：**

- `score_parts.hf_score`：从约 262 下降为基于图像域相关性的分数（预期：watermarked ≫ 0，non-watermarked ≈ 0）
- `content_score`：不再由单一 HF 项主导（HF 贡献 0.3 权重下降至合理范围）
- `score_parts.hf_detect_trace`：路径 L2 统计可保留为诊断字段，不影响主评分

**注意**：修复后 `content_score` 分布将整体下降（从约 78 降至图像相关性尺度）。这意味着 `calibration_record.threshold_value` 将随之变化，但 NP 校准框架会基于新分布重新估计阈值，统计口径自动一致。

#### P0-C：生成真正独立的统计样本集 [STATIC-8]

**问题根因（代码证据）：**  
`scripts/run_onefile_workflow.py:1517-1760`，`_prepare_detect_records_with_minimal_ground_truth()` 无论走 dual-branch 路径还是 clone fallback 路径，正样本均为对 `source_detect_path`（单一 detect record）进行 `pair_count` 次 JSON 深拷贝，结果是 100 个文件共享同一 `run_id`、`cfg_digest`、`infer_trace_canon_sha256`、`content_score`，有效统计样本量 = 1。

**修复文件：** `scripts/run_onefile_workflow.py`

**修复位置：** `_run_dual_branch_embedding_and_detection()` 函数（第 1259 行起）及其被调用路径。

**修复目的：** 使每个 `pair_index` 对应一次真实独立的 SD3 embed + detect 推理运行（不同 `run_id`、不同 `infer_trace_canon_sha256`），使统计样本满足独立性假设。

**修复原则：**

1. 对 `pair_count=N` 的每个 `pair_index`，必须独立执行 embed + detect 推理（各一次 SD3 前向推理）。
2. 正样本（watermarked）和负样本（non-watermarked dual-branch）必须在不同的随机种子或不同的 prompt 下各自独立运行。
3. 每个样本文件的 `run_id` 必须唯一（`uuid` 独立生成）。
4. `infer_trace_canon_sha256` 必须唯一（不同推理过程的轨迹 SHA256 天然不同）。

**输入变化：**

- `pair_count=100` 时，脚本需执行 100 次 embed 运行 + 100 次正样本 detect 运行 + 100 次负样本 detect 运行（共约 300 次 SD3 推理）。
- 需传入 100 条不同 prompt（可来自 `prompts/paper_small.txt` 文件的前 100 行），通过现有 `minimal_ground_truth_prompts_file` 字段指定。

**输出变化：**

- 统计样本文件集合：各文件 `run_id` 唯一，`content_score` 值呈真实分布（不再全等）。
- `calibration_record.n_selected_null`：仍为 100，但由 100 个独立 detect 运行产生。
- `evaluate_record.n_neg`：同上。

**重要约束**：

- 此修复属于 Colab runtime 前提条件（参见 RUNTIME-1）；若统计样本分布仍重叠（即使 100 个独立运行），FPR 改善取决于 P0-B（HF 分数设计）是否生效。
- 本修复不改变 `main/` 任何代码，仅修改 `scripts/` 层的编排逻辑。

#### P1-A：打通 calibrate → detect 的 canonical 阈值注入链路 [STATIC-4]
