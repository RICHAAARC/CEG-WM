二、总判断

结论是：**当前项目尚不满足“论文级机制已经闭环成立”的判断条件。**

问题不在于“没有工程骨架”，而在于**机制定义、实现路径、检测路径、`impl_id` 语义、子空间语义解释**之间仍存在多处不一致。
其中最关键的问题有四项：

（一）**HF 通道的嵌入机制与检测机制并非同一算法闭环。**
（二）**`subspace` 的 `LF / HF` 划分与“语义显著性区域”的绑定关系存在概念混叠。**
（三）**`embed` 与 `detect` 两侧的子空间重算输入域不对称，导致 JVP 证据级别不稳定。**
（四）**当前创新点更接近“多模块拼接 + 工程收口”，而不是“机制层严格对齐的新方法”。**

因此，若现在直接据此撰写“本文提出了一种语义自适应、双通道、可解释、几何鲁棒的统一水印机制”，该表述会被代码事实削弱。

三、内容链审查

（一）语义掩码层：成立为“路由前提”，不成立为“水印证据本体”

`SemanticMaskProvider.extract()` 在 `main/watermarking/content_chain/semantic_mask_provider.py:147-193` 的语义是明确的：
当掩码成功生成时，输出的是 **structural evidence**，`score=None`，并以 `mask_digest`、`mask_stats`、`resolution_binding` 等形式作为后续规划绑定条件，而不是直接作为检测分数。

这一路径本身没有明显错误。问题在于：
你当前方法叙事里，语义掩码被赋予了“内容自适应子空间划分”的高机制地位，但代码上它目前更像**路由与摘要绑定器**，而不是强语义—频域耦合器。
换言之，掩码层目前能证明“后续机制依赖了某个 mask”，但不能证明“该 mask 以物理一致的方式决定了 LF / HF 子空间”。

（二）子空间规划层：当前不是严格的“语义—频域联合划分”，而是“掩码区域划分 + SVD 子空间估计”的并置

1. `build_region_index_spec_from_mask_v1()` 将 `mask_true_grid` 直接定义为 `HF` 区域，将其补集定义为 `LF` 区域，见
   `main/watermarking/content_chain/subspace/subspace_planner_impl.py:40-136`。

这意味着当前实现里：

* `HF region = mask_true_grid`
* `LF region = mask_false_grid`

这不是频域定义，而是**空间区域定义**。
如果你的方法目标是“纹理复杂区域走 HF，平滑区域走 LF”，那么“显著性前景区域 = 高频区域”这一映射并不成立。
显著主体区域可能是大面积平滑物体；背景区域也可能包含密集纹理。
因此，这一映射缺少频率语义根据。

2. `SubspacePlannerV2` 又把这两个区域直接别名为：

* `smooth_semantic_region = lf_region`
* `texture_semantic_region = hf_region`

见 `main/watermarking/content_chain/subspace/subspace_planner_impl.py:3025-3030`。

这一步不是“推断”，而是“直接赋值”。
因此，代码目前并没有真正证明“LF 域对应平滑语义、HF 域对应纹理语义”，只是把已有区域标签改名。

3. `_estimate_low_dim_subspace()` 的 `LF / HF basis` 估计与上述区域定义并不等价，见
   `main/watermarking/content_chain/subspace/subspace_planner_impl.py:611-787`。

其核心流程是：

* 从轨迹样本构造特征矩阵；
* 做 `mask-conditioned` 列选择；
* 拼接 `JVP` 样本；
* 对组合矩阵做 `SVD`；
* `LF basis = leading singular subspace`
* `HF basis = 后续一段 singular vectors`

这一定义是“代数子空间分解”，而不是“空间高频 / 低频分解”。
所以你当前代码里同时存在两套不同的 `LF / HF` 语义：

1. 空间区域上的 `LF / HF`
2. 代数子空间上的 `LF / HF`

这两者没有被严格打通。

（三）`mask → feature` 的映射方式可审计，但解释性不足

`build_subspace_conditioning()` 见
`main/watermarking/content_chain/subspace/subspace_planner_impl.py:861-982`。

其核心映射模式为：

`mapping_mode = "mask_index_hash_projection_v1"`

即：把 `mask` 网格索引通过 `mask_digest + feature_dim` 的哈希投影到特征维度。
这条路径有两个性质：

1. **优点**：确定性强，可复算，可审计。
2. **问题**：它不是几何一致映射，也不是基于真实 token / patch / attention 对齐的映射。

因此，这一层在工程上是稳定的，在论文机制上却偏弱。
它回答了“如何稳定绑定”，但没有回答“为什么这组特征维度代表该语义区域”。

（四）`HF subspace` 的实际构造与命名存在不一致

`_build_high_freq_subspace_spec()` 见
`main/watermarking/content_chain/subspace/subspace_planner_impl.py:1268-1304`。

它不是从 `hf_projection_matrix` 或高频谱能量直接构造，而是：

* 读取 `subspace_spec.top_feature_indices`
* 取其尾半部分
* 命名为 `HF subspace summary`

即：

`selector = "planner_top_feature_tail_half"`

这意味着当前 `HF` 摘要并不是“真实高频分量”的定义，而是“重要特征索引列表的后半段”。
这一命名会造成解释上的过度承诺。

（五）LF 通道：相对对齐

LF 通道是当前内容链里最接近闭环成立的部分。

1. 嵌入侧
   `LatentModifier` 在 `main/watermarking/content_chain/latent_modifier.py` 中调用
   `channel_lf.compute_lf_basis_projection()` 与 `channel_lf.apply_low_freq_encoding()`。
   `channel_lf.apply_low_freq_encoding_torch()` 的核心是：

* 由 `plan_digest` 导出固定 `LDPC codeword`
* 生成 step-conditioned 的 `pseudogaussian` 幅值
* 在 `LF coeffs` 上执行加性编码

见 `main/watermarking/content_chain/channel_lf.py:176-237`。

2. 检测侧
   `LowFreqTemplateCodecV2.detect_score()` 在
   `main/watermarking/content_chain/low_freq_coder.py:727-859` 中：

* 读取 `lf_basis.projection_matrix`
* 投影得到 `coeffs`
* 复现相同 `LDPC` 模板
* 计算 whitened correlation
* 通过 logistic 变换输出 `lf_score`

这条链条虽然仍有启发式成分，但“嵌入—检测”的统计对象是一致的，因而 LF 通道基本成立。

（六）HF 通道：当前为最严重的不对齐点

这是本次审查中最关键的问题。

1. 论文配置声明的 HF 实现是
   `impl.hf_embedder_id: "high_freq_template_codec_v2"`，见
   `configs/paper_full_cuda.yaml:121-127`。

2. 正式 registry 绑定的 `high_freq_template_codec_v2` 是 `HighFreqTemplateCodecV2`，见
   `main/registries/content_registry.py:145-163`。

该类的机制定义非常明确，见
`main/watermarking/content_chain/high_freq_embedder.py:53-60, 137-314`：

* `embed`：在选定位置上加性注入 `keyed Rademacher template`
* `detect`：对同一位置集合做 whitened correlation

这是一条**模板加性注入—模板匹配检测**闭环。

3. 但真正的 `latent per-step` 主嵌入路径并没有调用这套 `HighFreqTemplateCodecV2.embed()`。
   实际执行的是 `LatentModifier.apply_latent_update()` 中的 HF 分支，见
   `main/watermarking/content_chain/latent_modifier.py:214-252`：

* 先投影到 `hf_basis`
* 再调用 `channel_hf.apply_hf_truncation_constraint()`
* 然后重建回 latent

`channel_hf.apply_hf_truncation_constraint_torch()` 的实际操作是对系数做百分位阈值截断，见
`main/watermarking/content_chain/channel_hf.py:103-154`。
这是一条**尾部截断 / 稀疏约束**机制，而不是模板加性注入机制。

4. 检测侧却明确调用 `HighFreqTemplateCodecV2.detect()`，见
   `main/watermarking/detect/orchestrator.py:924-970`。
   而该检测器期望观测到的是模板相关信号，见
   `main/watermarking/content_chain/high_freq_embedder.py:223-314`。

因此，当前 HF 通道实际上是：

* 嵌入：`truncation-constrained coefficient modification`
* 检测：`keyed template matched correlation`

这两者不是同一水印家族。

（七）更严重的问题：同一个 `impl_id` 同时指代两种不同机制

`channel_hf.py` 顶部直接写死：

`HF_CHANNEL_IMPL_ID = "high_freq_template_codec_v2"`

见 `main/watermarking/content_chain/channel_hf.py:24-25`。

但 `channel_hf` 本身实现的是“尾部截断约束”，不是 `HighFreqTemplateCodecV2` 中定义的“Rademacher 模板加性注入 + 模板匹配检测”。

这意味着：
**同一个 `impl_id` 在系统内同时指代两种不同算法。**

这不是简单的“实现细节差异”，而是 `impl_id` 语义漂移。
如果你的项目目标包含“可审计、可复现实验、论文级机制声明”，那么这种 `impl_id` 复用在机制层面不可接受。

四、`embed / detect` 两侧子空间规划是否对齐

结论是：**不完全对齐。**

（一）`embed` 侧会把 `unet / transformer` 传入规划器
见 `main/watermarking/embed/orchestrator.py:525-533`。

（二）`detect` 侧的 `_build_planner_inputs_for_runtime()` 只传 `pipeline` 与 trajectory cache，不传 `unet / transformer`
见 `main/watermarking/detect/orchestrator.py:2360-2368`。

（三）而 `_estimate_jvp_matrix()` 的逻辑是：

* 有 `jvp_operator`，则走真实 JVP
* 否则若有 `unet`，走真实网络 JVP
* 否则退化为 `surrogate_transition`

见 `main/watermarking/content_chain/subspace/subspace_planner_impl.py:1534-1642`。

因此，在当前代码下：

* `embed` 侧更有可能形成 `real_unet / runtime_operator` 级别的 JVP 证据；
* `detect` 侧则更容易退化到 `surrogate_transition`。

这会导致两侧 `plan_digest / basis_digest / evidence_level` 的构成条件不同。
如果你的论文叙事是“检测侧重建同一规划并进行一致性验证”，那么当前输入域并不对称。

五、几何链审查

（一）几何链总体上比 HF 内容链更自洽

当前正式注册路径不是旧的泛化 `AttentionAnchorExtractor`，而是：

* `attention_anchor_map_relation_v2`
* `geometry_latent_sync_sd3_v3`

见 `main/registries/geometry_registry.py`。

（二）注意力锚点路径要求真实 `runtime_self_attention`

`AttentionAnchorMapRelationV2.extract()` 要求：

* `attention_maps` 必须存在
* `attention_maps_source == "runtime_self_attention"`

否则直接 `failed`，不允许 proxy，见
`main/watermarking/geometry_chain/attention_anchor_extractor.py:672-810`。

`detect` 侧也确实会尝试把运行时抓取的 self-attention maps 送入 geometry runtime inputs，见
`main/watermarking/detect/orchestrator.py:3983-3997`，
以及 `main/diffusion/sd3/infer_runtime.py:386-506`。

因此，几何锚点链条在“真实性”这一点上是成立的。

（三）同步链的主分数定义清晰

`GeometryLatentSyncSD3V3.extract()` 中：

* 几何主分数 `geo_score = template_match_score`
* `relation_digest` 作为前置绑定条件
* 不确定度过高时直接 `mismatch`

见 `main/watermarking/geometry_chain/sync/latent_sync_template.py:1190-1315`。

这意味着几何链不是一个松散的附属特征，而是“关系锚点 + 频域同步模板”的组合证据。
就链内一致性而言，这一部分比内容链 HF 分支更完整。

（四）但几何链仍然是“辅助 / rescue”，不是与内容链联合编码的统一消息机制

当前几何链与内容链共享的绑定更多体现为：

* `relation_digest`
* `sync_digest`
* `plan_digest`
* 运行时上下文与审计摘要

它并没有与 LF / HF 的消息编码形成“同一消息在不同不变域上的并行编码”。
所以，几何链目前更像**辅助认证链**，而不是**与内容链联合承载同一水印消息的第二通道**。

如果你的研究目标是“多证据链联合承载同一生成声明 / watermark message”，那么这一层还没有到位。

六、融合层与最终判断机制

（一）最终判决逻辑是清晰的

`NeumanPearsonFusionRule.fuse()` 的逻辑为，见
`main/watermarking/fusion/decision.py:243-327`：

1. 内容分数做 `NP` 主判决；
2. 仅当内容为 `False` 且几何为 `ok` 时，几何链才可能触发 `rescue`；
3. 几何链不能把 `True` 翻成 `False`。

这条设计在统计语义上是可解释的。
从“系统安全性”角度看，这一设计是合理的。

（二）但 `content_score` 的组合规则偏启发式

`compute_content_score()` 的规则是：

* `LF` 为主；
* 若 `HF` 可用，则 `content_score = lf_score + 0.3 * hf_score`；
* 否则仅用 `LF`

见 `main/watermarking/content_chain/detector_scoring.py:218-284`。

这一规则不是错误，但它是**固定权重启发式**，而不是严格来源于似然比或联合校准模型。
若论文主张“内容链是双通道最优统计融合”，则当前代码不能支撑该表述。
若论文仅表述为“LF 主通道，HF 辅助增强通道”，则该实现可以作为一个工程化近似，但仍需要在实验中给出单独校准与消融证明。

七、是否符合你的目标

如果你的目标定义为：

“构建一个内容自适应证据链 + 几何不变证据链的统一扩散模型水印系统，其中内容链中 LF / HF 具有明确机制分工，编解码闭环对齐，可解释且可用于论文主结论。”

那么当前状态是：**部分接近，但尚未达到。**

更具体地说：

（一）**内容自适应**：只完成了“路由绑定”层，未完成“语义—频域统一映射”层。
（二）**双通道闭环**：LF 基本成立，HF 不成立。
（三）**几何不变证据链**：辅助链成立度高于内容链 HF 分支。
（四）**统一机制叙事**：目前不成立，因为 `impl_id`、嵌入算子、检测算子、语义命名之间存在漂移。
（五）**论文级可解释性**：当前能解释“工程绑定关系”，不能充分解释“机制为何如此划分”。

八、创新性评估

当前创新性不足不在于“没有新模块”，而在于“新模块之间尚未形成不可替代的统一机制关系”。

目前的结构更接近：

* `PRC-like LF`
* `T2S-like HF truncation`
* `template-based HF detector`
* `semantic mask routing`
* `attention anchor`
* `latent sync template`
* `NP fusion + geo rescue`

这些模块分别都有研究价值，但当前代码并没有把它们压缩为一个足够干净的核心创新命题。
如果现在投稿，审稿人很可能会追问以下问题：

1. 为什么 `mask_true` 必然对应 `HF` 区域？
2. 为什么 `HF` 嵌入使用截断约束，而 `HF` 检测使用模板匹配？
3. 为什么同一个 `impl_id` 对应两种不同机制？
4. 为什么 `detect` 侧子空间重算不使用与 `embed` 相同的 `JVP` 输入域？
5. 几何链到底是“第二证据链”，还是“与内容链共享消息的第二编码链”？

如果这些问题不先在代码层解决，创新性论述会停留在“组合设计”层，而不是“机制贡献”层。

九、优先级最高的调整项

（一）先统一 HF 通道家族

你需要二选一，而且只能保留一种正式机制定义。

方案 1：保留 `HighFreqTemplateCodecV2`
则 `latent per-step` 主嵌入路径必须改为真正调用“模板加性注入”语义，而不是 `channel_hf` 的尾部截断约束。
这是与当前配置、registry、detect 路径最一致的方案。

方案 2：保留 `channel_hf` 的 truncation 机制
则应删除或更名 `high_freq_template_codec_v2`，并把检测器改写为与“截断后系数统计”匹配的检测器。
这会要求你放弃当前 HF 模板检测叙事。

当前项目更适合走方案 1。

（二）拆开“区域路由”与“频域子空间”两个概念

建议不要再让：

* `hf_region_index_spec`
* `lf_region_index_spec`

同时承担“空间路由区域”与“频域语义区域”两种含义。

更合理的结构应当是：

1. `semantic_region_index_spec`：只表示 mask / saliency / texture routing；
2. `spectral_subspace_spec`：只表示 LF / HF basis；
3. `routing_policy`：决定哪一类区域走哪一类通道。

否则当前 `HF = mask_true = texture = high_frequency` 这一串等号会持续造成方法解释崩塌。

（三）使 `detect` 侧规划输入域与 `embed` 侧对称

至少应保证：

* `detect` 侧也能拿到 `unet / transformer` 或真实 `jvp_operator`；
  或
* 检测时不再重算弱化版 `plan`，而是只验证嵌入时 `plan_digest` 与运行时支持证据。

当前的“嵌入强输入域、检测弱输入域”不利于论文论证。

（四）替换 `mask_index_hash_projection_v1`

若你要强调“语义显著性驱动的可解释子空间规划”，则哈希投影不应再作为正式方法核心。
它可以保留为 fallback 或审计辅助，但正式路径应当转向：

* token / patch 级真实索引映射；
* attention / Jacobian attribution 映射；
* 或至少是可解释的固定几何投影。

十、最终结论

最终结论可以压缩为一句话：

**当前项目已经具备论文级工程框架，但尚未形成论文级机制闭环。**

更精确地说：
