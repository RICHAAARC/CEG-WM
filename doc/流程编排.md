## Colab 会话隔离约束

所有 notebook 都应视为在**单独隔离的 Colab 会话**中执行。

这意味着：

1. `PW00` 到 `PW05` 之间不存在可直接共享的进程内状态、内存对象或临时运行时上下文；
2. 后续流程唯一能够读取的上游信息，只能是**已经明确落盘到 Google Drive 的文件**；
3. 后续 notebook 不能读取、检查或依赖其他流程会话中**未落盘**的环境依赖状态、临时缓存、`/tmp` 文件、Colab runtime 局部安装结果或其他临时会话文件。

因此，凡是需要跨 notebook 传递的事实、工件状态、依赖检查结果或 readiness 结论，都必须先写成 Google Drive 中可复核的落盘文件，再由后续流程读取；不能把“上一个 Colab 会话里看起来存在”视为可验证输入。

## notebook 说明

### （一）`PW00_Paper_Eval_Family_Manifest.ipynb`

这是轻流程，不耗重 GPU。

职责只有五个：

1. 冻结本次 paper evaluation 的家族身份；
2. 读取 prompt 列表与 seed 规划；
3. 生成 event 级样本网格；
4. 生成 shard 计划；
5. 生成全局 identity snapshot。

输出至少包括：

1. `paper_eval_family_manifest.json`
2. `source_event_grid.jsonl`
3. `source_shard_plan.json`
4. `attack_event_grid.jsonl`
5. `attack_shard_plan.json`

这个 notebook 是整条 paper 路径的总入口。

### （二）`PW01_Source_Event_Shards.ipynb`

这是**重 GPU notebook**。
但它本身应设计成**参数化模板**，同一个 notebook 既可跑正样本 shard，也可跑 clean negative shard。

建议参数：

1. `family_id`
2. `sample_role`，取值为 `positive_source` 或 `clean_negative`
3. `shard_index`
4. `shard_count`
5. `local_worker_count`，你当前建议固定为 `2`

它的职责是：

1. 读取 `source_shard_plan.json`；
2. 取出本 shard 对应的 event 列表；
3. 对每个 `(prompt, seed)` 事件运行生成与检测；
4. 写出本 shard 的逐 event 工件与 shard manifest；
5. 不做全局阈值；
6. 不做全局 evaluate。

这里有一个很重要的设计点：

**你不能再沿用 `prompt_{index}` 作为基本目录单位，而要改成 `event_{index}`。**

也就是每个事件都要显式记录：

1. `event_id`
2. `prompt_index`
3. `prompt_text`
4. `prompt_sha256`
5. `seed`
6. `sample_role`
7. `runtime_config_path`
8. `embed_record_path`
9. `detect_record_path`
10. `source_image_path`
11. `attestation_statement_view`
12. `attestation_bundle_view`
13. `attestation_result_view`

#### 正样本 shard 的语义

执行链为：

`preview / generation -> embed -> detect`

#### clean negative shard 的语义

执行链为：

`plain generation without watermark -> detect`

这里的关键是：
clean negative 的生成与检测必须与正样本共用同一模型、同一 prompt 分布、同一 seed 口径、同一 detect 配置，但**不进行 watermark embed**。

### （三）`PW02_Source_Merge_And_Global_Thresholds.ipynb`

这是轻流程或中轻流程。
它是整个 paper family 的核心收口点。

职责分三层：

#### 1. 合并 source shards

读取所有 `positive_source` 与 `clean_negative` shard package，进行硬一致性校验，然后分别合并为：

1. `positive_source_pool_manifest.json`
2. `clean_negative_pool_manifest.json`

#### 2. 生成全局 calibration / evaluate split

这里不要直接拿全量 source pool 做一次混合统计，而要显式区分：

1. `calib_pos`
2. `calib_neg`
3. `eval_pos`
4. `eval_neg`

也就是说，family manifest 里就应该提前定义 split 规则，而不是在合并时临时切。

#### 3. 统一构建全局阈值并执行一次全局 evaluate

这个阶段只做一次全局：

1. `content_chain_score` 阈值构建；
2. `event_attestation_score` 阈值构建；
3. clean evaluation；
4. source finalize package 写出。

输出建议包括：

1. `paper_source_finalize_manifest.json`
2. `thresholds/content/thresholds.json`
3. `thresholds/attestation/thresholds.json`
4. `calibration_record_content.json`
5. `calibration_record_attestation.json`
6. `evaluate_record_clean_content.json`
7. `evaluate_record_clean_attestation.json`
8. `evaluate_record_clean_system_final.json`

### （四）`PW03_Attack_Event_Shards.ipynb`

这是第二个重 GPU notebook，但它不再做 clean generation。

它的输入是 `PW02` 生成的 finalized positive source pool。
它的职责是：

1. 读取 finalized positive source events；
2. 按 attack protocol 派生 attacked images；
3. 对 attacked images 运行 detect；
4. 写出 attacked shard package；
5. 不在 shard 内做全局 metrics。

它的基本单位应是：

[
\text{event_id}, \text{attack_family}, \text{attack_config}
]

建议每个 attacked 事件至少记录：

1. `parent_event_id`
2. `attack_family`
3. `attack_params_digest`
4. `attacked_image_path`
5. `detect_record_path`
6. `sha256`
7. `runtime_config_snapshot`
8. `threshold_binding_summary`

这一阶段不应再重新生成 clean 正样本图像，只从已有 source pool 派生 attacked 版本。

### （五）`PW04_Metrics_Tables_And_Tail_Estimation.ipynb`

这是只读统计 notebook，不应重跑生成。

职责包括五类：

1. clean TPR / FPR
2. attacked TPR
3. content / attestation / system_final 三层指标
4. 图表与论文表格
5. 可选的 tail estimation

这里建议固定输出三套主指标：

1. `content_chain`
2. `event_attestation`
3. `system_final`

不要只报一个。

同时，还应显式输出以下论文强相关指标：

1. `rescue_rate`
2. `geo_helped_positive_count`
3. `geo_not_used_count`
4. `attested_rate`
5. `clean false accept count`
6. `attack true accept count by family`
7. `bootstrap confidence interval`

若你要尝试 `1e-4 / 1e-5` 的外推，这个 notebook 内再调用 `pw06_estimate_tail_fpr.py` 即可，但要把结果单独命名为：

1. `estimated_tail_fpr_1e4`
2. `estimated_tail_fpr_1e5`

不要和 empirical clean FPR 混写。

### （六）`PW05_Release_And_Signoff.ipynb`

这是最终只读封装与审计流程。

职责包括：

1. 打包 manifests、configs、thresholds、metrics、tables、figures；
2. 写出 paper release manifest；
3. 执行最终 signoff；
4. 生成对论文正文与附录都可引用的包结构。

## 脚本说明

### （一）`pw_common.py`

只放共用逻辑：

1. 路径标准化；
2. digest 计算；
3. family manifest 读取；
4. shard 计划读取；
5. secret fingerprint 生成；
6. identity snapshot 比较；
7. event id 生成；
8. shard manifest 校验。

### （二）`pw00_build_family_manifest.py`

负责：

1. 读取 `pw_base.yaml`；
2. 读取 prompt 文件；
3. 读取 seed 规划；
4. 构造 event grid；
5. 切分 calibration / evaluation split；
6. 生成 `source_shard_plan.json` 与 `attack_shard_plan.json`。

### （三）`pw01_run_source_event_shard.py`

这是 paper family 中最关键的运行脚本。

它应复用现有 `scripts/01_run_paper_full_cuda.py` 中的底层 stage 调用逻辑，但自己新增 **event-bound** 的 wrapper。
建议自己实现一个本地函数语义：

1. `_run_positive_source_event(...)`
2. `_run_clean_negative_event(...)`

### （四）`pw02_merge_source_event_shards.py`

它做四件事：

1. 校验所有 source shard 的一致性；
2. 合并成 positive / negative 两个全局 event pool；
3. 基于 calibration split 构建全局 thresholds；
4. 基于 eval split 生成 clean metrics。

### （五）`pw03_run_attack_event_shard.py`

只负责：

1. 从 finalized positive source pool 读取 parent events；
2. 施加 attack；
3. detect；
4. 输出 attacked shard package。

### （六）`pw04_merge_attack_event_shards.py`

只负责把所有 attack shard 合并成统一的 `attack_pool_manifest.json`。

### （七）`pw05_compute_metrics.py`

只读计算：

1. clean metrics
2. attack metrics
3. content / attestation / system_final 三层表格
4. confidence intervals
5. rescue metrics

### （八）`pw06_estimate_tail_fpr.py`

这是可选脚本，用于：

1. 拟合 negative score upper tail；
2. 输出 `estimated_tail_fpr`；
3. 输出拟合诊断图与 stability summary。

### （九）`pw07_release_signoff.py`

负责最终发布与签收，不改方法工件。

## 运行流程

### （一）步骤 1：单会话运行 `PW00`

作用：

1. 生成 family manifest；
2. 固化 prompt / seed / split / shard plan；
3. 写入 Drive。

只跑一次。

### （二）步骤 2：多会话并行运行 `PW01`

并行方式如下。

#### 正样本并行

多个会话同时运行同一个 `PW01` notebook，但参数不同：

1. `SAMPLE_ROLE = "positive_source"`
2. `SHARD_INDEX = 0, 1, 2, ..., K-1`
3. `SHARD_COUNT = K`

#### 负样本并行

同理，再开若干会话：

1. `SAMPLE_ROLE = "clean_negative"`
2. `SHARD_INDEX = 0, 1, 2, ..., M-1`
3. `SHARD_COUNT = M`

你可以让正样本与负样本混合并发，也可以分两批跑。
为了简化，我更建议先统一 `SHARD_COUNT`，例如都设为 `8`。

### （三）步骤 3：单会话运行 `PW02`

作用：

1. 校验所有 source shards 是否完成；
2. 合并正负样本池；
3. 构建全局阈值；
4. 输出 clean paper metrics；
5. 生成 source finalize package。

只跑一次。

### （四）步骤 4：多会话并行运行 `PW03`

它只基于 `PW02` finalized positive source pool 做 attack detect。
多个会话运行同一个 `PW03` notebook，但参数不同：

1. `ATTACK_SHARD_INDEX = 0, 1, 2, ..., A-1`
2. `ATTACK_SHARD_COUNT = A`

### （五）步骤 5：单会话运行 `PW04`

作用：

1. 合并所有 attack shards；
2. 生成论文表格、图形与 clean / attack 指标；
3. 可选做 tail estimation。

### （六）步骤 6：单会话运行 `PW05`

作用：

1. 打包 release；
2. signoff；
3. 固化论文引用用结果包。
