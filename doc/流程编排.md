## 一、更强的全链路语义条件化算子

### 1. 当前落点

你现在的项目已经有较强的语义内容自适应基础，但它主要落在：

1. `semantic_mask_provider.py` 负责生成语义掩码与 `mask_summary`；
2. `subspace_planner_impl.py` 根据 `mask_summary` 构造 `hf_region_index_spec / lf_region_index_spec`；
3. 注入阶段按 region routing 去做 `LF/HF` 区域条件化注入。

但是，`trajectory_feature_space.py` 里写得很明确：
当前对 `mask_summary` 的支持还是**可选但未真正进入 feature operator 主体**，当前版本本质上仍是 `M = 1（全 latent）` 的实现，只是在 region routing 与注入使用上体现语义条件化。

### 2. 推进难度判断

这一项**能推进，而且技术上是中等难度**。
原因是你已经有了三块现成基础：

1. 语义掩码真实存在；
2. region routing 已真实存在；
3. trajectory feature 与 JVP operator 已真实存在。

因此，往前推并不是从零开始，而是把：

* 当前“mask-guided routed subspace”

推进到更强的：

* “mask-conditioned feature operator”

### 3. 主要阻力

主要阻力不是工程入口，而是**保持 formal 一致性与统计稳定性**。具体有三点：

1. 你要让 `mask_summary` 真正进入 feature normalization / projection operator，意味着：

   * `trajectory_feature_space.py`
   * `subspace_planner_impl.py`
   * detect 侧 exact feature extraction

   三处都要同步改。
2. 一旦 feature operator 本体改变，`plan_digest`、`basis_digest`、`lf/hf` 子空间统计分布都会变，连带可能影响当前已经稳定的 `LF` 主线。
3. 如果 mask 真进入 operator，必须重新验证：

   * `LF repair`
   * `GEO on/off`
   * sidecar statistics
     否则会破坏你刚建立起来的可判责性。

### 4. 发表收益

这一项的收益是：

**理论叙述与实现一致性提升很大，创新表述会更强。**

因为你当前论文里最容易被质疑的一点，就是“语义条件化到底是作用在路由层，还是作用在算子层”。
如果你补齐这一项，论文方法叙述会明显更完整。

### 5. 结论

这一项**值得推进**，但更适合作为：

* 投稿前的**增强项**
* 或下一版论文 / 扩展版工作的重点

而不是你现在最优先动的点。
因为它会触及内容链主干，风险高于当前你正在做的 ablation 收口。

## 二、更通用的 arbitrary timestep inversion

### 1. 当前落点

你当前 `LF formal exact` 修复已经成立，但它是**受条件约束**的：

1. `run_detect.py` 里 formal exact evidence 已走 `input-image-conditioned` 路径；
2. 但当前主线实际上只稳定支持 `edit_timestep = 0`；
3. `extract_image_conditioned_latent(...)` 本质上是：

   * 输入图像
   * `VAE encode`
   * 得到 latent
4. 它不是一个完整的 arbitrary timestep diffusion inversion framework。

### 2. 推进难度判断

这一项是三者中**最难推进**的。
因为它不只是工程补丁，而是会牵涉到：

1. image-conditioned latent recovery；
2. 任意 timestep 的 trajectory consistency；
3. formal exact evidence 与 expected sign / expected codeword 的时刻对齐；
4. detect 侧 reconstruction residual 的定义与 diagnostics 重写。

换言之，它会从当前的：

* “input-image-conditioned exact evidence”

推进到：

* “full arbitrary timestep inversion / reconstruction evidence”

这已经接近新增一条完整的方法链，而不是小修。

### 3. 主要阻力

当前阻力非常大，主要有四点：

1. 你现在没有现成的 arbitrary timestep inversion 主干；
2. 当前 diagnostics、`lf_retain_breakdown`、`protocol_control` 都围绕现有 `edit_timestep = 0` 语义构建；
3. 一旦支持 arbitrary timestep，当前很多“residual drift”分类都需要重写；
4. 这会显著增加 formal path 的复杂性，甚至可能引入新的不稳定项。

### 4. 发表收益

理论上，这项推进后论文会显得更完整、更“强”。
但从实际投稿收益看，它属于：

**高风险、高成本、但当前并非必须。**

因为你当前项目已经能在 formal 主线上成立 `LF`，而审稿人并不会因为你没有 arbitrary timestep inversion 就否定整篇工作；相反，如果你为了追求这一点而把现有稳定主线弄乱，反而得不偿失。

### 5. 结论

这一项**不适合当前阶段继续推进**。
更合理的定位是：

* 放进论文的 `discussion / limitation / future work`
* 作为后续版本的增强方向

如果现在强推，它最容易破坏你当前已经稳定下来的 formal 主线与消融判责性。

## 三、更统一的几何主分构造

### 1. 当前落点

这一项是当前最接近、也最容易收口的一项。

你现在的 GEO 链已经从“raw template score 太弱”推进到了：

* raw `template_match_score`
* `quality_score`
* repaired `template_confidence`
* `geo_score_repair.enabled`
* `active_geo_score_source`

也就是说，你现在实际上已经有：

1. 原始模板分；
2. 质量分；
3. repaired confidence 分；
4. 一整套 diagnostics；
5. `GEO off/on` 单变量 detect 复跑。

这说明几何链的主要问题已经不是“没有东西”，而是：

**主分表达还不够统一。**

### 2. 推进难度判断

这一项是三者中**最好推进**的，属于中低风险。
原因是：

1. GEO repair 已经真实存在；
2. `template_confidence` 已经进入正式结果；
3. diagnostics 工件已经能表达 `active_geo_score_source` 与 `geo_repair_direction_classification`；
4. 你已经具备 `GEO off/on` 单变量实验入口。

换言之，你现在不是在发明 GEO 主分，而是在做：

**主分语义统一与论文表述统一。**

### 3. 主要阻力

这一项的阻力相对小，主要在两点：

1. 你需要决定论文最终把 GEO 主分表述成什么：

   * raw template score
   * repaired template-derived confidence
   * 或“template-derived geometry confidence”
2. 你需要保证：

   * `GEO off/on` 对照完整
   * compare summary 完整
   * base-embed 追溯锚点完整

也就是说，当前阻力更多是**实验工件与方法叙述统一**，不是方法主干缺失。

### 4. 发表收益

这一项推进的收益很高，因为它会直接改善论文中最容易被问到的一个问题：

**“你几何链的正式主分到底是什么？”**

如果你把这一点收口得好，论文方法章节会显得非常完整，而且与你当前代码实现高度一致。
这对投稿非常有利。

### 5. 结论

这一项**最好推进，也最值得现在推进**。
因为：

1. 方法基础已经具备；
2. 实验入口已经具备；
3. 风险小；
4. 发表收益高；
5. 还能顺带提升你当前 `GEO off/on` 消融的说服力。

## 四、三者的综合排序

如果从“当前项目推进难度 × 风险 × 投稿收益”综合排序，我的建议是：

### 第一优先级：更统一的几何主分构造

原因：

1. 当前基础最成熟；
2. 实现最接近；
3. 风险最小；
4. 对论文方法叙述帮助最大。

### 第二优先级：更强的全链路语义条件化算子

原因：

1. 有真实代码基础；
2. 能显著增强创新叙事；
3. 但会触碰内容链主干，风险高于 GEO 统一主分。

### 第三优先级：更通用的 arbitrary timestep inversion

原因：

1. 技术跨度最大；
2. 最容易引入新不稳定项；
3. 当前不是发表必要项；
4. 更适合 future work。
