## 适用范围、目标与术语定义

### 适用范围

本方案适用于满足以下工程前提的项目：

1. 框架采用 **Frozen Contract** 约束 records 语义坐标系；
2. 采用 **policy_path + runtime whitelist** 约束运行策略语义；
3. 采用 **registry + impl_id** 承载真实方法注入；
4. 采用 **records + run_closure + bundle/manifest** 形成可审计证据包；
5. 已具备 `scripts/` 与 `tests/` 的最小闭环。

### 审计目标

审计目标是对“是否允许进入下一阶段构建 / 是否允许冻结签署 / 是否允许注入真实方法”给出可复核判定，重点覆盖：

* 冻结是否完整且单次 run 内唯一；
* 门禁是否合理合规且不可绕过；
* 注册表是否安全、可封存、可审计；
* 路径约束与关键产物写盘是否同源受控；
* 可复现性证据是否充分且可复算；
* 统计与判决语义是否符合 Neyman–Pearson 口径；
* 失败语义是否闭环、单一主因、可解释；
* 工程是否形成可重复运行的审计与回归闭环；
* 是否需要 E2E（最小端到端）执行以补足静态证据不足。

### 关键术语

1. **冻结面（Frozen Surface）**：records 字段结构与语义、fail/mismatch/absent 语义、policy_path 语义映射、阈值与统计口径、用于单因素证明的最小锚点字段集合。
2. **扩展壳（Extensible Shell）**：通过 registry 注入的 impl 集合（content/geometry/fusion/subspace 等），不进入冻结契约，但必须被可审计身份约束。
3. **阻断冻结级（BLOCK）**：违反即判定 `BLOCK_FREEZE` 的失败类型，仅来自本方案明确标注为 BLOCK 的检查项。
4. **非阻断级（NON_BLOCK）**：工程增强与闭环要求，失败会提升风险但不直接阻断冻结，除非你在未来将其升级为冻结要求并版本化。

## 审计输出强制规范

### 单项输出

* 所有检查项必须逐条输出结论：`PASS` / `FAIL` / `N.A.`（仅当该项确实不适用于仓库结构时允许 `N.A.`）。
* 每个 `FAIL` 必须包含四元组：

  1. 违反规则 →
  2. 证据定位（文件路径 + 行号区间或函数/类名） →
  3. 影响说明 →
  4. 修复建议（最小改动且不引入新风险）
* 禁止使用“可能 / 建议关注 / 视情况而定”等模糊措辞。
* 所有检查项结论：`PASS` / `FAIL` / `N.A.`需要放在每项的最开始，并且必须大写且加粗。

### 对抗式扫描强制要求

涉及以下类别的扫描必须给出命中列表（空列表也必须显式输出）：

* 写盘路径：`open(...,"w/a/x")`、`json.dump`、`Path.write_*`、`shutil.copy/move` 等；
* YAML 加载：`yaml.load`、非 safe loader；
* 动态执行：`eval/exec/compile`、`pickle/dill/cloudpickle`；
* 网络访问：`requests/urllib/httpx`、`hf_hub_download/snapshot_download`、`curl/wget/git clone` 等。
  未给出命中列表不得宣布已完成扫描，对应检查项必须 `FAIL`。

### 顶层判定输出（审计末尾必须重复）

* `FreezeSignoffDecision`：`ALLOW_FREEZE` / `BLOCK_FREEZE`
* `BlockingReasons`：列表（仅包含 BLOCK 级 FAIL）
* `RiskSummary`：`HIGH` / `MED` / `LOW`

### 风险摘要计算规则（强制）

* 存在任一 BLOCK 级 `FAIL` → `RiskSummary=HIGH`
* 否则若 `FAIL` 数量 >0 且无 BLOCK → `RiskSummary=MED`
* 否则（全 PASS 或仅 N.A.）→ `RiskSummary=LOW`

## 审计执行层级与适用场景

### 静态审计（默认必做）

不依赖 GPU，不要求生成真实模型产物。覆盖冻结唯一性、门禁不可绕过、注册表安全、路径约束、危险 I/O、动态执行与网络访问等。

### 运行审计（建议必做，placeholder 可运行即可）

执行最小 CLI 或最小工作流，验证：records 写盘确实触发冻结门禁、facts 绑定唯一、digest 可复算、run_closure 必产出、bundle/manifest 一致性。

### Freeze sign-off 审计（冻结签署必做）

强制固定审计集合与固定入口，输出证据包，并给出 `ALLOW_FREEZE` / `BLOCK_FREEZE`。

### E2E 执行（条件触发）

仅在静态证据不足或变更触及 E2E 触发规则时执行，用于验证端到端可运行性、环境差异锚定、真实管线与 hook 不破坏闭环。

## 检查项清单（必须逐条输出 PASS/FAIL/N.A.）

> 下述每项均标注 **BLOCK / NON_BLOCK**。只有 BLOCK 级 `FAIL` 可进入 `BlockingReasons` 并触发 `BLOCK_FREEZE`。

### A. 冻结面完整性（Frozen Surface Integrity）

**A1（BLOCK）冻结契约一致性**

* 单次 run 内 `contract_version` 唯一且在 records / run_closure 一致；
* `frozen_contracts.yaml` digest 被记录且可复算；
* 冻结字段 append-only，禁止语义变更；
* records 写盘前 interpretation 必达且已绑定。

**A2（BLOCK）schema 权威化**

* required 字段全集由解释面唯一给出；
* `schema.validate_record()` 强制要求 interpretation；
* required 仅允许“锚点类字符串字段”；
* 校验失败包含异常类型与 `field_path`，可追溯。

**A3（BLOCK）records schema_version 与 append-only**

* records 必含 `schema_version`（或等价）；
* 不允许回退或漂移；新增字段必须 append-only；
* 存在可绕过注入即 `FAIL`。

**A4（BLOCK）解释面语义漂移防护**

* 同一 `contract_version` 下不得存在条件分支导致同字段语义不唯一；
* 枚举集合、字段含义变化必须升级 `contract_version`。

**A5（BLOCK）解释面对象进程内唯一性**

* 一次 run 内冻结对象来自单一权威加载点；
* CLI 与 records_io 不得各自独立解释并混用。

**A6（BLOCK）冻结配置加载安全（YAML 安全子集）**

* 禁止 `yaml.load` 与不安全 loader；仅允许 `yaml.safe_load`；
* YAML 解析入口唯一且可枚举。

**A7（BLOCK）冻结文件权威路径与快照锚定**

* frozen/whitelist/semantics 必从 `configs/` 权威路径加载；
* digest 与必要快照必须锚定到 run_closure（或等价证据字段）。

---

### B. 门禁执行完整性（Gate Enforcement）

**B1（BLOCK）写盘不可旁路**

* records 写入只能经 `records_io` 受控入口；
* 写盘前必经 `freeze_gate.assert_prewrite()`；
* 对抗式扫描必须列出全部命中点并判定可达性。

**B2（BLOCK）impl_id 白名单执行**

* impl_id 校验必须在写盘前执行；缺失默认拒绝；
* impl_identity 必写入 records 且可定位至 run_closure。

**B3（BLOCK）policy_path 语义门禁**

* policy_path 必来自 whitelist；
* policy_path_semantics 版本与 digest 绑定一致且可复算；
* 与 records 的执行语义一致。

**B4（BLOCK）run_root 复用默认拒绝与显式放行审计**

* 非空 run_root 默认拒绝；仅显式 override 放行；
* override_applied 与 run_closure 必锚定放行事实与 reason。

**B5（BLOCK）关键产物写盘同源性与路径门禁**

* `run_closure.json`、manifest、cfg_audit、env_audit、path_audit 等关键产物必须受控写盘；
* 禁止旁路 `open/json.dump/Path.write_*` 直写关键路径；
* 对抗式扫描必须给出命中列表并判定目标路径风险。

**B6（BLOCK）records 与 artifacts 语义旁路防护**

* artifacts 不得承载应由 schema+门禁约束的 records 语义锚点字段；
* 出现旁路字段或同名语义字段且未受控同步即 `FAIL`。

**B7（BLOCK）门禁失败语义 fail-fast 且可定位**

* 门禁失败必须包含 `gate_name`、`field_path`、reason 枚举；
* 门禁失败不得被吞并继续写盘。

**B8（BLOCK）override 封闭与审计锚定**

* override 字段必须白名单；未知字段拒绝；
* override_applied 必形成稳定审计段并可复算 digest。

### C. 注册表与扩展壳安全性（Registry & Extension Safety）

**C1（BLOCK）registry seal/lock 与不可注入性**

* seal 后不可 register/override；
* runtime_resolver 不接受外部 registry 注入；
* 对抗式检查环境变量、CLI、import side-effect 注入面。

**C2（BLOCK）impl 身份可审计**

* impl_identity 至少包含 impl_id/impl_version/impl_digest；
* 写入 records 与 run_closure 锚定；
* 生成逻辑可复算（canonical json + digest）。

**C3（BLOCK）capability 兼容性拒绝**

* impl 声明 capabilities（冻结友好类型）；
* 聚合 `impl_set_capabilities_digest`；
* 不兼容组合写盘前 fail-fast，原因可定位。

**C4（BLOCK）解析确定性与无外部副作用**

* resolve 不得触发隐式下载、动态生成、临时写盘、环境分支漂移；
* 同一输入在同环境下 resolve 结果确定。

### D. 路径与环境审计（Path & Environment Audit）

**D1（BLOCK）输出路径唯一派生与逃逸拒绝**

* path_policy 唯一派生 run_root；
* 拒绝目录逃逸、相对路径注入、符号链接污染；
* records/artifacts/logs 布局固定。

**D2（BLOCK）路径审计证据锚定**

* 记录原始输出输入与归一化 run_root；
* path_audit 产物或等价字段必须锚定 digest 至 run_closure；
* 只执行不留证据即 `FAIL`。

**D3（BLOCK）cfg_digest 输入域封闭与 cfg_audit 必产物**

* cfg_audit 必产出且可复算 cfg_digest；
* cfg_pruned_for_digest 与 canon_sha256 锚定至 run_closure。

**D4（BLOCK）时间戳与时区语义（UTC + Z）**

* run_closure 等时间字段必须 UTC 且以 `Z` 结尾；
* 写盘前校验可拒绝非规范值。

**D5（BLOCK）输入资产溯源（input_provenance）**

* 输入溯源字段与 digest 必存在；缺失必须显式 `<absent>`。

**D6（BLOCK）模型与外部资产溯源锚定**

* 至少满足本地 sha256 或远端 repo+revision 锁定并锚定 run_closure；
* 缺失必须 `<absent>`，禁止静默缺失仍成功写盘。

**D7（BLOCK）运行环境差异锚定（torch/cuda/gpu）**

* env_audit 必包含 torch/cuda/cudnn/gpu_name 等关键字段，不可用则 `<absent>`。

**D8（NON_BLOCK）依赖锁定与供应链基线**

* lock 文件或等价固定策略；
* 缺失则 `FAIL`（非阻断），除非未来升级为冻结要求并版本化。

**D9（BLOCK）危险动态执行与反序列化禁用**

* 禁止 eval/exec/compile 与 pickle/dill load 等；
* 命中即 `FAIL`（BLOCK）。

**D10（BLOCK）网络访问与隐式下载治理**

* 禁止未审计网络访问；若允许下载，必须锚定来源与 hash；
* 否则命中即 `FAIL`（BLOCK）。

**E4（BLOCK）RNG 语义可审计（seed 与确定性开关）**

## 版本化与向后兼容规则

### 必须版本化的对象

1. `contract_version`：冻结语义坐标系版本；
2. `schema_version`：records 结构版本；
3. `runtime_whitelist_version` 与 `policy_path_semantics_version`：策略语义版本；
4. `audit_policy_version`：本审计方案版本（含 BLOCK 集合与判定规则）；
5. `thresholds_semantics_version`（或等价）：统计口径版本（与 digest 同时存在更稳健）。

上述版本必须被锚定到 run_closure（或等价闭包字段），以支持历史解释。

### 允许变更而不升级 contract_version 的范围（可向后兼容）

满足以下条件可视为“向后兼容变更”：

1. **新增 records 字段**：仅 append，不改变既有字段含义、单位、枚举集合、默认值语义；
2. 新增 run_closure 字段：仅补充证据，不改变既有闭包判定逻辑；
3. 新增 artifacts 字段：不承载 records 语义锚点，不构成语义旁路；
4. 新增 impl：仅通过 registry 注入并写入 impl_identity，不触碰冻结面语义坐标系；
5. 增强审计与测试：新增 scripts/tests 不影响冻结面，仅提升闭环覆盖。

### 必须升级 contract_version 或语义版本的变更（否则视为语义漂移）

出现任一情况必须升级相应版本：

1. 既有字段含义、单位、类型、枚举集合、判决映射发生变化；
2. fail/mismatch/absent 语义发生变化或优先级改变；
3. policy_path 映射语义改变、whitelist 解释规则改变；
4. 统计口径与阈值选择规则改变（含默认参数注入导致的规则变化）；
5. `<absent>` 语义或缺失策略改变；
6. 任何将原本 NON_BLOCK 约束升级为冻结要求的动作（必须升级 `audit_policy_version` 并固化新的 BLOCK 集合）。

## 使用操作性规则

### 每次变更后的固定流程

1. 执行 L0：`scripts/run_all_audits.py`，产出聚合 `audit_report.json`；
2. 执行 L1：最小 pytest（`pytest -q`），确保关键门禁与闭包仍成立；
3. 若触发 E2E 规则（见 8 节），执行最小三条命令级 E2E；
4. 依据 BLOCK 规则输出 `FreezeSignoffDecision`：

   * 存在任一 BLOCK FAIL → `BLOCK_FREEZE`；
   * 否则 `ALLOW_FREEZE`。

### 单因素对比与消融实验就绪度判据（框架层）

消融实验就绪度要求以下锚点具备并可审计：

* contract_version、schema_version、policy_path_semantics_version、runtime_whitelist_version 一致；
* cfg_digest、thresholds_digest（及 metadata_digest）可复算且一致；
* 唯一变化项显式标注（通常是 impl_identity 或特定 override 审计段）；
* run_closure 与 bundle/manifest 可重放，失败语义闭环。

## E2E 触发规则（仅在静态不足或高风险变更时触发）

### 必须触发 E2E 的变更类型

满足任一条件即触发：

1. 修改 `main/diffusion/`（管线加载、hook、scheduler、latent_io、attention 相关）；
2. 修改 `main/core/records_io.py` 或 `main/policy/path_policy.py`（写盘与路径派生）；
3. 修改 `main/watermarking/fusion/`（阈值与判决口径）；
4. 修改 `configs/` 下 frozen/whitelist/semantics 任一文件；
5. 引入新的外部资产来源（模型下载策略改变、输入来源改变）。

### E2E 最小三条命令级步骤（模板要求）

若触发 E2E，审计报告必须给出三条最小步骤，并明确：

* 必测项与目的；
* 预期产物（records、run_closure、path_audit、cfg_audit、env_audit、manifest/bundle）；
* PASS/FAIL 判定标准（字段级）。
  未触发 E2E 时必须明确声明“静态与 L1 证据足以支撑判定，无需 E2E”。
