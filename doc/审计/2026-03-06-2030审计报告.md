- `main/watermarking/detect/orchestrator.py:1197-1206`：`_bind_raw_scores_to_content_payload()` 将 LF 状态写入 `score_parts["lf_detect_trace"]` 与 `score_parts["prc_latent_status"]`，未镜像到顶层 `score_parts["lf_status"]`
- `main/watermarking/detect/orchestrator.py:1208-1224`：HF 分支显式写入 `score_parts["hf_status"]`，形成不对称

**修复方案：** 在 `_bind_raw_scores_to_content_payload()` 中，当 `lf_trace["lf_status"]` 非空时，同时写入 `score_parts["lf_status"] = lf_trace["lf_status"]`。

**修复后的下一个阻断点：** detect 阈值仍为 test-only fallback；即使 `lf_status` 写盘完整，NP 语义仍不闭合。

### 【HIGH】BP 解码未收敛，LF 通道证据状态语义不一致

**目标字段（修复后应达到的状态）：**
- `detect_record.score_parts.lf_detect_trace.bp_converged == true`，或 `lf_status` 在不收敛时标记为 `"degraded"`

**当前实际状态：**
- `bp_converged = false`，`bp_iteration_count = 10`（达到最大迭代次数）
- `lf_score = 0.46875`（parity check 比率，接近随机猜测阈值 0.5 ± 1/√64）
- `lf_status = "ok"`（即使 BP 不收敛仍标记为 ok，状态语义不一致）

**代码级根因：**
- `main/watermarking/content_chain/low_freq_coder.py`：BP 解码当 `converged=False` 时仍走 parity check 路径，返回值未降级
- `ContentDetector._extract_channel()` 对 BP 未收敛的情况未做状态降级处理

**修复方案：** 在 `low_freq_coder.py` BP 解码路径中，当 `converged=False` 时将 `lf_status` 标记为 `"degraded"` 而非 `"ok"`；在 `ContentDetector` 中将 `"degraded"` 状态视为降级（影响融合权重但不完全阻断）。

**修复后的下一个阻断点：** 若 `lf_status` 被标记为 `"degraded"`，`ContentDetector` 对非 "ok" 状态的处理路径是否正确产出 lf_score；现有 `lf_score_drift_status` 字段的联动需验证。

### 【MEDIUM】detect 侧子空间一致性走代理路径，非主路径

**目标字段（修复后应达到的状态）：**
- `detect_record.content_evidence_payload.subspace_evidence_level == "primary"`
- `detect_record.content_evidence_payload.subspace_primary_path == true`

**当前实际状态：**
- `subspace_evidence_level = "surrogate"`，`subspace_primary_path = false`
- `subspace_consistency_status = "absent"`（不阻断 real 模式）

**代码级根因：**  
detect orchestrator 向 SubspacePlannerV1 传入 `final_latents`（最终 latent），而主路径需要逐步 JVP（每 timestep 的 Jacobian）；当前数据流中无逐步 JVP，主路径无法被激活（`main/watermarking/content_chain/subspace_planner.py`，evidence_level 写入逻辑）。这是"代码路径存在但数据流不通"的典型案例。

**修复方案：** 在 infer_runtime.py 中增加逐步 latent JVP 捕获选项（`capture_jvp_trajectory: true`），将逐步 Jacobian 传入 subspace_planner 激活主路径（推理时间约 3-5x 增加）。

**修复后的下一个阻断点：** 主路径下 `subspace_consistency_status` 可能变为 `"inconsistent"`，届时 `detect_runtime_mode` 可能从 "real" 降级，需真实运行验证。

### 【MEDIUM】latent 域 HF 分数缺失（`detect_hf_score = null`）

**目标字段（修复后应达到的状态）：**
- `detect_record.content_evidence_payload.detect_hf_score != null`

**当前实际状态：**
- `detect_hf_score = null`
- `detect_lf_score = 0.3631`（latent LF 成功）
- HF 分数由 image-domain sidecar 兜底

**代码级根因：**  
`extract_hf_score_from_detect_latents()` 在 `hf_basis` 为 None 时返回 `(None, "absent")`；plan_payload 中 `hf_basis` 在 surrogate 模式下未正确填充（`main/watermarking/detect/orchestrator.py` scorer 调用段）。

**修复方案：** 检查 `subspace_planner.plan()` 输出的 `plan_payload.plan.hf_basis` 在 surrogate 模式下是否被填充；若为 None，显式标记 `detect_hf_score_absent_reason = "hf_basis_not_computed_in_surrogate_mode"`。该字段为诊断字段，不阻断主决策。

### 【MEDIUM】`decision_status` 语义枚举与论文目标不对齐

**当前实际状态：**  
`main/watermarking/fusion/interfaces.py:L46`：`Literal["decided", "abstain", "error"]`   
- `"abstain"`：内容证据 absent → 无法判决（`decision.py:218-241`）  
- `"error"`：输入非法或技术失败  
- 论文目标语义 `"rejected_insufficient_evidence"` 不存在于枚举中

**修复方案（可选）：** 在 `interfaces.py` 中将 `"error"` 细化为 `{"error", "rejected_insufficient_evidence"}`，分别承载技术失败与证据审计驱动拒绝两种语义，并保证向后兼容。

### 【LOW】模型路径使用相对路径，可移植性风险

**当前实际状态：**  
`paper_full_cuda.yaml: mask.semantic_model_path: "../models/inspyrenet/ckpt_base.pth"`

**代码级根因：**

## 四、需要真实 SD Workflow 才能判定的差距项

### SD-1：几何与噪声攻击后的 TPR 鲁棒性

**静态已确认前提：**
- `attack_protocol.yaml` 定义 rotate/resize/gaussian_noise/jpeg/crop 等攻击族；代码路径 `main/evaluation/attack_coverage.py` 具备按攻击条件分组计算指标的逻辑
- `LatentSyncTemplate`/`LatentSyncSD3V2` 已部署，理论上可应对 rotate/resize；uncertainty 门控在 `uncertainty > 0.5` 时返回 `status="mismatch"`（`main/watermarking/geometry_chain/sync/latent_sync_template.py:981-995`）

**无法静态判断的原因：**  
施加 rotate 后 `sync_status` 是否仍为 `"ok"` 依赖真实 pixel-space 攻击 + SD3 推理；`LatentSyncTemplate.validate_sync_inputs()` 中 `sync_unsupported_model_or_pipeline` 分支的 pipeline 兼容性不可静态预测。

**验收字段（真实运行后检查）：**
- `evaluate_record.metrics_by_attack_condition[*].tpr_at_fpr_primary`（各攻击族 TPR）
- `evaluate_record.metrics.fpr_empirical ≤ 0.01`
- `geometry_evidence_payload.sync_status == "ok"` 在各攻击条件下的覆盖率

---

### SD-2：多图像多 prompt 下 mask_digest 分布稳定性

**静态已确认前提：**  
当前单张图像（landscape prompt）`mask_stats.area_ratio = 0.194`（19.4% 前景覆盖率）。

**无法静态判断的原因：**  
在高度对称/平滑图像（纯色或纯文字）上 inspyrenet 可能产出极小或空 mask，导致 HF 选点集退化。

**验收字段：**
- 100 个不同 prompt 的 `embed_record.content_evidence.mask_stats.area_ratio` 分布（期望 > 0.1）
- `embed_record.content_evidence.mask_stats.connected_components` 分布

---

### SD-3：primary subspace 路径下 `detect_runtime_mode` 稳定性

**静态已确认前提：**  
当前 `detect_runtime_mode="real"` 不依赖主路径 JVP（`subspace_consistency_status="absent"` 不阻断）；主路径代码分支存在（evidence_level="primary" 时），但当前未被激活。

**无法静态判断的原因：**  
若修复 surrogate → primary 后，`subspace_consistency_status` 可能变为 `"inconsistent"`，届时 `detect_runtime_mode` 降为 fallback——静态无法预测真实 JVP 下 embed 与 detect 子空间的一致性。

**验收字段：**
- `detect_record.content_evidence_payload.subspace_evidence_level = "primary"`（or "hybrid"）
- `detect_record.content_evidence_payload.subspace_primary_path = true`
- `detect_record.detect_runtime_mode` 仍为 `"real"`

## 五、消融实验可行性评估

### 配置基础

`configs/ablation/paper_ablation_cuda.yaml` 定义消融开关面板（`ablation.enable_*: null`），与 `paper_full_cuda.yaml` 共享相同基础配置。各模块通过 `_get_ablation_normalized()` 统一响应开关（`main/watermarking/detect/orchestrator.py`，`main/watermarking/embed/orchestrator.py`）。

### 逐维度评估

| 消融维度 | 可行性 | 说明 |
|----------|--------|------|
| `enable_content=false` vs baseline | ❌ **不可行（论文级对比）** | content 禁用时 `fusion_result.decision_status="abstain"`，所有样本放弃判决，TPR/FPR 数值对比无意义（TPR=0，reject_rate=100%）。仅可用于验证系统拒判语义，不能与 baseline 做 TPR/FPR 对比。 |
| `enable_geometry=false` vs baseline | ⚠ **有限可行** | 几何链禁用不阻断内容链主决策，可评估几何同步模块的边际收益。当前 rescue_rate=0.0（无攻击场景），需在有几何攻击的场景下才能量化有意义的消融差异。 |
| `enable_hf=false` vs baseline | ✅ **可行** | HF 禁用时 `ContentDetector` 退化为 `lf_only_when_hf_absent_v1` 规则（`main/watermarking/content_chain/content_detector.py:335-358`），可量化"仅 LF"与"LF+HF"的差异。当前 HF 归一化后权重极大，消融预期显著降低 content_score，探察 LF 单独判别力有效。 |
| `enable_lf=false` vs baseline | ❌ **不可行（论文级对比）** | `ContentDetector.extract()` 对 LF 具有主导性依赖：`lf_status ∈ {mismatch, failed, absent}` 时整体退出非 `ok`（`main/watermarking/content_chain/content_detector.py:233-250`）。系统无真正独立的 "HF-only 可判决主链"，关闭 LF 实质上等于关闭内容主链，不是有效的单因素消融。 |
| `enable_anchor=false` vs baseline | ⚠ **有限可行** | attention anchor 为辅锚点，禁用后主同步路径仍可工作；`geo_trusted` 条件变 False → rescue band 被阻断。当前无攻击场景 rescue_rate=0 → 消融无可观测差异。**需在有几何攻击的场景下才能量化 anchor 的边际贡献。** |

### 消融实验整体评估

消融框架**代码层面完备**（开关已实现，各模块正确响应），但当前实验条件不满足：

1. 无攻击场景导致 TPR=1.0（天花板效应），各消融变体难以区分差异
2. FPR 超标（3x），统计基准不稳
3. 30 个样本量使消融对比的置信区间极宽

**建议在以下条件下重新运行消融实验：**
- 每个攻击族 × 每个消融变体：至少 30 正样本 + 30 负样本
- FPR 校准负样本：≥ 100 个（不同 prompt）

## 八、总评

当前 `colab_run_paper_full_cuda` 产物表明：系统能够在单次 Colab workflow 下打通语义掩码、子空间规划、内容检测、几何检测、校准与评估的若干链路，各核心创新组件（subspace_projection、T2SMark、PRC、geometry sync、attention anchor）初步可达。

但以论文研究目标要求的"端到端可审计、可拒绝、遵循 NP 范式的证据体系"衡量，存在以下七个结构性缺口：

1. **evaluate 无攻击覆盖**：TPR@FPR=1.0 仅在无攻击条件下成立，对几何/噪声攻击鲁棒性无任何外部有效性
2. **detect 无 `final_decision` 字段**：最终判决缺少冻结落盘口径，判决对象已生成但未以目标字段名写出
3. **detect NP 语义失效**：fusion 判决使用 test-only threshold fallback（0.01），而非校准工件阈值（78.545），两者相差约 5 个数量级；current `is_watermarked=true` 的判决依据不具论文有效性
4. **发布核心无正负样本闭环**：`configs/ + main/` 本身不提供 calibrate/evaluate 所需输入路径，当前统计可计算性依赖辅助脚本
5. **FPR 超标 3x**：30 个负样本下 NP 校准统计功效不足，实测 FPR=0.0333 > target=0.01
6. **`score_parts.lf_status` 未落盘**：LF 检测算法实际成功，但顶层字段口径缺失
7. **BP 解码不收敛且状态未降级**：LF 通道在 BP 不收敛时仍标记 `lf_status="ok"`，状态语义与算法实际不一致

据此，本项目当前更接近**"研究原型在辅助 workflow 下可出结果"**，尚不能认定为**"发布核心已对论文目标实现自洽闭环"**。上述七项中前四项属于结构性根本缺口，需在发布前逐一修复并重新运行完整 Colab workflow 验证。
