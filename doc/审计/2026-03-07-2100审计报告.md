### NEW_BLOCKER_1：detect 端 `plan_digest_expected` 使用 fallback plan 的 digest，而非真实子空间 plan

**位置：** `main/watermarking/detect/orchestrator.py:1761-1820`（`_resolve_expected_plan_digest`）

**影响链路：** A、B、C

**说明：**

`_resolve_expected_plan_digest(input_record)` 在 input_record 无直接 `plan_digest` 时，会尝试从以下路径读取：

```
input_record.content_evidence.injection_site_spec.injection_rule_summary.plan_digest
```

当前 embed 输出的 `injection_rule_summary.plan_digest = "4940ded..."` 是 **latent_direct_fallback** 计划的摘要，并非语义掩码驱动的子空间计划。detect 端以此作为 `expected_plan_digest`，导致 `plan_digest_expected = "4940ded..."` 而非 `null`，从而：

1. **绕过了 `forced_absent` 分支**（`forced_absent` 需要 `expected_plan_digest` 为空才触发），实际进入 `else` 分支
2. **掩盖了真实语义：** 从审计字段看，detect 侧认为"有期望的计划"，但该计划实为 fallback 计划，不具备子空间论文语义
3. **影响 ContentDetector 的判断路径：** `expected_plan_digest` 非空，ContentDetector 不走 `detector_no_plan_expected` 路径，而走正常检测链（然后因 lf_evidence 缺失而 absent/failed）

修复 Chain A（`parents[3]`）后，若 planner 能产出真实子空间 plan，此路径才能形成正确的 `expected_plan_digest` 绑定。

## 七、总审计结论

- 链路 D 的阻断是统计样本量不足（30 < 100）+ 分数语义问题（链路 A/C 修复后自然解决）

**结论二：当前产物能证明的内容**

- SD3.5 完整 pipeline 已跑通（`geometry_evidence_payload.status="ok"`, `geo_score=0.931`）
- 几何链（latent sync）正常工作
