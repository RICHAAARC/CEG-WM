# 项目风险与研究差距综合审计报告（整理版）

> 审计版本号：2026-03-01-2330  
> 审计方式：静态代码审计 + 既有产物复核  
> 审计目标：按“静态可确认差距项 / 真实 SD workflow 才能判定项 / 后期消融实验判断项”给出可执行整改结论。

## 静态可确认差距项

### 1) HIGH-CRITICAL：内容链在 embed 阶段缺少语义掩码输入，核心创新链路未闭环

**差距项说明**
- `SemanticMaskProvider.extract()` 在无 `image/latent/image_path` 输入时，固定返回 `mask_extraction_no_input`。
- embed 编排仅在存在输入图片路径时才构造 `content_inputs`，否则传入 `None`。
- `paper_full_cuda` 未配置 `embed.input_image_path`，导致该失败在论文配置下稳定触发。

**证据位置**
- 代码：`main/watermarking/content_chain/semantic_mask_provider.py`（`mask_extraction_no_input` 失败分支）
- 代码：`main/watermarking/embed/orchestrator.py`（`_build_content_inputs_for_embed()`）
- 配置：`configs/paper_full_cuda.yaml`（`embed` 段无 `input_image_path`）
- 产物：`outputs/colab_complete_workflow/colab_run_paper_full_cuda/records/embed_record.json`
  - `content_failure_reason: "mask_extraction_no_input"`
  - `status: "failed"`
  - `mask_digest/plan_digest/basis_digest: null`

**影响说明**
- “语义掩码 → 子空间规划 → HF/LF 证据”主链路无法执行，核心研究主张无法被实证支撑。

---

### 2) HIGH-CRITICAL：detect 阶段内容链同样缺少可计算主输入，检测无法进入 real 证据闭环

**差距项说明**
- detect 主流程里 `content_extractor.extract(cfg)` 未传入 `inputs`，与 embed 同类问题重复出现。
- detect 虽有 `_resolve_detect_image_path()`，但该路径用于 LF/HF 侧路，不是语义掩码提取的主输入路径。

**证据位置**
- 代码：`main/watermarking/detect/orchestrator.py`（主流程 `content_extractor.extract(cfg)` 调用）
- 代码：`main/watermarking/detect/orchestrator.py`（`_resolve_detect_image_path()` 与 LF/HF 检测分支）
- 产物：`outputs/colab_complete_workflow/colab_run_paper_full_cuda/records/detect_record.json`
  - `content_failure_reason: "mask_extraction_no_input"`
  - `content_evidence_payload.status: "failed"`

**影响说明**
- detect 内容证据无法恢复，后续 runtime 判定长期停留在 fallback。

### 3) HIGH-CRITICAL：检测端长期 fallback，未形成可用最终判决路径

**差距项说明**

- real 模式门控需满足 `detect_lf_status == "ok"`、`runtime_built`、`not synthetic_pipeline_runtime` 等条件；当前 LF 信号 absent，门控不成立。

**证据位置**

- 产物：
  - `outputs/colab_complete_workflow/audit_diagnostics/records/detect_record.json`
  - `outputs/colab_complete_workflow/colab_run_paper_full_cuda/records/detect_record.json`

**影响说明**
- 策略路径无法产出可解释最终决策，校准/评估结果仅具形式执行意义。

### 4) HIGH：几何同步不确定性过高，几何链稳定进入 mismatch

**差距项说明**

- 默认 `sync_strength = 0.01`，在当前扩散步数和攻击设定下疑似不足以提供稳定残留相关性。

**证据位置**
- 代码：`main/watermarking/geometry_chain/sync/latent_sync_template.py`
  - 不确定性门限分支（`uncertainty > 0.5`）
  - `_resolve_sync_strength()` 默认值 `0.01`
- 产物：
  - `outputs/colab_complete_workflow/audit_diagnostics/records/detect_record.json`
  - `outputs/colab_complete_workflow/colab_run_paper_full_cuda/records/detect_record.json`
  - 字段：`geometry_failure_reason: "sync_uncertainty_too_high"`

**影响说明**
- 几何证据无法稳定参与融合，降低双证据链抗攻击可信度。

## 真实 SD workflow 才能判定项

### 1) 语义掩码是否真正与生成图语义对齐

**为什么必须 workflow**
- 该项依赖真实模型推理输出、真实掩码模型推断结果和图像语义一致性，静态代码只能确认“会不会执行”，不能确认“语义是否正确”。

### 2) HF/LF 证据在攻击强度阶梯下的可分离性

**为什么必须 workflow**
- 需观测真实样本上 HF/LF 分数分布随攻击强度的变化曲线；代码实现本身无法证明统计判别力。

### 3) 几何锚点在 rotate/crop/resize 下的稳健性

**为什么必须 workflow**
- 需真实 attention map 与几何扰动后的 relation_digest 统计，静态审计无法覆盖运行期特征漂移。

### 4) NP 阈值控制下的 TPR@FPR 是否成立

**为什么必须 workflow**
- 需多样本、可重复实验矩阵和真实分数分布；单次或静态检查无法给出可信置信区间。

### 5) 相对基线（如 T2SMark）是否有统计显著优势

**为什么必须 workflow**
- 必须在统一攻击协议、统一指标和足够样本规模下进行对照实验，静态代码无法替代实测比较。

## 后期消融实验判断项

### 1) 有利条件（当前已具备）

- `experiment_matrix.ablation_variants` 已覆盖 `enable_geometry/enable_content/enable_anchor/enable_hf/enable_lf`。
- 白名单与开关归一化逻辑已支持消融参数。
- failure semantics 字段体系已具备，便于后续分原因统计。

### 2) 阻断条件（当前未解决）

- 内容链输入缺失导致多数变体可能同态失败（都落入 `mask_extraction_no_input`），消融差异不可解释。
- sidecar-disabled 使 HF/LF 普遍 absent，导致“禁用模块”与“模块本就无信号”难以区分。
- onefile 对校准记录的回填路径可能污染“失败语义 vs 分数语义”边界。

### 3) 结论

- 当前阶段**不建议产出正式消融结论**。  
- 应先完成 P0/P1 修复，再启动多样本消融矩阵。

## 建议修复顺序

### P0（立即修复，阻断级）

1. 修复 embed 内容链输入接入（确保 `SemanticMaskProvider` 获得有效 `inputs`）。
2. 修复 detect 内容链输入接入（避免 detect 侧再次触发 `mask_extraction_no_input`）。

**目标验收**
- `embed_record.content_result.status == "ok"`
- `detect_record.content_evidence_payload.status == "ok"`
- `mask_digest/plan_digest` 非空且可复算

### P1（高优先级）

3. 修复检测 real 模式门控链路，使 `detect_runtime_mode` 从 fallback 进入 real。
4. 调整并验证几何同步参数（含 `sync_strength`），降低 `sync_uncertainty_too_high` 占比。
5. 明确 onefile 在 paper profile 的校准记录策略，避免结构性失败被分数化进入正式校准样本。

**目标验收**
- `detect_runtime_is_fallback == false`（达到预设比例阈值）
- `geometry_failure_reason=sync_uncertainty_too_high` 占比显著下降

### P2（统计闭环）

6. 扩充样本规模（最少 30，建议 100+）并补齐 ground truth 字段。
7. 重新执行校准与评估，输出可用 `TPR@FPR`、置信区间与攻击分组指标。

**目标验收**
- `n_samples >= 30`
- `missing_ground_truth` 拒绝率显著下降
- `tpr_at_fpr` 可计算且可复现
