## 二、项目目标完成度总判定

**总判定：部分完成，核心机制闭环仍未成立。**

上次审计提出的四项关键问题（HF 通道 embed/detect 不对齐、LF/HF 区域与子空间语义混叠、JVP 输入域不对称、impl_id 漂移），当前代码**均未修复**。此外，本次审计发现新的严重问题：

- LF 通道的**检测侧从未真正计算 LDPC 相关性**。从检测端（ContentDetector 主路径）走的是"轨迹能量统计"路径（`mean_std + mean_abs`），而非与编码端码字的相关匹配，这使得 LF 嵌入是否成功在检测端根本无法被正确验证。
- **TFSW 轨迹评分**（`detect_lf_score`、`detect_hf_score`）虽已被实现，但在当前代码中是**在 fusion 决策做出之后追加到 record 中**的，对最终判决无任何影响。
- 几何链的 FFT 同步模板（embed → detect）形成了真实闭环，但不提供几何参数估计，对大幅几何攻击无实质鲁棒性。

### 3.5 低频通道消息承载、LDPC、伪高斯采样

**状态：部分完成（Embed 侧存在，Detect 侧不对应）**

**证据锚点（Embed 端）**：  
- `channel_lf.py`，`apply_low_freq_encoding_torch()`（约 176-240 行）：
  - 从 `cfg.lf_plan_digest` 派生 LDPC 码字（`_derive_ldpc_codeword_from_cfg`）
  - 生成伪高斯幅度：`pseudogaussian_factors = torch.randn(n, generator=generator, ...)`
  - 加性注入：`encoded_coeffs = coeffs + strength * codeword * abs(pseudogaussian_factors)`
  
  这条路径存在且语义正确：有消息比特 → LDPC 码字 → 伪高斯采样 → 加性注入。

**证据锚点（Detect 端）**：  
- `content_detector.py`，`compute_lf_score()`（约 527-558 行）：  
  ```python
  observed = mean_std + mean_abs         # 轨迹统计量
  alignment = 1.0 / (1 + abs(observed - expected_delta))
  boost = 1.0 + max(0.0, embed_hint)
  return observed * alignment * boost
  ```
  检测端计算的是**轨迹能量统计量**，而非与 LDPC 码字的相关性。

- `detector_scoring.py`，`extract_lf_score_from_detect_trajectory()`（约 351-408 行）：
  ```python
  lf_coeffs = np.dot(phi, proj_np)
  coeffs_norm = float(np.linalg.norm(lf_coeffs))
  detect_lf_score = min(1.0, max(0.0, coeffs_norm / lf_score_threshold))
  ```
  TFSW 路径也只是**系数范数 / 阈值**，不是 LDPC 相关性。

**与目标不一致**：嵌入侧将特定 LDPC 码字注入 LF 系数，但检测侧从未计算"观测系数与预期码字的内积（白化相关）"，无法验证消息是否被正确写入。检测端的能量统计对任何 LF 修改都会有响应，无法区分"注入了正确码字"与"注入了随机信号"。

**此问题为本次审计新发现，是比上次审计更严重的闭环断链。**

### 3.6 高频通道截断式嵌入

**状态：已偏离目标（embed/detect 机制家族不匹配）**

**证据锚点（Embed 端）**：  
- `latent_modifier.py` 调用 `channel_hf.apply_hf_truncation_constraint()`（约 230-260 行）
- `channel_hf.py`，`apply_hf_truncation_constraint_torch()`（约 100-155 行）：  
  ```python
  threshold_value = torch.quantile(abs_coeffs, quantile)
  constrained_coeffs = torch.where(abs_coeffs >= threshold_value, coeffs, zeros)
  ```
  嵌入机制：**百分位阈值截断**（稀疏化），保留幅度最大的 HF 系数，清零其余。

**证据锚点（Detect 端）**：  
- `detect/orchestrator.py`，`_build_hf_detect_evidence()`（约 924-1020 行）→ 调用 `HighFreqTemplateCodecV2.detect()`  
- `high_freq_embedder.py`，`HighFreqTemplateCodecV2.detect()`：计算与 Rademacher 模板的白化相关，检测**模板加性注入**的痕迹。

**偏差总结**：

| 阶段 | 机制 |
|------|------|
| Embed | 百分位截断约束（sparsification） |
| Detect | Rademacher 模板白化相关（template matching） |

两者属于**不同水印家族**。截断约束不留下模板相关痕迹，模板检测器对截断约束产生的信号没有区分能力。

**此问题与上次审计一致，未修复。**

此外，`channel_hf.py` 顶部仍写死：
```python
HF_CHANNEL_IMPL_ID = "high_freq_template_codec_v2"  # 实际实现为截断约束，非模板注入
```
与 `high_freq_embedder.py` 中的 `HighFreqTemplateCodecV2`（模板注入 + 模板检测）共用同一 impl_id，**impl_id 语义漂移问题未修复**。

  - `extract_sync()`（约 285-380 行）：detect 侧用相同 seed 重建模板，计算 `_compute_template_match_metrics`（模板 vs 实际 FFT 的相关分数）。

1. **不提供几何参数估计或低阶几何反演**。当图像经过 crop/resize/rotate 后，embed 时固定的 FFT bins 不会自动对齐，模板相关性会下降，但系统不知道"降低了多少"也不尝试补偿。
2. 几何链提供的是"是否匹配"的能力，而不是"几何变换后如何恢复"的能力，实质是**固定频率注入 + 相关检测**，对较大几何攻击的恢复能力有限。
3. `AttentionAnchorMapRelationV2` 提供二级锚点证据，仅在 sync 成功时参与一致性评估（本身不产生 geo_score）。

### 3.11 融合层是否符合"内容主判、几何兜底"

**状态：已完成（融合逻辑实现正确）**

**证据锚点**：  
- `decision.py`，`NeumanPearsonFusionRule.fuse()`（约 220-340 行）：
  1. `content_score >= np_threshold` → True；否则 False
  2. 仅当内容为 False 且满足 rescue band 下界条件时，检查几何门控

## 四、按工程阶段审计

### 4.1 Embed 阶段

**真实做了什么**：
1. SD3 推理过程中，每步回调调用 `LatentModifier.apply_latent_update()`
2. LF 通道：投影 latent → 加性注入 `strength * LDPC_codeword * abs(pseudogaussian)` → 重建回 latent
3. HF 通道：投影 latent → 百分位截断稀疏化 → 重建回 latent
4. 几何链：FFT 同步模板注入（`embed_inject`）
5. 子空间规划：SVD of trajectory + JVP

**没有做什么**：
- LF 通道没有记录足够信息供检测侧重建 LDPC 相关性（检测侧需要知道哪个 plan_digest 对应哪个码字，但 `lf_plan_digest` 的传递需要在 detect 时精确匹配）
- HF 通道没有记录模板注入的痕迹（因为 embed 用的是截断，不是模板注入）

**是否与论文目标对齐**：LF 机制意图对齐，但检测端无法验证。HF 机制与论文声明（`tail_truncation_mode` + `high_freq_template_codec_v2`）存在歧义，截断与模板检测不是同一家族。

**是否存在语义污染**：无明显失败语义改写。`combined_status` 在任一通道失败时传播 failed 状态，未发现强行改写为 ok 的逻辑。

---

### 4.2 Detect 阶段

**真实做了什么**：
1. 重新执行子空间规划（detect 侧重建 plan/basis）
2. 从 embed 记录中读取 `lf_evidence`（含能量统计）
3. 调用 `HighFreqTemplateCodecV2.detect()` 获取 HF 模板相关分数
4. 调用 `ContentDetector.extract()` 计算 `content_score`（实际路径：轨迹能量统计驱动，而非 LDPC 相关）
5. 调用几何链：`_run_sync_module_for_detect()` 计算 `geo_score`（FFT 模板相关分数）
6. `NeumanPearsonFusionRule.fuse()` 输出唯一 `is_watermarked` 判决
7. **之后**再计算 TFSW 轨迹分数（`detect_lf_score`、`detect_hf_score`），追加至 record（对最终判决无影响）

**没有做什么**：
1. **没有验证 LF LDPC 码字**（最关键断链）
2. HF detect（模板相关）与 HF embed（截断约束）不对应
3. 几何链不进行参数估计，无法纠正几何变换后的图像
4. TFSW 轨迹评分虽被计算，但在不影响最终判决的情况下被追加，作用仅为审计记录

**是否存在失败语义改写**：`geometry_mismatch` 被降级为 `absent` 进入 content-only 路径，这是工程安全设计，但在记录字段中可见性不足。未发现 score=0 冒充 ok 的情况。

**是否存在旁路判决**：融合规则出口唯一，未发现旁路。

### 4.3 Calibrate 阶段

**无法完全静态确认**（需运行实验矩阵），但从配置与代码可静态确认以下：

- `paper_full_cuda.yaml`：`calibration.detect_records_glob: null`，须由实验矩阵注入。`experiment_matrix.seeds: [0, 1]`（仅 2 个 seed，不满足 FPR ≤ 1% 所需的 ≥100 样本要求）。
- `allow_threshold_fallback_for_tests: false`（生产禁止 fallback，正确）
- 校准是否能正确区分 failed/absent/mismatch 样本（避免偷渡为有效样本），静态上代码逻辑有 `allow_failed_semantics_collection` 开关，设置为 `true`——这意味着校准时允许收集失败语义样本，是否会混入阈值估计需仔细审查运行时逻辑。

**疑点（静态可发现）**：若 Detect 端的 LF/HF 分数本身不可靠（detect 不验证 embed 机制），则 calibration 估计的 NP 阈值也是基于"错误分数分布"的阈值，与目标不一致。

### 4.4 Evaluate 阶段

- 严格只读已校准阈值（`__thresholds_artifact__`），生产禁止 fallback。
- `experiment_matrix.seeds: [0, 1]`：仅两个 seed 的评测结果无法支撑论文级 FPR ≤ 1% 的统计主张（需 ≥ 100 对 positive/negative 样本）。
- 评测能否区分内容链性能 / 几何链性能 / 融合后性能：`content_evidence_payload`、`geometry_evidence_payload`、`fusion_result` 均在 record 中独立存在，评测脚本可分别提取，结构上支持分离分析。
- 当前主要评测问题在于**评测数据不足**，以及**被评测的 detect 分数本身不对应正确机制**（LF 能量统计 ≠ LDPC 验证）。

### 5.1 内容链主路径（Detect 端）

```
  └── _run_sync_module_for_detect()
```

### 5.3 融合判决唯一路径


融合出口唯一，无旁路。但其输入（content_score）的语义不正确，导致整链结论不可信。

## 七、论文级阻断项

### 阻断项 B-1：LF 检测不验证 LDPC 码字

**文件**：`main/watermarking/content_chain/content_detector.py:compute_lf_score()`

**描述**：论文方法的核心主张之一是 LF 通道通过 LDPC 码字 + 伪高斯采样注入可承载消息信息。但检测端 `compute_lf_score()` 计算的是轨迹能量统计（`mean_std + mean_abs`），与任何码字无关。TFSW 路径（`extract_lf_score_from_detect_trajectory`）计算系数范数，也与码字无关。**系统目前无法区分"注入了正确 LDPC 码字"与"注入了随机信号"。**

这直接阻断"LF 通道实现了消息承载式水印"的论文主张。

---

### 阻断项 B-2：HF 通道嵌入与检测机制不对应

**文件**：  
- Embed：`main/watermarking/content_chain/channel_hf.py:apply_hf_truncation_constraint_torch()`  
- Detect：`main/watermarking/detect/orchestrator.py:_build_hf_detect_evidence()` → `HighFreqTemplateCodecV2.detect()`

**描述**：嵌入采用百分位截断（sparsification），检测采用 Rademacher 模板白化相关（template matching）。检测器期望观测模板注入信号，但嵌入端没有注入任何模板。

即使两者都输出分数，分数含义无物理对应，`hf_score` 的数值在当前系统中仅为噪声，不反映 HF 水印的存在性。

---

### 阻断项 B-3：impl_id 语义漂移导致可审计性失效

**文件**：`channel_hf.py:HF_CHANNEL_IMPL_ID = "high_freq_template_codec_v2"` 与 `high_freq_embedder.py:HighFreqTemplateCodecV2`

**描述**：同一 `impl_id` 同时指代两种不同机制，违背"可审计、可复现"的核心工程约束。

---

### 阻断项 B-4：LF/HF 频域语义主张在代码中无支撑

**文件**：`subspace_planner_impl.py:build_region_index_spec_from_mask_v1()`

**描述**：若论文主张"对语义显著性区域（HF）与背景区域（LF）进行内容自适应子空间划分"，该划分目前是"前景 = 空间索引集合"而非"前景 = 高频频域子空间"，缺乏频域物理支撑。SVD 导出的 LF/HF basis 是代数子空间（principal components），与空间区域划分正交，两者没有被桥接。

---

### 阻断项 B-5：TFSW 轨迹评分不参与最终判决

**文件**：`detect/orchestrator.py`，`fusion_result` 在约 585 行生成，TFSW 分数在约 612 行之后计算

**描述**：若 TFSW（Trajectory Feature Space Watermarking）路径是论文方法的主验证路径（latent 空间精确评分），但当前代码实现中它在 `fusion_result` 已确定后才被执行，对判决无意义。这使得论文中如果声称"基于轨迹特征精确评分"，将与代码实际判决路径不符。

---

### 阻断项 B-6：统计实验规模不足以支撑论文级 FPR 主张

**文件**：`paper_full_cuda.yaml:experiment_matrix.seeds: [0, 1]`

**描述**：FPR ≤ 1% 需要在 ≥ 100 个负样本上统计，当前配置仅 2 个 seed（2 对 positive/negative），离散 FPR 粒度为 1/2 = 50%，根本无法支撑任何 FPR 精度主张。

---

## 八、最小修复顺序

以下修复顺序以"先打通主链 → 再修统计口径 → 再补几何兜底"为原则：

### 修复步骤 1：统一 HF 通道家族（消除 B-2 & B-3）

**目标**：选择一种 HF 机制，使 embed 与 detect 成为同一家族。

**推荐方案**：保留 `HighFreqTemplateCodecV2` 的模板注入 + 模板检测闭环：
- 修改 `latent_modifier.py` 的 HF 分支：不再调用 `channel_hf.apply_hf_truncation_constraint()`，改为调用 `HighFreqTemplateCodecV2.embed()`（Rademacher 模板加性注入）
- `_build_hf_detect_evidence()` 已调用 `HighFreqTemplateCodecV2.detect()`，不需要修改
- 删除或重命名 `channel_hf.py` 中的 `HF_CHANNEL_IMPL_ID`，消除 impl_id 冲突

**消除的阻断项**：B-2、B-3

---

### 修复步骤 2：修复 LF 检测端，使其验证 LDPC 码字（消除 B-1）

**目标**：detect 端计算真实的 LDPC 码字相关性，而非能量统计。

**方案**：  
- 在 ContentDetector（或 TFSW 路径）中，从 `lf_basis` 取出 `projection_matrix`，从 `cfg.lf_plan_digest` 重建同一 LDPC 码字（逻辑已在 `_get_ldpc_codeword_bits()` 中存在，可复用），计算 `dot(lf_coeffs, codeword)` 作为主评分，替换 `compute_lf_score` 的能量统计路径。
- 改造 `extract_lf_score_from_detect_trajectory()`，在计算系数后做码字内积（白化相关），而非只取范数。

**消除的阻断项**：B-1

---

### 修复步骤 3：将 TFSW 评分移至 fusion 决策之前（消除 B-5）

**目标**：若 TFSW z_{t_e} 精确评分是论文主验证路径，需在 `fusion_result` 计算之前得到 `detect_lf_score`、`detect_hf_score`，并将其作为 `lf_score`/`hf_score` 输入 ContentDetector。

**方案**：在 `detect/orchestrator.py` 中，将 TFSW 评分块（约 612-680 行）移至 `content_result = impl_set.content_extractor.extract()` 之前，并将 `detect_lf_score` 注入 `detector_inputs["lf_score"]`。

**消除的阻断项**：B-5（前提：步骤 2 已修复使 TFSW 分数语义正确）

---

### 修复步骤 4：拆分 LF/HF 两套定义（修复 B-4 、改善方法论有效性）

**目标**：使"语义掩码区域→频域子空间"有物理连贯的解释。

**最小修复**：在方法论叙事层修正：将 `hf_region = mask_true` 重新解释为"语义主体区域（用于 HF 嵌入位点路由）"，而 SVD 子空间的 leading/trailing vectors 解释为"主变化方向（LF）/ 次变化方向（HF）"，在文章中说明两套定义的不同语义边界，不再声称两者等价。

**或**：工程上拆分 `region_index_spec`（路由区域）与 `spectral_subspace_spec`（频域基底），使它们分别承担不同角色。

**消除的阻断项**：B-4（方法论层修复）

---

### 修复步骤 5：扩展实验 seed 规模（消除 B-6）

**方案**：将 `experiment_matrix.seeds` 扩展为 `range(100)` 或 `range(200)`，确保 positive 与 negative 各有 ≥ 100 对，满足 FPR ≤ 1% 的离散粒度要求。

**消除的阻断项**：B-6
