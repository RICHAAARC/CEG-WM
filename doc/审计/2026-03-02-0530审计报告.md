### 1）HIGH-CRITICAL：embed 端语义掩码模型权重缺失 → 核心创新机制未激活

**现象与证据**

真实 Colab SD3.5-medium 运行的 embed_record：
- `content_evidence.content_failure_reason: "saliency_source_model_unavailable"`
- `content_evidence.audit.saliency_source_attempted: ["model_v2"]`，`fallback_used: false`
- `content_evidence.mask_digest: null`，`plan_digest: null`
- `injection_rule_summary.injection_mode: "latent_direct_fallback"`（而非 `"subspace_projection"`）

注意：`injection_status: "ok"`，`injection_metrics.delta_norm_mean: 4.625`，说明 LF/HF 注入**确实发生于真实 SD3.5 28 步生成**，但使用**固定 fallback plan**（`run_embed.py:330–352`），而非语义掩码引导的自适应子空间。

**代码级根因（静态确认）**

- `semantic_mask_provider.py:798` → `_probe_model_v2_availability()` 检查 `.pth` 文件是否存在，Colab 环境中未下载返回 `False`
- `smp.py:295` → 当 `source_selected == SALIENCY_SOURCE_MODEL_V2` 且 `not model_available` 且 `not fallback_used` → 返回 `status="failed"` + `content_failure_reason="saliency_source_model_unavailable"`
- `run_embed.py:330–352` → `plan_digest` 为 `None` 时进入 fallback 分支，注入使用固定 rank=8 计划

**与研究目标的根本性冲突**

论文核心创新："根据每张图像的语义显著性**动态选择自适应流形区域**"。
当前状态：真实 SD 注入已发生，但注入到**非自适应的固定子空间**，研究核心机制未在任何一次产出中真实生效。

**整改行动**

在 Colab 运行时下载 `inspyrenet_plus_ultra.pth` 至 `configs/paper_full_cuda.yaml::mask.semantic_model_path` 对应路径。
验收：`embed_record.injection_rule_summary.injection_mode == "subspace_projection"`；`mask_digest != null`。

### 2）HIGH-CRITICAL：detect 端图像路径跨环境不可解析 → 内容证据始终 `mask_extraction_no_input`

**现象与证据**

真实 detect_record：
- `content_evidence_payload.content_failure_reason: "mask_extraction_no_input"`
- `detect_runtime_mode: "fallback_identity_v0"`，`detect_runtime_is_fallback: true`
- `score_parts.lf_absent_reason: "image_domain_sidecar_disabled"`，`hf_absent_reason: "image_domain_sidecar_disabled"`

**代码级根因（静态确认）**

`run_onefile_workflow.py:745` 已将 `--input embed_record.json` 传入 detect 阶段，`_build_content_inputs_for_detect`（`orchestrator.py:1230`）也已支持 image → latent → image_path 的优先级读取。真正的失败原因是：

- embed_record.json 中记录的生成图像路径为 **Colab GPU 环境路径**（如 `/content/...`），在后续运行的 detect 阶段，该路径在当前文件系统中不可解析（`path.exists()` 返回 `False`）
- `_resolve_detect_image_path`（`orchestrator.py:1210–1228`）遍历所有候选路径后返回 `None`
- `SemanticMaskProvider.extract(cfg, inputs=None)` → 第一道门控 `inputs is None` → `mask_extraction_no_input`

配置加剧因素：
- `paper_full_cuda.yaml:detect_runtime.image_domain_sidecar_enabled: false` → HF/LF 双 absent，即使路径可解析也无法产生 LF/HF 分数

**影响**

detect 端内容链路无法产出任何有效证据，核心创新的实证支撑缺失，后续判定长期停留在 fallback。

**整改行动**

在 Colab 工作流中，embed 阶段产出图像后将图像路径显式写入 embed_record 或通过 CLI 参数注入，确保 detect 阶段能在同一 Colab 文件系统环境中解析到该路径。
验收：`detect_record.content_failure_reason == null`；`mask_digest != null`。

### 3）HIGH：ContentDetector 输入域已补齐透传，但上游证据仍不足

**现象与差距描述**

即使第 2 条的图像路径问题被修复（路径可解析），`_build_content_inputs_for_detect` 产出的 `{"image_path": ..., "input_source": "image_path"}` 只能满足 `SemanticMaskProvider` 的 mask 提取需求，**不能满足 ContentDetector 的检测需求**。

`ContentDetector.extract()` 核心期望输入（`content_detector.py:75–130`）：
- `expected_plan_digest`（或 `plan_digest_expected`）
- `observed_plan_digest`
- `lf_evidence`、`hf_evidence`、`statistics`、`injection_evidence`

当前代码已在 `_build_content_inputs_for_detect` 中补充 `expected_plan_digest / observed_plan_digest / lf_evidence / hf_evidence / statistics / injection_evidence` 的透传；但 embed 侧产物大多数场景仍未形成可直接复用的 `lf_evidence/hf_evidence` 实证字段，导致 detect 侧多数情况下仍无法产出稳定的 `lf_score/hf_score`。

**影响**

已完成输入域打通后，剩余瓶颈转移为“上游证据生成与可传递性”，即便路径可解析也可能仅获得 absent/mismatch 语义。

**整改行动**

继续推进 detect 端两阶段闭环：
1. 第一阶段（路径可解析）：保证 `image_path` 在同一运行环境可读取。
2. 第二阶段（证据可计算）：在 embed/detect 之间建立可复用的 LF/HF 统计证据载荷。

验收：`detect_record.content_evidence_payload.expected_plan_digest` 或等价字段存在；且在非 sidecar-disabled 条件下，`lf_status` 可进入 `ok`。

### 9）MEDIUM：sidecar-disabled 路径的正式校准样本携带合成 score=0.0

**当前代码状态**

`run_onefile_workflow.py:991` 中，`calibration_sample_is_synthetic_fallback: true` 标记仅写入诊断文件（`detect_record_for_calibration_diagnostic.json`）；正式校准文件（`detect_record_for_calibration.json`）已移除该标记，替换为 `calibration_sample_usage: "formal_with_sidecar_disabled_marker"`（`run_onefile_workflow.py:1008–1009`）。标记分离本身正确。

**残留风险**

正式校准文件中的 `score` 值来源于：
```python
diagnostic_score = diagnostic_content_payload.get("detect_lf_score")
if not isinstance(diagnostic_score, (int, float)):
    diagnostic_score = 0.0
```
在 `sidecar_enabled: false` 条件下 `detect_lf_score` 为 `None`，因此 `score=0.0` 是合成默认值，而非真实检测信号。该样本以 `score=0.0` 直接进入正式 NP 校准，影响阈值计算有效性。

**影响**

NP 校准的输入分布被一个无意义的零分值污染，阈值结论不可用于论文正式声明。

**整改行动（P1-3）**

在校准端增加可配置严格过滤：`calibration.exclude_formal_sidecar_disabled_marker=true` 时过滤该类样本；默认保持 `false` 以保障 onefile 连通性。

### 当前阻断条件（消融结论不可产出）

**阻断 1：内容链双失败导致消融无差异**

- embed 端：inspyrenet 模型缺失 → 所有消融变体注入使用同一 fallback plan，"启用内容"与"禁用内容"实质等价
- detect 端：embed_record 中的图像路径在 Colab 环境跨阶段不可解析 + ContentDetector 输入域不匹配 → HF/LF 双 absent →"禁用 HF/LF 模块"与"模块本就无信号"无法区分

**阻断 2：ground truth 缺失 → 消融指标不可计算**

所有消融变体的 evaluate 均为 `missing_ground_truth=1.0`，`tpr_at_fpr_primary=null`。

**阻断 3：融合 `decision_status="error"` → 消融拒绝口径不可比**

禁用某链路 → mismatch → `"error"`，不同消融组的拒绝行为无法按研究语义区分和对比。

**结论：当前阶段不建议产出正式消融结论。应先完成 P0/P1 修复，再启动多样本消融矩阵。**
