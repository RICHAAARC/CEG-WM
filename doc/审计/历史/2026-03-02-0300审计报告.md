## 静态可确认差距项

### 1) HIGH-CRITICAL：embed 端内容链输入路径未配置，语义掩码主链路无法执行

**差距项说明**

- `configs/paper_full_cuda.yaml` 的 `embed:` 段无 `input_image_path` 字段。
- `embed/orchestrator.py:135` 调用 `_build_content_inputs_for_embed(cfg)` → `_resolve_embed_input_image_path(cfg)` → 候选列表（`__embed_input_image_path__`、`input_image_path`、`embed.input_image_path`）全部为 None → 返回 None。
- 因此 `content_inputs=None` → `SemanticMaskProvider.extract(cfg, inputs=None)` → 第一道门控（`inputs is None`）直接返回 `content_failure_reason="mask_extraction_no_input"`。

**证据位置**

- 代码：`main/watermarking/embed/orchestrator.py:135-138`（`content_inputs=None` 分支）
- 代码：`main/watermarking/embed/orchestrator.py:904-946`（`_resolve_embed_input_image_path` 候选链）
- 配置：`configs/paper_full_cuda.yaml`（`embed:` 段无 `input_image_path`）
- 产物：`records/embed_record.json`
  - `content_evidence.content_failure_reason: "mask_extraction_no_input"`
  - `content_evidence.mask_digest: null`

**影响说明**

"语义掩码 → 子空间规划 → HF/LF 证据"主链路在 paper 配置下稳定失败，核心创新链路未形成可运行闭环。

### 2) HIGH-CRITICAL：detect 端内容链同类失败，但根因已变——依赖 image_path 解析路径

**差距项说明**

- detect 端 `detect/orchestrator.py:170` 已正确调用 `_build_content_inputs_for_detect(cfg, input_record)`，传入结果给 extractor。
- `_build_content_inputs_for_detect` → `_resolve_detect_image_path(cfg, input_record)` → 若 input_record 无已生成图像路径且 cfg 无显式 `detect_image_path`，返回 None → inputs=None。
- 在当前 `paper_full_cuda` + Colab 模拟运行下，embed 未产出真实图像文件，detect 侧同样无法解析到 image_path，导致与 embed 端同样的 `mask_extraction_no_input`。
- **根因不是代码实现错误，而是 embed 未产出真实图像输出路径传递给 detect。**

**证据位置**

- 代码：`main/watermarking/detect/orchestrator.py:1226-1248`（`_build_content_inputs_for_detect`）
- 代码：`main/watermarking/detect/orchestrator.py:1210-1224`（`_resolve_detect_image_path`）
- 产物：`records/detect_record.json`
  - `content_evidence_payload.content_failure_reason: "mask_extraction_no_input"`

**影响说明**

detect 内容证据无法恢复，后续 runtime 判定长期停留在 fallback。解决路径依赖于 embed 产出真实图像。

### 3) HIGH-CRITICAL：detect_runtime_mode 永远 fallback，策略路径无法给出可解释最终判决

**差距项说明**

- 进入 `real` 模式的门控（`detect/orchestrator.py:700-705`）：
  1. `detect_lf_status == "ok"`（当前：absent，由 sidecar 禁用导致）
  2. `subspace_consistency_status != "inconsistent"`
  3. `runtime_built`（由 `__pipeline_runtime_meta__.status` 控制）
  4. `not synthetic_pipeline_runtime`
- 四个条件全部不满足，`detect_runtime_mode` 固定停留在 `fallback_identity_v0`。
- 根本障碍：`detect_runtime.image_domain_sidecar_enabled: false` 使条件 1 永远不成立。

**证据位置**

- 代码：`main/watermarking/detect/orchestrator.py:698-705`（real 模式门控）
- 配置：`configs/paper_full_cuda.yaml:image_domain_sidecar_enabled: false`
- 产物：`records/detect_record.json`
  - `detect_runtime_mode: "fallback_identity_v0"`
  - `detect_runtime_is_fallback: true`
  - `final_decision: null`

**影响说明**

策略路径无法产出可解释最终决策，校准/评估结果仅具形式执行意义。

### 4) HIGH：融合层 `decision_status="error"` 破坏 NP 统计口径，样本不可分类

**差距项说明**

- `_build_mismatch_fusion_decision`（`detect/orchestrator.py:2183`）在任何 content/geometry mismatch 下，统一产出 `decision_status="error"` + `is_watermarked=None`。
- 这把"可拒绝证据链"与"流程异常/不可评估"混为一类，破坏了 NP 统计框架对样本的三态（`accepted / rejected / invalid`）分类。
- 在消融实验中，"禁用某链路"本应被归类为 `rejected`（有信息量的拒绝），但当前被归类为 `error`（无信息量的异常），使不同消融配置的拒绝口径不可比。

**证据位置**

- 代码：`main/watermarking/detect/orchestrator.py:2150-2193`（`_build_mismatch_fusion_decision`）
- 产物：`records/detect_record.json`
  - `fusion_result.decision_status: "error"`
  - `fusion_result.is_watermarked: null`
  - `fusion_result.audit.failure_reason: "geometry_mismatch"`

**影响说明**

影响后续消融比较的一致性。建议为"链路禁用/链路失败"场景引入独立的 `decision_status="rejected_insufficient_evidence"` 语义。

## 真实 SD workflow 才能判定项

### 1) 语义掩码是否真正与生成图语义对齐

**为什么必须 workflow**

该项依赖真实模型推理输出与图像语义一致性。静态代码只能确认"执行路径"，不能确认"语义是否正确对齐"。

### 2) HF/LF 证据在攻击强度阶梯下的可分离性

**为什么必须 workflow**

需观测真实样本上 HF/LF 分数分布随攻击强度的变化曲线；代码实现本身无法证明统计判别力。

### 3) 几何锚点在 rotate/crop/resize 下的稳健性

**为什么必须 workflow**

需真实 attention map 与几何扰动后的 `relation_digest` 统计，静态审计无法覆盖运行期特征漂移。

### 4) NP 阈值控制下的 TPR@FPR 是否成立

**为什么必须 workflow**

需多样本、可重复实验矩阵和真实分数分布；单次或静态检查无法给出可信置信区间。

### 5) `semantic_saliency_v2` 模型权重的真实推理可用性

**为什么必须 workflow**

`_probe_model_v2_availability()` 使用 `torch.load()` 读取 `.pth` 权重文件，静态无法判定权重格式与推理框架是否匹配；当前失败在 embed 侧（`mask_extraction_no_input`，优先于模型加载），需先修复输入路径再用真实 workflow 验证。

### 6) 相对基线（如 T2SMark）是否有统计显著优势

**为什么必须 workflow**

必须在统一攻击协议、统一指标和足够样本规模下进行对照实验，静态代码无法替代实测比较。

## 后期消融实验判断项

### 1) 有利条件（当前已具备）

- `experiment_matrix.ablation_variants` 已覆盖 5 个核心因子（`enable_geometry / enable_content / enable_anchor / enable_hf / enable_lf`）。
- Whitelist 已注册对应 `ablation_enable_*` arg_name，override 解析路径歧义已修复（本次）。
- `cfg_digest` 随 ablation flag 变化而改变，可区分实验组。
- `grid_summary` 已包含 `total / succeeded / failed` 字段，便于批次监控。
- failure semantics 字段体系（`content_failure_reason`、`geometry_failure_reason`）已具备，便于分因失败统计。

### 2) 阻断条件（当前未解决）

- 内容链 embed/detect 输入缺失，导致所有消融变体在内容链维度同态失败（均落入 `mask_extraction_no_input`），"禁用内容"与"内容本就无法运行"难以区分，消融差异不可解释。
- `detect_runtime.image_domain_sidecar_enabled: false` 使 HF/LF 信号普遍 absent，导致"禁用 HF/LF 模块"与"模块本就无信号"的消融组无法对照。
- 融合层 `decision_status="error"` 与"可拒绝证据链的合法失败"混用，消融组间拒绝口径不可比。
- ground truth 缺失，消融矩阵的 TPR/FPR 指标无法计算，无法形成论文级量化结论。

### 3) 结论

**当前阶段不建议产出正式消融结论。** 应先完成 P0/P1 修复，再启动多样本消融矩阵。

## 建议修复顺序

### P0（立即修复，阻断级）

**P0-1：为 paper_full_cuda 配置添加 embed 端 input_image_path，或在 run_onefile_workflow.py 中注入该参数**

目标：使 `_build_content_inputs_for_embed` 返回非 None，`SemanticMaskProvider.extract` 能获得 `image_path` 输入。

实现选项：
- 选项 A（配置）：在 `paper_full_cuda.yaml` 的 `embed:` 段添加 `input_image_path: "<待生成图像路径>"` 占位，由 onefile workflow 动态注入。
- 选项 B（代码注入）：在 `run_onefile_workflow.py` 的 embed 阶段前，通过 CLI override 注入 `input_image_path=<path>` 参数。

验收：`embed_record.content_evidence.status == "ok"`、`mask_digest` 非 null。

**P0-2：为 detect 端提供对应的已生成图像路径（或 latent 路径）**

目标：使 `_resolve_detect_image_path` 能返回有效路径，content_inputs 非 None。

验收：`detect_record.content_evidence_payload.status == "ok"`、`mask_digest` 非 null。

---

### P1（高优先级）

**P1-1：评估 sidecar 的替代策略，使 detect_runtime_mode 能进入 real 模式**

- 如果 paper 真实 SD 路径可以提供 latent 注入，考虑在论文实验时开启 sidecar 或提供等价的 final_latents 证据。
- 验收：`detect_runtime_is_fallback: false`（至少在真实 SD 生成场景下）。

**P1-2：明确 onefile 校准记录构造策略，将合成 fallback 分数排除在正式校准样本之外**

- 为 `PAPER_SIDECAR_DISABLED_FALLBACK` 路径添加显式标记，确保使用 `calibration_sample_is_synthetic_fallback: true` 的样本不进入正式 NP 阈值计算。
- 验收：校准记录的样本来源字段 `calibration_sample_origin` 被评估端检查，合成样本被过滤。

**P1-3：引入融合层"证据不足拒绝"语义，与异常状态分离**

- 建议新增 `decision_status="rejected_insufficient_evidence"` 用于"链路禁用/信号 absent 导致的合法拒绝"，保留 `"error"` 用于流程异常。
- 验收：消融组在禁用某链路时，`decision_status` 产出 `rejected_insufficient_evidence` 而非 `error`。

---

### P2（统计闭环，论文级）

**P2-1：扩充样本规模（最少 30，建议 100+）并补齐 ground truth 字段**

验收：`n_samples >= 30`、`missing_ground_truth` 拒绝率显著下降、`tpr_at_fpr` 可计算且可复现。

**P2-2：调整几何同步参数，降低 sync_uncertainty_too_high 占比**

验收：在真实多样本 SD 生成下，几何链可用率达到论文声明阈值。

**P2-3：对照基线（T2SMark）在统一攻击协议下产出显著性检验结论**

验收：可报告攻击分组下的 TPR 置信区间与基线 p 值。
