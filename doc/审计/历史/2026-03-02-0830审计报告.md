>
> 实际链路为：
> 1. `run_onefile_workflow.py` sidecar_disabled 路径将 source detect_record 的 `status` 强制写为 `"ok"`, `score=0.0`（`_maybe_recover_score_from_sidecar_disabled`）
> 2. `_generate_ground_truth_bundle()`（`run_onefile_workflow.py:928-1028`）复制该 source record → 发现 `status=="ok"` → 强制覆写负样本 score：`score = float(-1.0 - pair_index * 1e-6)`（**line 1000 显式注入**）
> 3. 校准端 `load_scores_for_calibration()`（`orchestrator.py:2760-2857`）**严格拒绝**所有 `status != "ok"` 或 `not isinstance(score, (int, float))` 的记录 —— **从不替换 null 为 -1.0**；若无有效 score 则抛 ValueError
> 4. 因此 -1.0 进入校准是 onefile bundle 的显式设计，而非校准代码的 null 补全逻辑
>
> **正确机制**：
> 1. sidecar_disabled → source status 被改写为 "ok", score=0.0
> 2. bundle 生成器：负样本 score 覆写为 -1.0（`run_onefile_workflow.py:1000`）
> 3. 校准拿到 30 个合法 float score（全为 -1.0 范围）
> 4. NP 99th 分位（`order_stat_index_0based=29`）= **-1.0**（所有负样本最大值）
> 5. 阈值 = -1.0 → 任何 score ≥ -1.0 均判有水印
> 6. `fpr_empirical = 0.0333 > target_fpr = 0.01`（超标 3.3×），NP 保证不成立
> 7. `calibration_mode: "real"` 与 `threshold_value: -1.0` 并存，两个字段语义矛盾

**修复：** 根本修复依赖 P0-1 → P0-2 → P0-3 链路打通使 content_score 有真实值。辅助修复：在 `_generate_ground_truth_bundle` 中对 sidecar_disabled 负样本打标，并在校准输入校验中拒绝 `calibration_sample_usage="formal_with_sidecar_disabled_marker"` 且 score 为 -1.0 的条目；同时应将 `calibration_mode` 在检测到退化 bundle 时降级为 `"degenerate"` 并抛出 warning。

**下一个阻断点：** 检测链有真实分数后，需验证 NP 分位规则（`quantile_rule: "higher"`, `target_fpr: 0.01`）能给出 `0 < threshold < 1` 的合理阈值。

### FAIL-1 [MEDIUM]：`run_evaluate_orchestrator` 在 n_pos=0 或 n_neg=0 时静默写出 None 指标，无强制中止

**类型：** 评估指标完整性

**根因（代码行号已核查）：**
```python
# detect/orchestrator.py:3230
fpr_value = float(fp / total) if total > 0 else None
# detect/orchestrator.py:3391
tpr_value = float(tp / pos) if pos > 0 else None
# detect/orchestrator.py:3590-3591
"fpr_empirical": fpr_value,    # 可能为 None
"tpr_at_fpr_primary": tpr_value,  # 可能为 None
```

- `run_evaluate_orchestrator` 在 n_pos=0 或 n_neg=0 时不抛出异常，不中止流程
- `tpr_value = None` / `fpr_value = None` 被写入 evaluate 结果文件
- 下游 signoff / freeze 校验读取该字段时如不显式处理 None，可能静默通过

**影响：** 在 detect chain 全部失败（score=null → 评估端去掉非 float 记录后 n_pos=0）的情况下，evaluate 产出的 JSON 中 `tpr_at_fpr_primary: null`，与"NP 框架成立"的声明矛盾但不会主动 fail-fast。

**修复：** 在 `aggregate_metrics()` 或 `run_evaluate_orchestrator` 入口处增加 n_pos ≥ 1 AND n_neg ≥ 1 的强制校验，校验失败时抛出 `ValueError` 并写 `evaluate_status: "degenerate"` 而非静默产出 None 指标。

### FAIL-5 [HIGH]：`assert_impl_allowed` 在所有 CLI 入口均未调用，whitelist 对 impl_id 无运行期门控

**类型：** 安全/可审计性 — impl 一致性保证缺口

**根因（代码核查）：**

`main/policy/runtime_whitelist.py:1338` 定义了 `assert_impl_allowed(whitelist, impl_identity)` 函数，用于校验 impl_id 是否在 `runtime_whitelist.yaml` 允许列表中。

**但该函数在所有 CLI 入口均未被调用：**
- `main/cli/run_detect.py`：未调用（仅调用 `bind_whitelist_to_record`，后者为元数据绑定，无门控语义）
- `main/cli/run_embed.py`：未调用
- `main/cli/run_calibrate.py`：未调用
- `main/cli/run_evaluate.py`：未调用

**实际 impl_id 校验路径只存在于测试（`tests/test_align_impl_whitelist_enforcement.py` 等），不在运行时。**

`build_runtime_impl_set_from_cfg`（`runtime_resolver.py:328`）的执行顺序：
1. `parse_impl_identity_from_cfg(cfg)` — 仅解析 ID
2. `resolve_impl_factories(identity)` — 从 sealed registry 解析 factory（阻止未注册 impl）
3. `build_impl_set(resolved, cfg)` — **impl 实例化（执行构造函数）**
4. `assert_impl_set_compatible(...)` — 能力兼容性检查（非 whitelist 检查）

Registry 密封确保了"未注册 impl 无法执行"，但 whitelist 管控的是"已注册的 impl 中哪些是论文合规 impl"。两者不等价。

**影响：** 若某 impl_id 在 registry 中但不在 `runtime_whitelist.yaml`，当前代码允许其正常执行、计算、生成产物，只在事后 `bind_whitelist_to_record` 时将 whitelist 元数据附在记录上——但不会阻止写盘。测试套件会发现不合规 impl（若有），但单独运行 `run_detect.py` 不会。

**修复：** 在 `build_runtime_impl_set_from_cfg` 返回前（impl 实例化之前最佳，或至少在 impl 首次被调用之前），调用 `assert_impl_allowed(whitelist, identity.as_dict())`。需要将 `whitelist` 传入 `build_runtime_impl_set_from_cfg` 或在 CLI 层在 `build_runtime_impl_set_from_cfg` 调用后、其他 impl 调用前插入 `assert_impl_allowed`。
