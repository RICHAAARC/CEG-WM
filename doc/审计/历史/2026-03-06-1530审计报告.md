### GAP-D1 [CRITICAL]：NP 统计基础完全失效（产物 + 代码双确认）

**目标字段**：
- `calibration_record.calibration_samples ≥ 30`
- `evaluate_record.fpr_empirical ≤ 0.01`

**真实产物字段**：
- `calibration_record.calibration_samples = 1`
- `calibration_record.calibration_summary.threshold_value = 78.54547775`（= 负样本 content_score，统计无意义）
- `evaluate_record.fpr_empirical = 1.0`
- `evaluate_record.tpr_at_fpr_primary = 1.0`
- `evaluate_record.n_pos = 1, n_neg = 1`
- `evaluate_record.evaluation_breakdown.confusion = {fn:0, fp:1, tp:1, tn:0}`

**代码根因**

`scripts/run_onefile_workflow.py`，函数 `_prepare_detect_records_with_minimal_ground_truth()`（约 line 1248）：

```python
# dual_branch 路径段（当 branch_neg_detect_record.exists() 时）：
pos_path = workflow_cfg_dir / f"detect_records_{stage_name}_gt_positive.json"
neg_path = workflow_cfg_dir / f"detect_records_{stage_name}_gt_negative.json"
_write_artifact_text_unbound(run_root, pos_path, ...)  # 写 1 个正样本
_write_artifact_text_unbound(run_root, neg_path, ...)  # 写 1 个负样本
return str((workflow_cfg_dir / f"detect_records_{stage_name}_gt_*.json"))
```

`pair_count` 的循环 `for pair_index in range(pair_count)` **仅在 clone fallback 路径执行**（line ~1285）；dual_branch 路径不循环，固定产出 1 pos + 1 neg，`pair_count=30` 被完全忽略。

**调用链**：
```
paper_full_cuda.yaml: calibration.minimal_ground_truth_pair_count = 30
  ↓ _prepare_stage_cfg_path("calibrate", pair_count=30)
  ↓ branch_neg_detect_record.exists() = True → 执行 dual_branch 路径
  ↓ 固定写 1 pos + 1 neg（忽略 pair_count=30）
calibrate 运行：n_neg = 1
  ↓
calibration_samples = 1，阈值 = 78.54（唯一负样本 content_score）
  ↓
evaluate：fp=1（负样本 score=78.54 = 阈值）→ fpr_empirical = 1.0
```

**修复方案**：

在 `_prepare_detect_records_with_minimal_ground_truth()` 的 dual_branch 路径中增加 `pair_count` 支持：
- 方案 A（推荐）：dual_branch 路径支持传入 `pair_count`，循环调用 `_run_dual_branch_embedding_and_detection()` 生成多对正负样本；
- 方案 B：在 `paper_full_cuda.yaml` 中配置 `minimal_ground_truth_prompts_file`，由 prompt 列表驱动批量生成；此时 `pair_count = len(prompt_list)`，可绕过当前 bug。

**修复后的下一个阻断点**：
1. 30 次 dual_branch 推理（约 30× SD3 embed+detect）时间成本较高
2. 修复后需验证 30 个负样本 content_score 是否呈均匀分布，而非聚集于单值
3. 校准完成后，evaluate 重打分使用 NP-canonical 阈值，`fpr_empirical` 才具统计意义

### GAP-B1 [CRITICAL]：LF 信道主检测路径（PRC belief propagation）失败（产物 + 代码双确认）

**目标字段**：
- `detect_record.score_parts.lf_detect_trace.lf_status == "ok"`（论文：PRC 解码成功）
- `detect_record.score_parts.lf_detect_trace.lf_score != null`（论文：来自 PRC bp decoded 得分）

**真实产物字段**：
- `score_parts.lf_detect_trace.lf_detect_path = "lf_coder_prc_latent"`
- `score_parts.lf_detect_trace.lf_status = "failed"`（line 60 of detect_record.json）
- `score_parts.lf_detect_trace.lf_failure_reason = "lf_insufficient_latent_dimension"`（line 58）
- `score_parts.lf_detect_trace.lf_score = null`（line 59）
- `score_parts.lf_score = 0.4703086`（line 64，**来自 trajectory 统计替代路径**，非 PRC 解码）

**代码根因**

`main/watermarking/content_chain/low_freq_coder.py:1329`（`detect_score()` 函数；**核验补正**：1530 报告原写 `detect_lf_from_latent()`，为函数演进后的旧名，当前代码对应函数为 `detect_score()`，失败条件与语义不变）：

```python
block_length = int(ldpc_spec.get("n", message_length))  # n = message_length × (1 + ecc_sparsity) ≈ 256
if len(flat_latents) < block_length:
    return None, {
        "status": "failed",
        "lf_failure_reason": "lf_insufficient_latent_dimension",
        ...
    }
```

`block_length = n` 由 LDPC 矩阵决定（`message_length=64, ecc_sparsity=3`，保守估计 `n≈256`）；而 `flat_latents` 来自 `lf_basis` 子空间选出的特征，维度不足 256，导致解码失败。

**替代路径激活** (`unified_content_extractor.py:183-191`）：
```python
if lf_score is None and lf_statistics.get("status") == "ok":
    lf_score = compute_lf_score(lf_statistics=...)  # 轨迹统计替代
```
lf_statistics 路径使用 trajectory evidence 统计（`mean_std + mean_abs` 与 `expected_delta` 对齐），因此 content_score 中的 lf_score 来自**轨迹统计代理**，而非论文声明的 PRC 信念传播解码。

**论文声明与实现差距**：
- 论文声明：LF 区域采用 PRC-Watermark 编码，信念传播解码器提供稳健证据
- 实际：PRC-latent 解码路径失败；lf_score 来自轨迹统计代理

**修复方案**：

选项 A（推荐）：扩大 `lf_basis` 维度，确保 `dim(lf_basis) ≥ block_length`
- 在 `SubspacePlannerImpl.plan()` 中增加断言/自动扩展：若 `lf_basis_dim < ldpc_block_length(message_length, ecc_sparsity)`，则扩展 lf_basis
- 相关文件：`main/watermarking/embed/subspace_planner.py`（需读取确认 lf_basis 当前生成逻辑）

选项 B：降低 LDPC 参数：减小 `ecc_sparsity` 使 `block_length` 缩减到 `lf_basis_dim` 以下
- 代价：降低 ECC 纠错能力，降低 LF 信道鲁棒性

**修复后的下一个阻断点**：
1. PRC-latent 路径通过后，需确认 PRC-解码得分是否正确传入 UnifiedContentExtractor 的 `lf_score` 计算路径（当前架构中 UnifiedContentExtractor 优先使用 `lf_evidence`，而非 PRC-latent 输出）
2. `_bind_raw_scores_to_content_payload` 的 lf_status 覆写问题仍需独立修复（GAP-B2）

### GAP-B2 [HIGH]：score_parts 字段内部矛盾（代码确认）

### GAP-C1 [HIGH]：原始 detect 判决使用 fallback 阈值（有重要补充）

**目标字段**：论文计算的最终 TPR/FPR 应使用 NP 校准阈值进行判决

**真实产物字段**：

- `fusion_result.audit.np_threshold = 0.01`（直接使用 target_fpr，非校准值）
- **顶层**：`detect_record.threshold_source = "np_canonical"`（evaluate 阶段写入，line 2457）

**重要补充（0000 报告未充分说明）**：

GAP-C1 存在两个层次：
1. **detect 时原始 fusion 判决**：使用 `fallback_target_fpr_test_only`（必然，因 calibrate 在 detect 之后执行）
2. **evaluate 阶段重打分**：evaluate 的 `thresholds_path` 被注入校准工件，evaluate 使用 NP-canonical 阈值（78.54）重新对每条样本执行比较，write back `threshold_source = "np_canonical"` 至 detect_record 顶层

但此 NP-canonical 阈值本身在统计上无效（来自 GAP-D1，仅 1 个负样本），因此 evaluate 的重打分统计结果（`fpr_empirical=1.0`）仍然无意义。

**核心关系**：GAP-C1 是 GAP-D1 的下游症状。修复 GAP-D1（calibration_samples≥30）才能使 GAP-C1 的阈值在统计上有效。

**代码根因（结构性）**：

`scripts/run_onefile_workflow.py:340-343`：
```python
if stage_name == "detect":
    overrides.append("allow_threshold_fallback_for_tests=true")
```
detect 执行时校准工件尚未生成，fallback 是结构性必然，不可避免。

**修复策略**：GAP-D1 修复后，evaluate 阶段已有正确逻辑重打分，GAP-C1 会自动改善。无需单独修复 detect 阶段的 fallback 行为。
