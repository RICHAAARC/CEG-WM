### P0-B：修复 HF 评分信号判别性（STATIC-2）

**文件：** `main/watermarking/content_chain/content_detector.py`  
**修改位置：** `ContentDetector.extract()` 方法，HF 分支内 `_extract_channel()` 调用之后

**问题根因：**  
`_extract_channel()` 的 dict 分支在 `hf_evidence` dict 不含 `hf_score` 键时返回 `hf_score=None`，同时忽略 `default_score`（图像域旁路值 `hf_score_raw`）。后续 `compute_hf_score()` 基于 SD3 推理轨迹 L2 范数计算分数，对所有 SD3 图像均约为 262，对水印存在与否无判别性。

**修复内容：**  
在 `_extract_channel()` 返回 `hf_score=None` 后，新增图像域旁路分数优先逻辑：若 `normalized_inputs.get("hf_score")`（即 `hf_score_raw`）是有效浮点数，则 `hf_score = max(0.0, hf_score_raw)`，设 `hf_status="ok"`，跳过 `compute_hf_score()` 的轨迹 L2 路径。`compute_hf_score()` 函数签名及实现不变，降级为 `hf_score_raw` 缺失时的备用路径。

**代码变更摘要：**

```python
# 修复前：直接进入 extract_high_freq_statistics + compute_hf_score 路径
hf_statistics = extract_high_freq_statistics(normalized_inputs)
if hf_score is None and isinstance(hf_statistics, dict) and hf_statistics.get("status") == "ok":
    hf_score = compute_hf_score(...)

# 修复后：hf_score_raw 优先，compute_hf_score 降级为备用
if hf_score is None:
    hf_raw = normalized_inputs.get("hf_score")
    if isinstance(hf_raw, (int, float)) and not isinstance(hf_raw, bool):
        hf_score = max(0.0, float(hf_raw))
        hf_status = "ok"
        hf_failure_reason = None
hf_statistics = extract_high_freq_statistics(normalized_inputs)
if hf_score is None and isinstance(hf_statistics, dict) and hf_statistics.get("status") == "ok":
    hf_score = compute_hf_score(...)
```

**输出变化：**
- `score_parts.hf_score`：从轨迹 L2 均值约 262 变为图像域 HF 相关性分数（watermarked ≫ 0，non-watermarked ≈ 0）
- `content_score`：随 HF 权重（0.3）下降至与图像域相关性尺度一致
- `compute_hf_score()` 作为备用路径保留，冻结面零改动

### P2-A：消除 schema.py 顶层 threshold_source 误导性默认填充（STATIC-6）

**文件：** `main/core/schema.py`  
**修改位置：** `threshold_source` 字段注入块（约第 1462 行）

**问题根因：**  
`if "threshold_source" not in record: record["threshold_source"] = "np_canonical"` 无条件硬填常量，而融合层实际写入 `"fallback_target_fpr_test_only"`，造成顶层字段与融合层口径倒置。

**修复内容：**

**代码变更摘要：**

```python
# 修复前
if "threshold_source" not in record:
    record["threshold_source"] = "np_canonical"

# 修复后：透传融合层实际值
if "threshold_source" not in record:
    _fusion_ts = None
    _fusion_result = record.get("fusion_result")
    if isinstance(_fusion_result, dict):
        _audit_node = _fusion_result.get("audit")
        if isinstance(_audit_node, dict):
            _fusion_ts = _audit_node.get("threshold_source")
    if not isinstance(_fusion_ts, str) or not _fusion_ts:
        _final_decision = record.get("final_decision")
        if isinstance(_final_decision, dict):
            _fusion_ts = _final_decision.get("threshold_source")
    record["threshold_source"] = _fusion_ts if isinstance(_fusion_ts, str) and _fusion_ts else "<absent>"
```

**输出变化（修复后字段映射）：**

|-----------|-----------|-----------|
| `"np_canonical"` | `"fallback_target_fpr_test_only"` | `"fallback_target_fpr_test_only"` |
| `"np_canonical"` | `"np_canonical"` | `"np_canonical"` |

**回归影响：** 已预扫描 `tests/` 中的 `threshold_source` 断言，确认现有测试均断言融合层字段而非顶层字段，CI 无失败风险。

### P2-A-fix：修复 freeze_gate.py 两处门禁与 P2-A 的口径冲突（实测触发）

**文件：** `main/policy/freeze_gate.py`  
**修改位置：** `_validate_statistical_fields()`（第 1630 行附近）和 `_check_field_override_conflict()`（第 2045 行附近）

**问题根因（实测触发）：**  
P2-A 将 `schema.py` 的顶层 `threshold_source` 注入从硬编码 `"np_canonical"` 改为透传融合层实际值。当 calibrate 前的主流程 detect 步骤设置 `allow_threshold_fallback_for_tests=true` 时，融合层写入 `"fallback_target_fpr_test_only"`，schema 如实透传此值。但 `freeze_gate.py` 有两处门禁仍以「schema 恒注入 `"np_canonical"`」为前提做强制校验，导致 detect 阶段抛出 `GateEnforcementError` 并中止。

两处门禁：
1. `_validate_statistical_fields()`：`statistical.threshold.source.np_canonical` — 要求顶层字段必须为 `"np_canonical"`
2. `_check_field_override_conflict()`：`schema.controlled_fields.override_conflict` — 检测顶层字段是否与 schema 预期注入值 `"np_canonical"` 一致

**修复内容：**  
两处门禁均增加相同的豁免判断：若 `record["fusion_result"]["audit"]["allow_threshold_fallback_for_tests"] == True`，则表明当前记录是 calibrate 前合法的中间 detect 步骤（onefile 工作流明确启用了 fallback），schema 本身合法地注入了非 `"np_canonical"` 值，将该字段的检查降级为 `WARN` 而非 `ERROR`。未授权场景（`allow_threshold_fallback_for_tests=False`）行为完全不变，仍为硬阻断。

**代码变更摘要：**

```python
# _validate_statistical_fields()：修复前
if threshold_source != "np_canonical":
    if warn_mode:
        print(...)
    else:
        raise GateEnforcementError(...)

# 修复后：allow_threshold_fallback_for_tests=True 时降级为 WARN
_fallback_authorized = record["fusion_result"]["audit"].get("allow_threshold_fallback_for_tests", False)
_effective_warn = warn_mode or _fallback_authorized
if threshold_source != "np_canonical":
    if _effective_warn:
        print(f"[FreezeGate][WARN] ...")
    else:
        raise GateEnforcementError(...)

# _check_field_override_conflict()：同样在 threshold_source 字段循环体内加 _fallback_authorized 判断
_effective_warn = warn_mode or (field_name == "threshold_source" and _fallback_authorized)
```

**根因归因：**  
P2-A 修改了「schema 永远注入 `"np_canonical"`」这一前提，但未同步审查依赖此前提的下游门禁，属于跨模块一致性遗漏。此缺口仅在真实 Colab 运行中触发，静态测试无法覆盖（测试不走 `allow_threshold_fallback_for_tests=True` 的完整 detect 流程）。

### P2-A-schema-fix：撤回 P2-A 引入的 `"<absent>"` 默认值（实测发现 calibrate/evaluate 被阻断）

**文件：** `main/core/schema.py`

**代码变更摘要：**

```python
# P2-A 引入（错误兜底值）
record["threshold_source"] = _fusion_ts if isinstance(_fusion_ts, str) and _fusion_ts else "<absent>"

# P2-A-schema-fix（当前，恢复正确兜底值）

```

**修复后完整字段映射：**

### ATT-INTEGRATE：将密码学生成溯源机制接入 onefile 工作流

**文件：** `scripts/run_onefile_workflow.py`、`configs/paper_full_cuda.yaml`  
**问题根因：**  
ATT-NEW/ATT-LF/ATT-HF/ATT-EMBED/ATT-DETECT 已实现全部 provenance 子模块和 `build_embed_attestation()` / `verify_attestation()` 独立函数，但 `run_onefile_workflow.py` 的 step loop 中无任何调用，`paper_full_cuda.yaml` 中无 attestation 配置节，导致密码学生成溯源机制无法端到端执行。

**修复内容：**

**`configs/paper_full_cuda.yaml`** 新增 `attestation` 配置节（位于 `impl` 节之后）：

```yaml
attestation:
  enabled: true
  k_master_env_var: "CEG_WM_K_MASTER"  # 主密钥环境变量名
  k_prompt_env_var: "CEG_WM_K_PROMPT"  # prompt 承诺密钥环境变量名
  k_seed_env_var: "CEG_WM_K_SEED"      # seed 承诺密钥环境变量名
  threshold: 0.65
  lf_weight: 0.5
  hf_weight: 0.3
  geo_weight: 0.2
  use_trajectory_mix: false  # 子进程模式无法传递 latent tensor
```

**`scripts/run_onefile_workflow.py`** 新增两个 helper 函数及两处 step 钩子：

- `_run_attestation_after_embed(run_root, cfg_path)`：embed 完成后从环境变量读取密钥，从 `embed_record.json` 提取 `plan_digest`，调用 `build_embed_attestation()` 构造 statement，写入 `artifacts/attestation/attestation_statement.json`。
- `_run_attestation_verification_after_detect(run_root, cfg_path)`：detect 完成后加载 statement，从 `detect_record.json` 提取 `content_evidence` 和 `geo_score`，调用 `verify_attestation()` 进行三通道融合判决，写入 `artifacts/attestation/attestation_result.json`。

两个函数均为非阻断：密钥环境变量未设置时打印警告后跳过，不影响主流程返回码。

**端到端数据流：**
```
embed 子进程 → embed_record.json(plan_digest)
   ↓ post-embed hook
build_embed_attestation(k_master, model_id, prompt, seed, plan_digest)
   → artifacts/attestation/attestation_statement.json
```

### EVALUATE-DETECT-NP：修复 evaluate 阶段 detect_np 守卫对 paper_full_cuda 的破坏性覆写
