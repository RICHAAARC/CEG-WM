## B. 修复方案（第二阶段）

### B.0 整体思路

本版修复聚焦两项可在 `main/` + `scripts/` 层落地的差距：

| 优先级 | 差距项 | 修复策略 |
|--------|--------|---------|
| P0 | detect 判决使用 test-only threshold fallback | 在 `run_detect.py` 增加 `--thresholds-path` CLI 参数；在 `run_onefile_workflow.py` 于 calibrate 后追加一次 **re-detect（只读消费 thresholds）** 步骤 |
| P1 | `score_parts.lf_status` 在 BP 不收敛时仍为 "ok" | 在 `_bind_raw_scores_to_content_payload()` 中，当 `bp_converge_status="degraded"` 时将 `score_parts["lf_status"]` 覆写为 "degraded"；不改动 UnifiedContentExtractor 决策逻辑、不改动 `low_freq_coder` 的 `trace["status"]` |

**约束边界**：  
- 不修改 `FusionDecision.decision_status` 枚举。  
- 不修改现有 embed → detect → calibrate → evaluate 主 workflow 顺序（re-detect 作为 calibrate 之后的附加步骤，不替换原 detect）。  
- 不改动冻结字段语义。  
- re-detect 写盘到独立子目录，不覆盖原 detect_record.json；evaluate 阶段选择 re-detect record 而非原始 detect record。  
- `score_parts.lf_status="degraded"` 仅影响该诊断字段，不影响 `content_evidence_payload.status`（该字段由 UnifiedContentExtractor 决定，已在其之后写盘，不会改变当前运行的判决结果）。

#### B.1.2 详细实施

**`main/cli/run_detect.py` 改动点：**

1. `main()` 中 argparse 新增：
   ```
   --thresholds-path (str, 默认 None)
   用途：阈值工件路径，若提供则在 orchestrator 调用前注入 __thresholds_artifact__
   ```
2. `run_detect()` 签名新增参数 `thresholds_path: str | None = None`。  
3. 在 cfg 加载、impl_set 构建完成后（现有 `cfg_for_preplan` 段之前），增加阈值注入块：
   - 条件：`thresholds_path` 参数为非空字符串。  
   - 操作：调用 `detect_orchestrator.load_thresholds_artifact_controlled(thresholds_path)`（该函数已存在于 `orchestrator.py:L3044`），将返回的 dict 写入 `cfg["__thresholds_artifact__"]`。  
   - 错误处理：若加载失败，调用 `set_failure_status(run_meta, RunFailureReason.CONFIG_INVALID, exc)` 并 raise，不得 silent fallback。  
   - 审计字段：向 `run_meta` 中写入 `"thresholds_path_injected": thresholds_path`（append-only）。
4. `main()` 传递 `thresholds_path=args.thresholds_path` 给 `run_detect()`。

**注意**：`load_thresholds_artifact_controlled()` 内部已执行不可变性校验与必填字段验证（`threshold_value`、`target_fpr`），无需在 CLI 层重复实现校验逻辑。

---

**`scripts/run_onefile_workflow.py` 改动点：**

> 改动只在 scripts 层，不影响 `main/` 核心判决链。

1. 在 workflow 步骤链的 calibrate 步骤成功后（`step.name == "calibrate"` 且 return_code == 0），追加 re-detect 子步骤：
   - 输出目录：`run_root / "artifacts" / "detect_np"`  
   - 命令：在 `_build_stage_command("detect", ...)` 的基础上追加 `--thresholds-path <run_root>/artifacts/thresholds/thresholds_artifact.json`，`--out` 改为 `run_root/artifacts/detect_np`。  
   - `--input` 仍传 `embed_record.json`。  
   - 失败处理：`try/except`，失败时打印 WARN 并继续，不中断主 workflow（re-detect 失败不影响原始 detect record 的审计完整性）。  
   - 成功时在 `run_meta` 中记录 `detect_np_status = "ok"` 与其路径（append-only）。

2. `_prepare_stage_cfg_path()` evaluate 段：当 `detect_np` 子目录存在且包含有效 detect_record.json 时，将 `detect_records_glob` 指向 `artifacts/detect_np/records/detect_record.json`（替代原始 detect record），以保证 evaluate 阶段消费的 detect record 携带 NP canonical 阈值。  
   - 守卫：仅当 `artifacts/detect_np/records/detect_record.json` 存在且文件大小 > 0 才替换。  
   - 失败回退：若文件不存在，保留原始 detect record glob，不中断 evaluate。

### B.3 注意事项与约束关系

#### B.3.1 冻结面约束（必须显式遵守）

1. `FusionDecision` 的三个枚举值 `decided/abstain/error` 不变。  
2. `fusion_result` 原结构保持原样，不裁剪。  
3. re-detect 写入独立子目录，不覆盖 `records/detect_record.json`。  
4. evaluate 阶段 thresholds 只读：`run_evaluate_orchestrator()` 中的 `thresholds_digest_before/after` 校验不受影响（re-detect 不修改 thresholds 工件）。  
5. `freeze_gate` 约束：re-detect 输出写入 `artifacts/detect_np/`，该路径在 `path_policy` 管辖范围内属于 artifacts 子目录，满足现有写盘路径约束。

#### B.3.2 paper_full_cuda workflow 最小输出文件集保证

| 阶段 | 必须存在的文件 | 充分条件 | 修复后保证 |
|------|-------------|---------|-----------|
| embed | `records/embed_record.json`，`artifacts/run_closure.json` | embed 无异常 | 不影响 |
| detect（原始） | `records/detect_record.json` | detect 无异常，供 calibrate GT 闭环使用 | 不影响 |

| evaluate | `records/evaluate_record.json`，`artifacts/evaluation_report.json` | evaluate 无异常（使用 detect_np 记录或降级回原始 detect 记录） | 守卫保证降级兼容 |

#### B.3.3 失败语义保持可审计

- re-detect 失败：打印 `[re-detect] WARN: NP-threshold re-detect failed: <reason>`，`run_meta` 记录 `detect_np_status="failed"`，原始 detect record 不受影响，evaluate 降级使用原始 detect record（`threshold_source` 仍为 fallback，但不 silent）。  
- `--thresholds-path` 加载失败：`set_failure_status(..., RunFailureReason.CONFIG_INVALID, ...)` 并 raise，**不允许 silent fallback**。  
- `score_parts.lf_status` 降级：仅影响诊断字段，不触发 UnifiedContentExtractor 传播链，ContentEvidence.status 不变，workflow 产物完整性不受影响。

### C.2 新风险清单

| risk_item_id | 风险触发条件 | 影响范围 | 风险等级 | 最小缓解措施 |
|--------------|------------|---------|---------|------------|
| R-01 | re-detect 写入 `artifacts/detect_np/` 而 calibrate 产出仍在 `artifacts/thresholds/`；evaluate 可能读取错误路径的 detect record | `evaluate_record.metrics` 统计口径 | **MED** | evaluate 段严格使用 `artifacts/detect_np/records/detect_record.json` 存在性守卫，回退原始但打印明确 WARN；测试中验证路径解析。 |
| R-02 | re-detect 消费 NP 阈值后 `is_watermarked` 结果与原始 detect record 不一致（两个 detect record 可能产出不同判决） | 审计一致性；calibrate 与 evaluate 消费来源不同 | **MED** | 明确记录原始 detect record 用于 calibrate（score 来源），re-detect record 用于 evaluate（NP 语义判决）；两个 record 共存不构成语义冲突，均通过 run_meta 链路追踪。不应将 re-detect record 用于 calibrate 输入。 |

## D. 本版不纳入事项（继续排除）

以下项目继续排除，不纳入 方案-2：

| 排除项 | 原因 |
|--------|------|
| 发布核心正负样本闭环（`detect_records_glob` 在 yaml 发布核心中绑定） | 属于 workflow 结构性改造，超出最小修复面 |

| `decision_status` 枚举扩展（新增 "rejected_insufficient_evidence"） | 向后兼容复杂度高，不在本版引入 |
| BP `lf_status` 全链路降级（UnifiedContentExtractor 降权处理） | 连锁语义变化复杂，需完整 NP fusion weight 回归；本版仅做顶层诊断字段降级 |
