## B. 修复方案（模板同构，已合并审查修订）

## 1. 整体构建思路

1. 先修基础链路（输入可达、模型可加载、证据透传），再修样本纪律（D1/D2/D3），最后做统计验收。
2. 严格最小侵入：不改冻结枚举语义，不改门禁主路径，不新增旁路写盘。
3. 保证 embed/detect/calibrate/evaluate 四阶段产物持续可生成；失败保留可审计失败语义。

## 2. 详细实施方案（按优先级 P0/P1/P2）

### 2.1 P0

#### P0-1（A2 根因修复：model_v2 加载）
- 文件：
  - `main/watermarking/content_chain/semantic_mask_provider.py`
  - `configs/paper_full_cuda.yaml`
- 改动：
  1. `_load_saliency_model`：当 backend=inspyrenet 时，使用“模型实例化 + `load_state_dict`”流程，禁止 state_dict 直接作为模型对象。
  2. `_probe_model_v2_availability`：从“`torch.load` 成功即可用”改为“最小结构与权重可用性校验”。

4. Evaluate：`n_pos > 0` 且 `n_neg > 0` 且可计算 `tpr_at_fpr_primary`

## C. 方案审计结论（可行性 + 新风险 + 回归测试）

### 1. 方案可行性
- 结论：PASS

### 2. 新风险清单

1. risk_item_id: R1
   - 风险触发条件：InSPyReNet 架构实例化与权重 key 不匹配
   - 影响范围：embed records 字段、统计口径
   - 风险等级：HIGH
   - 最小缓解措施：`_load_saliency_model` 增加 strict key 校验与失败原因落盘；禁 silent fallback

2. risk_item_id: R2
   - 风险触发条件：evaluate 未绑定 dual-branch，回退 clone
   - 影响范围：GT 语义、`n_pos/n_neg` 有效性
   - 风险等级：HIGH
   - 最小缓解措施：`_prepare_stage_cfg_path` 对 evaluate 启用 dual-branch

3. risk_item_id: R3
   - 风险触发条件：在 dual-branch 不可用时错误开启 synthetic_closure 过滤
   - 影响范围：calibrate 样本量、workflow 连通性
   - 风险等级：MED
   - 最小缓解措施：`exclude_synthetic_negative_closure_marker` 默认 `false`，仅在 dual-branch 成功后显式开启

4. risk_item_id: R4
   - 风险触发条件：新增字段破坏 append-only 约束
   - 影响范围：records 兼容性、审计回放
   - 风险等级：LOW
   - 最小缓解措施：仅新增字段，不替换旧字段；新增兼容性测试

### 3. 审查合并后的执行判定

1. `FixPlanVerdict`: ACCEPT（按修订版）
2. `RiskVerdict`: NO_NEW_HIGH_RISK（无新增高风险；存在残余运行风险）
3. `WorkflowVerdict`: SAFE_OUTPUTS（在修订版约束下）

### 4. 执行约束（强制）

1. 执行顺序必须为：P0-1 → P0-2 → P0-3 → P0-4 → P1-x → P2。
2. `exclude_synthetic_negative_closure_marker` 本批不写入默认 yaml。
3. GAP-SIDECAR 作为并行项处理：
   - 方案：更新测试断言为与当前配置一致；更新测试期望为 True
4. 任一阶段失败时必须输出失败语义，不允许静默覆盖为 `ok`。
