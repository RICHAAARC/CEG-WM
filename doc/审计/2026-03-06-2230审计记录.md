## 一、明确排除项（修复方案第六节）

### 1.1 【HIGH-CRITICAL】detect NP 语义失效

**审计报告原结论：**  
`detect_record.fusion_result.audit.threshold_source = "fallback_target_fpr_test_only"`，`used_threshold_value = 0.01`（等于 `target_fpr`，与 NP 校准阈值 78.545 相差约 5 个数量级）；`is_watermarked=true` 的判决依据为 `content_score(78.545) > 0.01`，与 NP 语义无关。

**排除理由（修复方案原文）：**  
当前 workflow 顺序决定 detect 阶段在 calibrate 之前运行；若要彻底消除 detect fallback，需要单独设计 detect-after-calibrate 路径或调整主 workflow 顺序，属于 workflow 结构性改造，超出本版修复面。

**代码根因（未变）：**
- `main/watermarking/detect/orchestrator.py:583-589`：仅当 `input_record` 已携带 `thresholds_artifact` 时才注入阈值工件
- `main/watermarking/fusion/decision.py:78-95`：`__thresholds_artifact__` 缺失时走 `allow_threshold_fallback_for_tests` 分支，以 `target_fpr=0.01` 作为阈值

**目标状态：** `threshold_source = "np_canonical"`，`used_threshold_value ≈ 78.545`

**建议后续方案：** 在 `main/cli/run_detect.py` 增加显式阈值工件输入参数（`--thresholds-path`）；或在 onefile workflow 中重排 calibrate → detect 顺序（至少对 re-detect 阶段）；论文可信模式下禁止 fallback 写入正式 record。

---

### 1.2 【HIGH-CRITICAL】发布核心无正负样本闭环

**审计报告原结论：**  
`paper_full_cuda.yaml` 中 `calibration.detect_records_glob`、`evaluate.detect_records_glob`、`evaluate.thresholds_path` 均为 null；`calibrate/evaluate` 所需正负样本对由 `scripts/run_onefile_workflow.py:_prepare_detect_records_with_minimal_ground_truth()` 构造，不在 `configs/ + main/` 发布核心内。

**排除理由：** 未纳入本版修复面。

**目标状态：** `configs/ + main/` 发布核心本身能够定义正负样本来源路径，不依赖辅助脚本构造。

**建议后续方案：** 在 `main/` 内提供正式的 ground-truth bundle 生成入口；或在 `paper_full_cuda.yaml` 中补齐 `detect_records_glob` 与 `thresholds_path` 的正式绑定。

---

### 1.3 【HIGH】FPR 超标 3 倍，NP 统计功效不足

**审计报告原结论：**  
`calibration_samples = 30`（取最大值作阈值），评估集 `fpr_empirical = 0.0333`（1/30），超过 `target_fpr = 0.01` 约 3 倍；Wilson 95% CI 为 [0.001, 0.168]，区间极宽。

**排除理由（修复方案原文）：**  
`target_fpr=0.01` 下获得受控经验 FPR 需负样本 n ≥ 100，属于实验规模与运行成本问题，不作为代码层修复项。

**目标状态：** `fpr_empirical ≤ 0.01`（需 ≥ 100 个不同 prompt 的非水印 SD3 推理图像）

**建议后续方案：** 使用 `prompts/prompts_100.txt` 或更多 prompt 生成负样本集（约需 GPU 上 3-5 小时）。

---

### 1.4 【MEDIUM】`decision_status` 枚举与论文目标不对齐

**审计报告原结论：**  
`main/watermarking/fusion/interfaces.py:L46`：枚举值为 `Literal["decided", "abstain", "error"]`；论文目标语义 `"rejected_insufficient_evidence"` 不存在，当前由 `"abstain"` 承载。

**排除理由（修复方案原文）：**  
不扩展 `FusionDecision.decision_status` 枚举，不新增 `"rejected_insufficient_evidence"`。

**目标状态：** 如需对齐论文语义，应将 `"error"` 细化为 `{"error", "rejected_insufficient_evidence"}`，并保证向后兼容。

## 二、未纳入项（修复方案未覆盖）

### 2.1 【MEDIUM】detect 侧子空间一致性走代理路径，非主路径

**审计报告原结论：**  
`subspace_evidence_level = "surrogate"`，`subspace_primary_path = false`，`subspace_consistency_status = "absent"`；detect orchestrator 传入 `final_latents`（最终 latent），而主路径需要逐步 JVP（每 timestep 的 Jacobian），当前数据流中无逐步 JVP，主路径无法被激活。

**代码根因（未变）：**  
`main/watermarking/content_chain/subspace_planner.py`：evidence_level 判定依赖逐步 JVP 输入，当前 detect 侧未传入。

**目标状态：**  
`subspace_evidence_level = "primary"`，`subspace_primary_path = true`

**建议后续方案：** 在 `infer_runtime.py` 增加逐步 latent JVP 捕获选项（`capture_jvp_trajectory: true`），将逐步 Jacobian 传入 subspace_planner；注意主路径激活后 `subspace_consistency_status` 可能变为 `"inconsistent"`，需 SD3 真实运行验证（见需 SD3 验证项 SD-3）。

## 三、部分修复项（诊断已改善，根本原因未消除）

### 3.1 【HIGH】BP 解码不收敛，`lf_status` 仍为 `"ok"`

**本版已修复部分：**  
`main/watermarking/content_chain/low_freq_coder.py`：`detect_score()` 新增 `bp_converge_status`（`"ok"/"degraded"`），写在 `lf_trace_digest` 之后；`orchestrator.py` 增加透传，使 `content_evidence_payload.score_parts.lf_detect_trace.bp_converge_status` 落盘。

**残余问题：**  
`lf_status` 在 `bp_converged=False` 时仍为 `"ok"`（有意保留）；直接将 `lf_status` 标为 `"degraded"` 会触发 `ContentDetector` 对非 ok LF 状态的传播链，连锁语义变化超出本版修复面。

**当前产物状态：**  
`bp_converged = false`，`lf_status = "ok"`，`bp_converge_status = "degraded"`（新字段）；`lf_score = 0.46875`（接近随机基准 0.5）。

**目标状态：**  
`bp_converged=false` 时 `lf_status = "degraded"`，`ContentDetector` 对 degraded 状态有明确的融合权重降级处理，不完全阻断判决。

**建议后续方案：** 在 `ContentDetector._extract_channel()` 中对 `bp_converge_status = "degraded"` 做显式降级映射；同步修改 `_build_lf_only_*` 与 fusion weight 逻辑；需完整回归测试。
