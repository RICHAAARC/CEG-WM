# 第二阶段：修复方案

# 2026-03-02-0830 修复方案：核心链路打通与统计闭环

> **状态**：Draft
> **基线**：2026-03-02-0830审计报告
> **目标**：解决 Embed/Detect 核心链路断裂问题，确保 Colab 真实运行下的有效证据产出，并修复统计污染问题。

## 1. 整体构建思路

本次修复秉持“产物不可丢、门禁不可绕、错误不可吞”的红线标准。对于 `run_onefile_workflow.py` 不再用生硬阻止写盘的方式对抗环境缺失，而采用“记录环境失败但不干涉架构语义状态、防伪样本被严格清洗、最后对评价结果强终止（Fail-Fast）”的策略进行对冲：
1. **基础基建修复**（P0-2, P2-1）：修复 `paper_full_cuda.yaml` 配置缺口；打破 `_build_content_inputs_for_detect` 粗暴丢失 context 证据与 digest 强映射的问题，保障状态穿透不折断。
2. **校验端纪律加固**（P1-2, P1-3）：通过配置过滤并启动 sidecar 真实打分逻辑，杜绝了 `-1.0` 带来的无效校准假阳过关。
3. **闭环门禁严查**（FAIL-1, FAIL-5）：利用在 Evaluator 输出后的强制退化门禁异常拦截与 CLI 执行初期的主白名单断言拦截。

## 2. 详细实施方案

### 2.1 【P0-2/P0-3 级修复】跨环境路径与证据强制透传构建
* **目标**：不因路径跨系统失效引发截断，保证 `expected_plan_digest` 无论在什么情况下都有效送达 Detector 中，恢复 `absent` 到真正的由于缺乏模型本身的 `failed`。
* **文件**：`main/watermarking/detect/orchestrator.py`
* **操作点**：修复 `_build_content_inputs_for_detect`：
  即便当 `_resolve_detect_image_path(cfg, input_record)` 返回 `None`，依然必须获取 `content_evidence = input_record.get("content_evidence") or {}`，并在返回字典中构造退化路径兜底字符串（如原字段 `watermarked_path`），强制让 `expected_plan_digest` 有效透传递给 Detector。

### 2.2 【P1-2/P1-3 级修复】切断退化假样本引发的 FPR 指标粉饰
* **目标**：确保在 Sidecar 受限下的假阴阳样本在正式校准口被明确踢出，不再用于污染 `evaluation` 的门禁计算。
* **文件**：`configs/paper_full_cuda.yaml` 
* **操作点**：设置 `calibration.exclude_formal_sidecar_disabled_marker: true`，开启既有过滤开关，拒绝在 `run_onefile_workflow.py` 制造的前 -1.0 系列假样本进入校准集分布库，逼迫阈值退化拦截被激活。

### 2.3 【P2-1 级修复】重启 Sidecar 等价基线比较能力
* **目标**：令真实 workflow 在完整评估面与消融中保持有效区分。
* **文件**：`configs/paper_full_cuda.yaml`
* **操作点**：将 `detect_runtime.image_domain_sidecar_enabled` 改为 `true`。

### 2.4 【FAIL-1 级修复】显式捕捉零阳样本污染并打碎通过假象
* **目标**：防止 evaluate 在无有效分数下以 “None” 糊弄了事；在保留可审计产物的前提下，强制下游按失败处理。
* **文件**：`main/cli/run_evaluate.py`（落盘前标记 + 落盘后阻断）
* **操作点**：在 `run_evaluate()` 中，先检查 `metrics.n_pos` / `metrics.n_neg`。若任一为 0，则在待写入 record 中追加顶层字段 `evaluate_status: "degenerate"`（append-only）；随后执行 `records_io.write_json()` 正常落盘保留现场；落盘后立即抛出 `ValueError("Degenerate evaluation: n_pos or n_neg must be positive")` 中断流程。并在下游（`scripts/run_freeze_signoff.py`）把 `evaluate_status == "degenerate"` 识别为硬失败（blocking reason）。

### 2.5 【FAIL-5 级修复】执行级全局白名单运行时强隔离
* **目标**：阻止所有绕过配置文件强塞外层实现类的攻击。
* **文件**：`main/registries/runtime_resolver.py` + `main/cli/run_embed.py`、`main/cli/run_detect.py`、`main/cli/run_calibrate.py`、`main/cli/run_evaluate.py`
* **操作点**：
  1. 在 `build_runtime_impl_set_from_cfg` 增加可选参数 `whitelist`（留有 `None` 兼容位仅为了向后应对历史纯测试用例）。
  2. 当 `whitelist` 非空时，在 `build_impl_set` 之前调用 `assert_impl_allowed(whitelist, identity.as_dict())`，保证不在 whitelist 的 impl_id 无法进入实例化与执行阶段。
  3. **强烈约束**：在所有实际的主干 CLI 服务中（`run_embed`，`run_detect` 等），**必须全量强制载入并传入此 `whitelist`，严禁主干线路上行走到 default None 路径上**，避免校验逃逸。

### 2.6 修改文件清单与兼容策略（本轮）
* **文件清单与函数级改动**：
  1. `main/cli/run_evaluate.py`：`run_evaluate` 在退化分支写入 `evaluate_status="degenerate"`，并在落盘后抛错阻断。
  2. `main/registries/runtime_resolver.py`：`build_runtime_impl_set_from_cfg` 增加可选 `whitelist` 入参并在实例化前执行 `assert_impl_allowed`。
  3. `main/cli/run_embed.py` / `main/cli/run_detect.py` / `main/cli/run_calibrate.py` / `main/cli/run_evaluate.py`：调用 `build_runtime_impl_set_from_cfg` 时强制读取并硬塞入实际 whitelist。
  4. `scripts/run_freeze_signoff.py`：新增对 `evaluate_status=="degenerate"` 的阻断判定，确保下游强制识别失败。
  4. `configs/paper_full_cuda.yaml`：设置 `calibration.exclude_formal_sidecar_disabled_marker: true`；`detect_runtime.image_domain_sidecar_enabled: true`。
* **新增字段与兼容策略**：
  - 新增字段仅限 `evaluate_record` 顶层 `evaluate_status`，值域固定 `"degenerate"`（仅退化分支写入，append-only，不改既有字段含义）。
  - 向后兼容策略：未识别该字段的旧读取逻辑仍可读取原有 `metrics`；新门禁逻辑在 signoff 层将其强制判定为失败。
  - 不改变 `status/decision_status` 枚举语义，不改变 `cfg_digest/plan_digest/thresholds_digest/threshold_metadata_digest` 的输入域与序列化口径。

## 3. 注意事项与约束关系

1. **冻结面与门禁面约束**：决不修改枚举（包括 `error`、`abstain`、`failed` 及所有 `_status` ）。仅依靠在配置侧开通过滤和在最终 `Evaluate` 追加抛错实现安全化，未动摇任何状态协议映射与评价接口约定。
2. **输出产物一致性约束**：在遇到严重 P0（模型失踪）或 P1（无样本可配）时，系统至少保留 `run_closure` 与上游阶段 records；evaluate 退化场景允许写出 `evaluate_record.json`，但必须带 `evaluate_status="degenerate"` 且下游强制判失败。
3. **退化容错处理原则**：评价阶段如果抛错中断，是在“成功序列化并闭环后”，以此满足“不可由于中止而完全销毁产物”的承诺要求。

## 4. 执行计划

1. **第一天基座修复**：更正并提交 `paper_full_cuda.yaml` 开关修正与 Sidecar 开放配置（解决 P1-3/P2-1）。
2. **第一天骨架修复**：改写并完成 `orchestrator.py` 在探测提取链上对于 Evidence 无条件安全透穿（解决 P0-2/3、P1-1）。
3. **第二天门禁加锁**：对 `run_evaluate` 添加 0 Pos/0 Neg 的“落盘标记 + 抛错阻断”防线，并在 signoff 增加 `evaluate_status` 失败识别；在 `runtime_resolver` 注入统一 whitelist 运行期门禁并由 CLI 强制传参启用。
4. **验证与回归**：启动所有 CI 级和 Workflow 验证，检查日志断链与日志完整性。

## 5. 验收标准

1. **跨链路基准（P0-2/3）**：当输入文件缺失时，不再拦截为短路空，`content_failure_reason` 为真实原因，并且不会因为 `detector_no_plan_expected` 盖住下层原因。
2. **校准指标正常化（P1-1/2）**：`threshold_value` 产出不得为退化占位（如 `-1.0`）；退化伪正样本不会并入 `fpr_empirical`。
3. **真实打靶拒绝（FAIL-1）**：若在测试全过程缺失环境（无法出正分），Evaluate 必须写入 `evaluate_status="degenerate"` 后再抛出异常；且 signoff 必须把该字段识别为硬失败，`n_pos=0` 或 `n_neg=0` 不得通过。
4. **白名单防御（FAIL-5）**：任意强制将 CLI 的 `cfg.impl.subspace_planner_id` 指向测试 ID 并尝试启动时，CLI 顶层必须因为强制附入的 `whitelist` 断言立即爆出拦截异常并阻断。
