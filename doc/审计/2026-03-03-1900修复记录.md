## 17. signoff 仍 BLOCK_FREEZE：audit_attack_protocol_report_coverage 修复

### 17.1 问题来源

第 16 节修复生效后（commit `589cdcc1`），`signoff_report.json` 已正确生成，`failing_stage` 进一步收窄至 `signoff`，唯一阻断原因为：

```text

  root_cause: required_artifact_missing_under_bound_run_root
  status: "evaluation_report.json not found under bound run_root"
```

`checked_paths` 显示审计已正确检查了 `artifacts/evaluation_report.json`，文件**实际存在**，但 `reported_conditions=[]`，触发了"文件不存在"的 BLOCK FAIL 路径。

### 17.2 根本原因分析

**双重漏洞链**（均在 `scripts/audits/audit_attack_protocol_report_coverage.py`）：

**漏洞 A — `extract_reported_conditions` 中的 `status` 字段检查错误**：

```python
# 旧代码（错误）：要求 status in {"ok","success","completed"}
status = item.get("status")
if status not in {"ok", "success", "completed"}:
    continue  # status=None（缺失字段）也被跳过 → 所有条目被过滤
```

实际 `metrics_by_attack_condition` 条目无 `status` 字段（只有 `fp`、`fpr_empirical`、`n_total`、`group_key` 等字段）。`item.get("status") = None`，`None not in {"ok",...}` = True，**所有条目被跳过**，导致 `reported_conditions=[]`。

**漏洞 B — 空 `reported_conditions` 时的错误路径（当 `run_root` 绑定时）**：

```python
# 旧代码：reported_conditions=[] 一律返回 BLOCK FAIL "not found"
if len(reported_conditions) == 0 and isinstance(run_root, Path):
    return FAIL("evaluation_report.json not found under bound run_root")
```

即使文件存在且成功读取，若 `reported_conditions=[]`（无论原因），都以"文件不存在"为由返回 BLOCK FAIL。未区分：
- 情形 A：文件确实不存在（正确 BLOCK FAIL）
- 情形 B：文件存在但仅含 sentinel 条件（smoke test 场景），应返回 N.A./NON_BLOCK

**漏洞 A 触发后的额外过滤**：即使修复漏洞 A，`group_key="unknown_attack::unknown_params"` 是 `_UNKNOWN_CONDITION_SENTINELS`，在 `collect_reported_conditions_candidates` 中被 `is_unknown_condition_sentinel()` 过滤 → `reported_conditions` 仍为 `[]`。

**结论**：smoke test 评测报告只含 sentinel 条件，属于 N.A. 场景，不应 BLOCK 冻结。

### 17.3 修复内容

**已修改文件：`scripts/audits/audit_attack_protocol_report_coverage.py`**

if not isinstance(reported_conditions, list) or len(reported_conditions) == 0:

## 22. repro pipeline detect 融合崩溃修复（commit `8408038`）

### 22.1 根因分析

`audit_protocol_compare_outputs_schema` BLOCK 的根因链（commit `122ca5e` GPU 运行）：

```
run_freeze_signoff.py
  → run_multi_protocol_evaluation.py --mode repro
    → run_repro_pipeline.py
      → run_detect.py (repro stage, detect.content.enabled=True)
        → resolve_content_override_from_input_record(embed_record)
          → 返回 content_result = {status:"ok", score:None}
          → _adapt_content_evidence_for_fusion → content_score=None
          → make_decision: content_score must be number → ValueError
          → sys.exit(1)
      → RuntimeError: repro pipeline stage failed (exit code 1)
  → compare_summary.status = "fail"
→ audit BLOCK
```

根本原因：`resolve_content_override_from_input_record` 对 `content_evidence_payload`、`content_result`、`content_evidence` 三个键按顺序尝试，但仅对 `content_evidence` 做了 score 有效性校验（embed 侧常为 None）。`content_result` 来自 repro embed 的 embed 模式提取：`status=ok` 但 `score=None`（embed 模式不执行 content 检测），该结果被无过滤传入 detect 融合层，触发 `content_score must be number` ValueError。

目标：在第 27 节修复 seed=null 后，GPU 跑确认 `injection_mode=subspace_projection`，但 `detect_runtime_mode` 仍为 `fallback_identity_v0`，根因为 detect 侧 LF 分数提取维度不匹配。

### 28.2 根因分析

`extract_lf_score_from_detect_latents` 中：
- `projection_matrix` 形状为 `(128, 8)`（feature_dim=128，watermark_bits=8）
- detect 侧 latent 展平后形状为 `(65536,)` = 256×256
- 直接执行矩阵乘法 `latents_flat @ proj_matrix` 时维度不匹配（`65536 != 128`），抛异常
- catch except → 返回 `(None, "failed")` → `detect_runtime_mode=fallback_identity_v0`

lf_basis payload 中无降维参数，detect 侧无法复现 embed 侧降维 → GAP-C1 根因二。

## 30. SD-RUNTIME-3 几何链路 cross-similarity 修复（2026-03-06）

目标：修复 `geo_available_rate=0.0` 的根本原因——`_compute_sync_quality` 使用单侧 latent 统计，无法区分水印/非水印图像，`uncertainty=0.686 > 0.5` 导致 sync 永远 mismatch。

### 30.1 根因分析

`_compute_sync_quality` 原逻辑：
- 计算 detect 侧 latent 的内部统计（contrast_ratio、spectral_peak_ratio、peak_sharpness）
- 仅对单侧绝对值评估，无 embed 侧参照
- 水印图像与非水印图像的内部统计无显著差异 → `quality_score` 偏低 → `uncertainty > 0.5` → `sync_status=mismatch`

### 30.2 修复策略

将 embed 侧 latent 统计通过 embed_record → run_detect.py → cfg → runtime_inputs 的链路传递给 sync 模块，改用 embed vs detect cross-similarity 替代单侧统计。
