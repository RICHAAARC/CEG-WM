## FIX-P0-1：替换 `np.corrcoef` 为安全实现，消除几何同步链进程崩溃

**问题**：`_build_attention_maps_from_latents()` 在 Windows/NumPy 组合下调用 `np.corrcoef(channels)` 触发 Fatal Python error: Aborted（进程级，无法被 try/except 捕获）。触发路径：`sync_primary_anchor_secondary=true` → `_precompute_relation_digest_for_sync()` → 此函数。

**文件**：`main/watermarking/detect/orchestrator.py`

**步骤一**：在该函数所在同一模块中新增辅助函数（插入位置：`_build_attention_maps_from_latents` 定义之前，即 ~line 3560 附近）：

```python
def _safe_corrcoef(channels: np.ndarray) -> np.ndarray:
    """
    功能：用纯 NumPy 基础运算替代 np.corrcoef，避免 Windows/BLAS 进程崩溃。

    Compute a correlation-coefficient-like matrix without calling np.corrcoef,
    which triggers Fatal Python error: Aborted on Windows with certain NumPy builds.

    Args:
        channels: 2-D float array of shape [C, N_tokens].

    Returns:
        Correlation matrix of shape [C, C], dtype float64.
    """
    mat = channels.astype(np.float64)
    mat = mat - mat.mean(axis=1, keepdims=True)
    norms = np.sqrt((mat * mat).sum(axis=1, keepdims=True))
    norms = np.where(norms < 1e-10, 1.0, norms)
    mat = mat / norms
    return mat @ mat.T
```

**步骤二**：替换崩溃调用（`orchestrator.py:3597`）：

```python
# 替换前（崩溃）：
correlation = np.asarray(np.corrcoef(channels), dtype=np.float64)

# 替换后（安全）：
correlation = np.asarray(_safe_corrcoef(channels), dtype=np.float64)
```

**验证**：`pytest tests/test_anchor_gate_by_sync_status.py -v` 不再报 Aborted，全部用例通过。

**约束说明**：`_safe_corrcoef` 的数值输出与 `np.corrcoef` 等价（逐行中心化 + L2 归一化后内积），不改变下游 `relation_digest` 语义，不触碰任何冻结合约字段。

## FIX-P0-2：确保 DCT 块切片连续性，消除嵌入阶段进程崩溃

**问题**：`encode_low_freq_dct()` 内循环从 HWC 三维数组中按最后一维（channel）切片得到非连续内存视图，传入 `_dct2()` 做矩阵乘法时在 Windows BLAS 层 Abort。

**文件**：`main/watermarking/content_chain/low_freq_coder.py`，约第 964 行（函数 `encode_low_freq_dct` 内层 channel 循环中）

```python
# 替换前（崩溃）：
            block = work[r0:r0 + block_size, c0:c0 + block_size, channel_idx]
            coeff = _dct2(block, dct_matrix)

# 替换后（安全，确保连续内存）：
            block = np.ascontiguousarray(
                work[r0:r0 + block_size, c0:c0 + block_size, channel_idx]
            )
            coeff = _dct2(block, dct_matrix)
```

**验证**：`pytest tests/test_c3_real_embedding_pipeline.py -v` 不再 Aborted。

**约束说明**：`np.ascontiguousarray` 只保证内存连续性，不改变数值，不影响 DCT 结果，不引入新路径。

## FIX-P1-2：阻止 workflow 将 `status="failed"` 的证据改写为 `"ok"`

**问题**：`run_onefile_workflow.py:782-784` 在找到任意可恢复的数值型分数后，无条件把 `content_payload["status"]` 改写为 `"ok"` 并清空 `content_failure_reason`，即使原始 status 为 `"failed"` 也会被覆盖。这违背了"可拒绝证据链"的论文语义。

**文件**：`scripts/run_onefile_workflow.py`，第 782 行

```python
# 替换前（无条件覆盖 status，且仍写出 artifact）：
    if isinstance(recovered_score, float):
        content_payload["score"] = recovered_score
        content_payload["status"] = "ok"
        content_payload["content_failure_reason"] = None
        if profile == PROFILE_PAPER_FULL_CUDA:

# 替换后（failed 整体排出恢复路径，落入已有硬阻断分支）：
    if isinstance(recovered_score, float) and status_value != "failed":
        content_payload["score"] = recovered_score
        content_payload["status"] = "ok"
        content_payload["content_failure_reason"] = None
        if profile == PROFILE_PAPER_FULL_CUDA:
```

**效果**：
- `status="absent"` 且存在残余分数 → 升级为 `"ok"`（原有路径保留）
- `status="failed"` 且存在残余分数 → 整体跳过恢复路径，fall through 到已有的硬阻断分支（paper 模式 `raise ValueError`，非 paper 模式返回 None/原路径）；不写 `detect_record_for_scoring.json`，不污染 NP 校准输入分布

**约束说明**：
- `status_value` 变量已在第 736 行初始化（`status_value = content_payload.get("status")`），无需新增变量
- 改动只在 `if` 条件上追加 `and status_value != "failed"`，单行修改，不引入新分支
- 下方 `if profile == PROFILE_PAPER_FULL_CUDA: raise ValueError(...)` 的硬阻断分支已存在于原代码，此修复使 `failed` 证据自然落入该分支，不新增任何旁路

## NOTE-P2-2：`allow_failed_semantics_collection=true` 门禁放松——设计审查项

**现象**：`configs/paper_full_cuda.yaml:108` 中 `allow_failed_semantics_collection: true` 允许 detect 阶段在 `valid_content_score_samples=0` 时仍输出 `gate_relaxed=True`，使 calibrate/evaluate 继续执行。

**是否应修改**：取决于研究意图——

- **若消融变体（如 `enable_content=false`）的失败是"预期失败"**：保持 `true` 合理，但应在 experiment_matrix 汇总层为此类 variant 明确标注 `gate_was_relaxed=true`，使论文读者可区分"正常执行"与"容错执行"。
- **若所有 variant 均应有效 content score，失败是实现问题**：改为 `false`，让 pipeline 在无有效样本时直接 fail-fast（不产出伪闭环工件）。

**建议**：在修复 FIX-P0-1 / FIX-P0-2 / FIX-P1-1 之后重新评估——若三个 P0/P1 修复后实验可正常产出有效分数，则考虑将 `allow_failed_semantics_collection` 改回 `false` 以收紧门禁语义。**此项当前不强制修改**。

## 修复实施顺序

> ⚠️ **关键部署约束**：FIX-P1-2 与 FIX-P1-1 必须**同步部署**，不得单独先应用 FIX-P1-2。
>
> 原因：现有 Colab run 中 `content_evidence_payload.status = "failed"`（`mask_extraction_no_input`）。FIX-P1-2 应用后，`status="failed"` 的证据整体跳过恢复路径，在 paper 模式下直接 `raise ValueError`。若 FIX-P1-1（提供有效权重路径）未就绪，Colab 每次运行都将在 scoring 步骤硬失败——这是正确的语义保障，但需确认权重可用后再部署。

```
1. FIX-P0-2（low_freq_coder.py）     → 先让 embed 阶段能跑完（1850 前已存在的 bug）
2. FIX-P0-1（orchestrator.py）       → 再让 detect 阶段不崩溃（1850 P0-5 引入的 bug）
3. FIX-P2-1（测试 fixture）           → 恢复审计测试合约（1850 P0-3 引入的遗漏）
4. FIX-P1-1 + FIX-P1-2 同步部署     → 补权重路径 + 修复语义改写旁路（必须同步）
   确认 Colab 权重下载 URL 后一次性部署，禁止单独部署其中任何一项
5. NOTE-P2-2（设计审查）              → P0/P1 全部修复后再决策
```

**关于 Colab 完整 workflow 可运行性**：
- 修复 FIX-P0-1 + FIX-P0-2 后，embed 和 detect 阶段不再崩溃；1850 P0-5 引入的 `_precompute_relation_digest_for_sync()` 可正常运行，`relation_digest` 可提供给 sync，原有的几何 `mismatch` 问题将得到改善
- 修复 FIX-P1-1 + FIX-P1-2 后，paper 模式将产出真实有效的内容证据记录，而非伪造的 `status="ok"` 工件；若权重路径有效，content chain 将首次真正闭环
- 现有 Colab 产物（`detect_record_for_scoring.json` 中 `status="ok"`）是旧的"伪"成功证据，修复后将不再产出此类工件
