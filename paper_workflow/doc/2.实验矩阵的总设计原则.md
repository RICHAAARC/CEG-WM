下面继续给出一版可直接冻结的 **clean／attack、event-level、system-level 实验矩阵定义**。该版本的目标不是罗列所有可跑实验，而是建立一套**严格服务于论文主结论**的实验组织框架，使后续每一轮 run 都能明确回答某个 Research Question，而不是形成与论文主线脱节的结果堆积。

## 一、实验矩阵的总设计原则

在当前方法叙事已经固定为

`content_chain → event_attestation → system_final`

且 `payload` 与 `geometry` 已经降级为辅助模块之后，实验矩阵也必须同步收口为四层：

1. **clean 基线层**：验证内容链是否构成 primary evidence；
2. **attack 鲁棒性层**：验证主链在攻击条件下的退化规律与稳定性边界；
3. **event-level 结构层**：验证局部观测如何被组织为事件级证明；
4. **system-level 判定层**：验证多事件聚合后的最终系统结论是否稳定、可解释、可发布。

因此，后续实验不应再按“模块逐个跑”来组织，而应按“论文问题逐层回答”来组织。换言之，实验矩阵的第一原则是：

> **每一种实验设置，都必须能够明确映射到某个主张、某张主表或某个主图。**

---

## 二、实验矩阵的样本层级定义

为了避免 clean、attack、event、system 四层之间的统计口径混淆，建议先固定样本层级。

### （一）图像级样本（image-level sample）

这是最底层观测单元。
每个图像级样本可以是：

1. clean 观测；
2. attacked 观测。

图像级层面主要用于：

1. 内容链分数；
2. `payload` probe 字段；
3. 原子级检测结果。

该层不直接承担最终论文结论，只为 event-level 提供基础观测。

### （二）事件级样本（event-level sample）

这是本文最关键的中间层。
一个 event 应被固定理解为：

> 围绕同一验证对象所组织的一组局部观测及其结构化证明结果。

也就是说，event 不是单张图本身，而是一个**由若干图像级观测组成的局部证据单元**。其输出包括：

1. event attestation status；
2. event-level margin／confidence；
3. event 是否进入 boundary 区；
4. geometry optional claim 是否 eligible；
5. `payload` probe 是否提供附加一致性信号。

### （三）系统级样本（system-level sample）

这是最终系统判定单元。
一个 system-level 实例由多个 events 聚合形成，用于输出：

1. `system_final` 判定；
2. release／signoff 状态；
3. blocking reasons；
4. 辅助模块是否参与补充说明。

system-level 不是单纯的统计平均层，而是本文最终研究对象所在层。

---

## 三、实验矩阵与核心 Research Questions 的映射

实验矩阵应严格围绕之前冻结的四个核心问题展开。

### （一）RQ1：内容证据链是否是系统的主导判别来源

对应实验层：

1. clean 基线层；
2. attack 鲁棒性层中的主链退化分析。

### （二）RQ2：事件级证明层是否将局部观测组织为更稳定的中间结论

对应实验层：

1. event-level 结构层；
2. clean／attack 条件下的 event 稳定性对比。

### （三）RQ3：system-level final decision 是否更稳定、更接近真实部署目标

对应实验层：

1. system-level 判定层；
2. 多事件聚合收益分析。

### （四）RQ4：`payload` 与 geometry 是否提供补充性解释价值

对应实验层：

1. clean／attack 下的 `payload` probe 统计；
2. boundary event 子集上的 geometry optional claim 分析。

需要强调的是，**RQ1～RQ3 是主文主轴，RQ4 是主文次级问题或附录增强问题**。因此，实验资源分配也应体现这一主次关系。

## 四、clean 基线层实验矩阵

clean 层的作用不是“证明系统无攻击时表现很好”这么简单，而是为全文建立三个基础事实：

1. 内容链确实是 primary evidence；
2. event-level 组织不是多余结构；
3. system-level 判定在无攻击条件下具有稳定基线。

### （一）clean 层的核心实验目标

clean 层必须回答以下问题：

1. 正负样本在内容链分数空间中是否可分；
2. event-level 组织后是否比单图观测更稳定；
3. system-level 最终结论是否在 clean 条件下稳定成立；
4. `payload` 是否至少作为弱 probe 提供一致性补充。

### （二）clean 层的最小实验维度

clean 层建议固定为以下三个轴。

1. **label 轴**
   包含 positive 与 negative 两类。

2. **content difficulty 轴**
   建议按照内容链天然难度或语义复杂度进行分层，例如：
   低难度、中难度、高难度。
   该轴的目的不是做额外主张，而是避免 clean 结果被“容易样本占优”掩盖。

3. **replicate 轴**
   每类设置下保留多个重复观测，用于支撑 event-level 稳定性分析，而不是只看单次结果。

### （三）clean 层的主指标

clean 层的主指标应分三组。

1. **image-level 指标**
   内容链主分数分布；
   ROC／AUC；
   TPR@low FPR；
   clean 条件下的 margin 分布。

2. **event-level 指标**
   event attestation pass rate；
   event-level margin 分布；
   event 内部观测一致性；
   event-level 误判率。

3. **system-level 指标**
   final decision accuracy；
   false allow／false reject；
   blocking rate；
   signoff consistency。

### （四）clean 层直接服务的论文图表

clean 层至少应直接产出以下主图主表。

1. **主图 1**：positive／negative 在内容链分数空间中的 clean 分布图；
2. **主表 1**：clean 条件下的 image-level、event-level、system-level 核心指标表；
3. **主图 2**：从 image-level 到 event-level 再到 system-level 的稳定性提升示意图。

这三者共同服务于 RQ1 与 RQ2 的前半部分。

## 五、attack 鲁棒性层实验矩阵

attack 层不是简单地把所有攻击都跑一遍，而是要围绕论文主结论组织成**两类攻击集合**：

1. **主链鲁棒性攻击集合**；
2. **geometry rescue 候选攻击集合**。

这一区分非常关键。若不分开，往往会出现“攻击很多，但没有一个真正服务 geometry rescue”的问题。

### （一）主链鲁棒性攻击集合

这一集合的目标是回答：

1. 内容链在不同攻击家族下如何退化；
2. event-level 与 system-level 是否比单点观测更稳健；
3. 哪些攻击主要损伤 primary evidence。

这一集合建议覆盖四类 attack family。

1. **压缩与量化类**
   如 JPEG 质量退化、重编码等。
   该类用于验证内容链对常见失真扰动的耐受性。

2. **滤波与噪声类**
   如 blur、sharpen、Gaussian noise、denoise 等。
   该类用于验证内容证据在频域扰动下的稳定性。

3. **光度与颜色类**
   如 brightness、contrast、saturation、gamma 等。
   该类用于验证语义内容链是否过度依赖低层外观。

4. **几何扰动类**
   如 resize、crop、translation、mild rotation。
   该类用于观察内容链在空间变换下的退化，但在主链鲁棒性集合中，应优先选择**轻到中等强度**的版本。

### （二）geometry rescue 候选攻击集合

该集合不是为了全面评估攻击鲁棒性，而是为了**更高概率触发“content-only failure but geometry possibly recoverable”场景**。因此，它应与主链鲁棒性集合分开统计、分开汇报。

该集合建议重点使用两类攻击组合。

1. **中等强度几何扰动**
   其特征是能够损伤内容链的局部对齐或判别 margin，但不能彻底破坏潜在几何结构。
   例如：中度 crop、中度 resize、轻中度 rotation、轻度仿射扰动。

2. **几何扰动 + 轻度外观扰动的复合攻击**
   其目的在于进一步压低内容链 margin，同时避免进入“所有证据都崩溃”的极端区。
   例如：resize + JPEG、crop + mild blur、rotation + mild noise。

这里必须强调一个原则：

> geometry rescue 候选集合的目标不是覆盖面最广，而是命中率最高。

### （三）severity 设计原则

对于每个 attack family，不建议只设单一强度，而应采用三档 severity：

1. **S1：轻度扰动**
   用于观察主链的早期退化边界。

2. **S2：中度扰动**
   用于形成最关键的 margin 边界区，是 geometry rescue 最有可能出现的区间。

3. **S3：重度扰动**
   用于观察主链崩解区，但不应作为 geometry rescue 的主要指望区。

因此，针对 geometry rescue，真正的高价值区间通常不是 S1，也不是 S3，而是 **S2 附近的中间区间**。

### （四）attack 层的主指标

attack 层主指标也应按层级组织。

1. **image-level**
   各 family × severity 下的内容链分数退化；
   ROC／TPR@low FPR 退化曲线；
   positive 与 negative 的分布漂移。

2. **event-level**
   各 family × severity 下的 event attestation pass rate；
   boundary event 比例；
   event failure type 组成。

3. **system-level**
   各 family × severity 下的 final decision 变化；
   false allow／false reject；
   release consistency。

4. **auxiliary-level**
   `payload` probe_overall；
   `payload` probe_by_attack_family；
   geometry eligible ratio；
   geometry rescue applied ratio。

### （五）attack 层直接服务的论文图表

建议最少固定以下结果产物。

1. **主图 3**：各 attack family × severity 下的内容链鲁棒性退化曲线；
2. **主表 2**：attack 条件下 image-level、event-level、system-level 核心指标汇总表；
3. **主图 4**：event-level 与 system-level 相对于 image-level 的稳健性提升图；
4. **附图／附表**：`payload` probe 与 geometry optional claim 的辅助统计结果。

## 六、event-level 结构层实验矩阵

event-level 是本文方法成立与否的关键桥梁，因此这一层不能只作为工程内部日志存在，而必须形成独立实验组。

### （一）event-level 的核心研究问题

这一层要回答的不是“event-level 指标好不好看”，而是：

1. event-level 是否比单图观测更稳定；
2. event-level 是否能把局部不一致组织成更可靠的局部证明；
3. event-level 是否为 system-level 提供了语义一致的输入单元；
4. boundary event 是否能够被合理识别，而不是随意划分。

### （二）event-level 的分组方式

建议 event-level 实验按三种子集展开。

1. **clean event 子集**
   用于建立无攻击条件下的局部证明基线。

2. **general attacked event 子集**
   用于观察常规攻击对 event attestation 的影响。

3. **boundary attacked event 子集**
   用于专门分析 content margin 边界事件，并为 geometry optional claim 提供候选池。

这三者中，第三类最关键，因为它直接决定 geometry 模块是否能在论文中获得正向存在价值。

### （三）event-level 的主指标

event-level 建议至少固定以下指标。

1. event attestation pass rate；
2. event margin mean／median；
3. event 内部观测一致性指标；
4. boundary event ratio；
5. content-failed subset ratio；
6. geometry eligible ratio；
7. geometry rescue applied ratio；
8. event-level false positive／false negative。

### （四）event-level 的关键对比

这一层最重要的不是绝对数值，而是三种对比关系。

1. **image-level vs event-level**
   验证事件级组织是否提升稳定性。

2. **clean event vs attacked event**
   验证攻击如何改变局部证明结构。

3. **general attacked event vs boundary event**
   验证 margin 边界协议是否真正筛出了高价值困难子集。

2. **主表 3**：clean／general attack／boundary attack 三类 event 子集的核心统计；

## 七、system-level 判定层实验矩阵

system-level 是整篇论文最终要证明的对象，因此这一层必须围绕“最终系统可用性”来定义，而不是仅仅作为上层汇总表。

### （一）system-level 的核心研究问题

system-level 需要回答：

1. 多事件聚合是否比单事件判定更稳定；
2. system final decision 是否在 clean／attack 条件下保持可靠；
3. release／signoff 语义是否能够与实际证据状态一致；
4. 辅助模块是否在不改变主链地位的前提下提供补充说明。

### （二）system-level 的分组方式

建议 system-level 至少包含以下三个实验维度。

1. **condition 维度**
   clean system；
   attacked system。

2. **event count 维度**
   小规模 event 聚合；
   中规模 event 聚合；
   大规模 event 聚合。
   该维度的作用是验证 system-level 是否具有随证据数量增加而稳定收敛的性质。

3. **decision pathway 维度**
   content-only 主路径；
   content + auxiliary explanation 路径。
   这里的“+ auxiliary”不是改变主决策，而是记录是否伴随 probe／geometry 附加解释。

### （三）system-level 的主指标

system-level 应固定以下主指标。

1. final decision accuracy；
2. false allow rate；
3. false reject rate；
4. blocking rate；
5. release_status consistency；
6. signoff consistency；
7. decision stability across event counts；
8. auxiliary explanation coverage。

### （四）system-level 的关键对比

system-level 最有价值的对比是：

1. **single-event decision vs multi-event system decision**；
2. **small-event aggregation vs large-event aggregation**；
3. **clean system vs attacked system**。

通过这三类对比，可以直接支撑“本文不是单点打分系统，而是系统级验证框架”这一核心结论。

## 八、辅助模块在实验矩阵中的位置

由于 `payload` 与 geometry 已经降级为辅助模块，因此它们在实验矩阵中的位置也必须严格受限。

### （一）`payload` 的实验位置

`payload` 不应拥有独立于主链的大规模主表主图组，而应作为：

1. clean 层中的一致性 probe 统计；
2. attack 层中的弱恢复退化统计；
3. event-level／system-level 中的附加解释字段。

推荐其主要汇报形式为：

1. `probe_overall`；
2. `probe_by_attack_family`；
3. `probe_effective_n_bits`；
4. `probe_support_rate`；
5. `probe_bit_accuracy`。

论文中对 `payload` 的结论应严格限制为：

> 它能够提供弱容量的一致性探针信号，但不承担主判定职责。

### （二）geometry 的实验位置

geometry 也不应被组织成与主链并列的大型主实验块，而应作为：

1. boundary event 子集上的 optional auxiliary claim；
2. system-level 辅助说明中的非阻断组件。

geometry 相关实验最核心的不是整体通过率，而是以下三个比例：

1. boundary event ratio；
2. geometry eligible ratio；
3. geometry rescue applied ratio。

因此，geometry 更适合以“边界子集分析图”和“事件流转表”的形式汇报，而不是主文中大面积展开。

## 九、正式 paper run 前应冻结的矩阵版本

为了便于后续执行，建议将实验矩阵正式冻结为以下三个版本。

### （一）Version A：paper pilot 矩阵

这是中规模验证矩阵，用于在正式大规模 run 前检查：

1. 图表字段是否齐全；
2. RQ1～RQ4 是否均有可落地结果；
3. geometry rescue 是否开始出现非零命中；
4. 各层统计方差是否可接受。

其特点是：

1. family 数量较少；
2. severity 档位保留；
3. event 数量中等；
4. system 聚合规模受控。

### （二）Version B：main paper 矩阵

这是正式论文主实验矩阵，用于生成主文图表。
它必须只保留**真正服务主结论**的攻击集合与统计字段，避免冗余。

其特点是：

1. clean 基线完整；
2. 主链鲁棒性 attack family 完整；
3. event-level 与 system-level 指标完整；
4. geometry 与 `payload` 作为辅助结果纳入，但不喧宾夺主。

### （三）Version C：appendix 扩展矩阵

这是附录扩展矩阵，用于容纳：

1. 更多 attack family；
2. 更细粒度 severity；
3. 更详细的 `payload` probe 分析；
4. 更全面的 geometry case study。

这样可以避免主文过载，同时保留充分证据。
