# PW 流程编排（当前实现）

本文档描述当前仓库中 `paper_workflow` 的真实分段职责、落盘工件边界与推荐执行顺序。它以当前实现为准。

当前真实 notebook 文件为：

```text
PW00_Paper_Eval_Family_Manifest.ipynb
PW01_Source_Event_Shards.ipynb
PW02_Source_Merge_And_Global_Thresholds.ipynb
PW03_Attack_Event_Shards.ipynb
PW04_Attack_Merge_And_Metrics - 1.Prepare.ipynb
PW04_Attack_Merge_And_Metrics - 2.Quality.ipynb
PW04_Attack_Merge_And_Metrics - 3.Finalize.ipynb
PW05_Release_And_Signoff.ipynb
```

---

## 一、总设计原则

### 1. Colab 会话隔离

所有 notebook 都必须视为在彼此隔离的会话中运行。

这意味着：

1. `PW00` 到 `PW05` 之间不能依赖进程内状态、内存对象、临时缓存或 `/tmp` 文件。
2. 下游阶段只能读取上游已经明确落盘到 family 目录下的文件。
3. 后续阶段不能依据“上一个 notebook 当前环境里能 import 某个库”来作出方法闭环判断。

### 2. 工件优先于运行时状态

跨 notebook 传递的事实，必须体现为可审计的文件，而不是运行时布尔值。

当前设计中最关键的两个边界是：

1. `PW02` 只负责落盘 clean quality pair manifest，不再负责正式 clean LPIPS/CLIP 聚合结果。
2. `PW05` 只依据 `PW04` 已落盘的质量工件与指标工件做 signoff，不再根据当前会话能否导入 `lpips` 或 `open_clip` 判断通过与否。

### 3. Canonical 与 analysis-only 分层

当前 `PW` 设计明确区分两类输出：

1. canonical outputs：论文主线与 release 必需工件。
2. analysis-only outputs：扩展分析、补充导出、辅助一致性检查与论文附加材料。

`PW04 summary` 与 `PW05 release manifest` 都会显式记录这两类路径，防止后续阶段绕过主线工件链。

## 三、各阶段职责

### （一）PW00：冻结 family 身份与任务网格

入口：`PW00_Paper_Eval_Family_Manifest.ipynb`

核心脚本：`pw00_build_family_manifest.py`

职责：

1. 冻结 `family_id` 与基础运行身份。
2. 读取 prompt 列表、seed 规划与 attack protocol。
3. 生成 source event grid 与 attack event grid。
4. 生成 source shard plan 与 attack shard plan。
5. 固化后续阶段必须共享的 manifest。

典型输出：

1. `manifests/paper_eval_family_manifest.json`
2. `manifests/source_event_grid.jsonl`
3. `manifests/source_shard_plan.json`
4. `manifests/attack_event_grid.jsonl`
5. `manifests/attack_shard_plan.json`

不负责：

1. 图像生成
2. detect
3. 阈值构建
4. 指标聚合
5. release 打包

### （二）PW01：source shard 级生成与 detect

入口：`PW01_Source_Event_Shards.ipynb`

核心脚本：`pw01_run_source_event_shard.py`

辅助脚本：

1. `pw01_run_source_event_shard_worker.py`
2. `pw01_stage_runtime_helpers.py`

职责：

1. 按 `source_shard_plan.json` 读取本 shard 的 event 集合。
2. 运行 `positive_source` 或 `clean_negative` 两类 source 事件。
3. 为每个 event 写出图像、embed/detect 记录与 event 级工件。
4. 输出 shard manifest 与 records 汇总。

当前基本单位是 event，而不是旧式 `prompt_{index}` 目录。

每个 event 至少应绑定：

1. `event_id`
2. `prompt_index`
3. `prompt_text`
4. `seed`
5. `sample_role`
6. `source_image_path`
7. `detect_record_path`
8. `embed_record_path`（仅正样本）
9. `runtime_config_path`

不负责：

1. 全局阈值
2. 全局 evaluate
3. 论文表格
4. 全局 quality 汇总

### （三）PW02：source 合并、阈值、clean evaluate 与 clean pair manifest

入口：`PW02_Source_Merge_And_Global_Thresholds.ipynb`

核心脚本：`pw02_merge_source_event_shards.py`

扩展脚本：`pw02_metrics_extensions.py`

职责：

1. 校验全部 source shards 是否完成且 family 身份一致。
2. 合并 `positive_source` 与 `clean_negative` 两个 pool。
3. 依据 calibration/eval 规则生成全局阈值。
4. 输出 clean 主线 evaluate 与 finalize manifest。
5. 输出 operating metrics、payload clean summary 等补充分析工件。
6. 负责把 clean 图像对、prompt 文本与 event 绑定整理成 pair manifest，输出 `clean_quality_pair_manifest.json`，为后续 `PW04` 统一 quality 计算提供 clean pairs。

典型输出：

1. `exports/pw02/paper_source_finalize_manifest.json`
2. `exports/pw02/thresholds/content/thresholds.json`
3. `exports/pw02/thresholds/attestation/thresholds.json`
4. `exports/pw02/formal_final_decision_metrics.json`
5. `exports/pw02/derived_system_union_metrics.json`
6. `exports/pw02/quality/clean_quality_pair_manifest.json`
7. `exports/pw02/payload/payload_clean_summary.json`
8. `runtime_state/pw02_summary.json`

`pw02_summary.json` 中对下游最重要的字段包括：

1. `paper_source_finalize_manifest_path`
2. `clean_pair_artifacts_dir`
3. `clean_quality_pair_manifest_path`
4. `payload_clean_summary_path`
5. `analysis_only_artifact_paths`

不负责：

1. attacked image 指标
2. clean/attack 统一 quality 聚合
3. 论文 tradeoff 导出
4. release signoff

### （四）PW03：attack shard 级 attack 与 detect

入口：`PW03_Attack_Event_Shards.ipynb`

核心脚本：`pw03_run_attack_event_shard.py`

职责：

1. 从 `PW02` finalized positive source pool 读取 parent events。
2. 按 attack protocol 生成 attacked images。
3. 对 attacked images 执行 detect。
4. 写出 attacked event 记录与 shard manifest。

当前攻击事件的核心绑定是：

1. `parent_event_id`
2. `attack_event_id`
3. `attack_family`
4. `attack_condition_key`
5. `attack_config_name`
6. `attacked_image_path`
7. `detect_record_path`

不负责：

1. 全局 attack merge
2. 论文表格
3. 全局 quality 聚合
4. tail estimation

### （五）PW04：attack 合并、主指标、统一 quality、论文导出与可选 tail

入口 notebooks：

1. `PW04_Attack_Merge_And_Metrics - 1.Prepare.ipynb`
2. `PW04_Attack_Merge_And_Metrics - 2.Quality.ipynb`
3. `PW04_Attack_Merge_And_Metrics - 3.Finalize.ipynb`

Python 包装脚本：`PW04_Attack_Merge_And_Metrics.py`

主调度脚本：`pw04_merge_attack_event_shards.py`

相关脚本：

1. `pw_quality_metrics.py`
2. `pw04_prepare_quality_pairs.py`
3. `pw04_run_quality_shard.py`
4. `pw04_finalize_quality_metrics.py`
5. `pw04_paper_exports.py`
6. `pw04_metrics_extensions.py`
7. `pw04_notebook_common.py`
8. `pw_local_runtime.py`

`PW04` 当前是整条 paper 路径中最重的汇总阶段，职责分为四层。

当前真实入口不再是单本 `PW04` notebook 包打天下，而是 3 个 notebook 对同一个主调度脚本的显式分段包装：

1. Prepare notebook 内部固定 `PW04_MODE=prepare`。
2. Quality notebook 内部固定 `PW04_MODE=quality_shard`。
3. Finalize notebook 内部固定 `PW04_MODE=finalize`。
4. 三者都调用 `PW04_Attack_Merge_And_Metrics.py`，再由它转发到 `run_pw04_merge_attack_event_shards(...)`。

#### 1. attack shards 合并与 canonical attack metrics

`pw04_merge_attack_event_shards.py` 会：

1. 合并全部 attack shard 结果。
2. 生成 attacked positive pool 与 attacked negative pool。
3. 生成 formal attack final decision / attestation / union / negative metrics。
4. 在 prepare 阶段先写出 attack-side canonical 工件；最终 `runtime_state/pw04_summary.json` 只在 finalize 阶段落盘。

#### 2. clean 与 attack 统一 quality 流水线

这是当前设计相较旧版最重要的结构变化。

当前 `PW04` 内部显式拆成三段：

1. `pw04_prepare_quality_pairs.py`
2. `pw04_run_quality_shard.py`
3. `pw04_finalize_quality_metrics.py`

具体含义如下：

##### a. prepare 阶段

`run_pw04_prepare_quality_pairs(...)`：

1. 读取 `PW02 clean_quality_pair_manifest.json`
2. 读取 attacked positive rows
3. 统一生成 clean pairs 与 attack pairs
4. 生成 `exports/pw04/quality/quality_pair_plan.json`
5. 对 clean/attack pairs 做 shard 分配

##### b. shard 阶段

`run_pw04_quality_shard(...)`：

1. 读取 `quality_pair_plan.json`
2. 只处理一个 `quality_shard_index` 对应的 clean/attack 子集
3. 调用 `pw_quality_metrics.py` 计算 shard-local 质量摘要
4. 写出 `exports/pw04/quality/shards/quality_shard_XXXX.json`

这个阶段的边界必须保持明确：

1. 只写 shard 局部 quality 输出。
2. 不写最终 clean/attack quality JSON。
3. 不写 tables。
4. 不写 figures。

##### c. finalize 阶段

`run_pw04_finalize_quality_metrics(...)`：

1. 合并全部 quality shard 输出。
2. 校验 pair id 与计划一致。
3. 写出 `exports/pw04/metrics/clean_quality_metrics.json`
4. 写出 `exports/pw04/metrics/attack_quality_metrics.json`
5. 写出 `exports/pw04/quality/quality_finalize_manifest.json`

#### 2.1 当前 3 个 PW04 notebooks 的参数暴露面

当前 notebook 层已经按阶段把用户可见参数收敛为最小集合：

1. Prepare notebook 只暴露 `FAMILY_ID`、`QUALITY_SHARD_COUNT`、`FORCE_RERUN`、`ENABLE_TAIL_ESTIMATION`。
2. Quality notebook 只暴露 `FAMILY_ID`、`QUALITY_SHARD_INDEX`、`FORCE_RERUN`、`QUALITY_DEVICE_OVERRIDE`、`QUALITY_LPIPS_BATCH_SIZE`、`QUALITY_CLIP_BATCH_SIZE`。
3. Finalize notebook 只暴露 `FAMILY_ID`、`FORCE_RERUN`。

其中 `Quality` notebook 的 runtime 选择是 notebook-only 编排逻辑，当前真实规则为：

1. 优先级是 notebook override > environment > device default。
2. 默认设备路径只保留 `CPU` 与正常 `CUDA` 两类，不再保留“低显存 CUDA”分支。
3. 当选择 `CPU` 时，`LPIPS` 与 `CLIP` 都回退到 `DEFAULT_QUALITY_BATCH_SIZE`。
4. 当选择 `CUDA` 时，notebook 默认 batch 为 `LPIPS=128`、`CLIP=256`。

#### 3. 论文表格与 supplemental metrics

`PW04` 还负责把 canonical metrics 与 unified quality 结果转换成论文与分析导出：

1. 主表 `main_metrics_summary.csv`
2. `attack_family_summary_paper.csv`
3. `attack_condition_summary_paper.csv`
4. `clean_imperceptibility.json`
5. `attack_distortion.json`
6. `quality_robustness_tradeoff.csv`
7. `system_final_auxiliary_attack_summary.json`
8. geometry、payload、wrong-event 等补充导出

这里的关键是：

1. clean imperceptibility 现在读 `PW04 clean_quality_metrics.json`
2. attack distortion 现在读 `PW04 attack_quality_metrics.json`
3. tradeoff 表不再引用旧的 `PW02 quality summary CSV/JSON`

#### 4. 可选 tail estimation

`PW04` 仍然可以生成 tail estimation 相关输出，但它们与 empirical clean FPR 是分离命名、分离导出的，不与主指标混写。

典型输出包括：

1. `estimated_tail_fpr_1e4.json`
2. `estimated_tail_fpr_1e5.json`
3. `tail_fit_diagnostics.json`
4. `tail_fit_stability_summary.json`

#### 5. PW04 summary 的关键字段

当前 `runtime_state/pw04_summary.json` 中与下游最相关的字段包括：

1. `clean_quality_metrics_path`
2. `attack_quality_metrics_path`
3. `quality_pair_plan_path`
4. `quality_shard_paths`
5. `quality_finalize_manifest_path`
6. `quality_shard_count`
7. `paper_scope_registry_path`
8. `analysis_only_artifact_paths`
9. `analysis_only_artifact_annotations`

### （六）PW05：artifact-only release 与 signoff

入口：`PW05_Release_And_Signoff.ipynb`

核心脚本：`pw05_release_signoff.py`

职责：

1. 读取 `PW04 summary` 与 canonical source artifacts。
2. 生成 `formal_run_readiness_report.json`。
3. 生成 `signoff_report.json`。
4. 生成 `release_manifest.json`。
5. 生成 `workflow_summary.json`、`run_closure.json`、`stage_manifest.json`。
6. 构造 package staging 目录并输出最终 zip 包。

当前 `PW05` 的质量闭环语义是：

1. `quality_clean` readiness 读取 `PW04 clean_quality_metrics.json`
2. `quality_attack` readiness 读取 `PW04 attack_quality_metrics.json`

典型输出：

1. `exports/pw05/formal_run_readiness_report.json`
2. `exports/pw05/signoff_report.json`
3. `exports/pw05/release_manifest.json`
4. `exports/pw05/workflow_summary.json`
5. `exports/pw05/run_closure.json`
6. `exports/pw05/stage_manifest.json`
7. `exports/pw05/package_manifest.json`
8. `exports/pw05/package_index.json`
9. `exports/pw05/*.zip`
10. `runtime_state/pw05_summary.json`

## 四、脚本职责映射

### 1. 通用脚本

#### `pw_common.py`

负责共用的 family / path / manifest / shard 计划工具逻辑，不承载阶段业务编排。

#### `pw_quality_metrics.py`

负责按 image pairs 计算：

1. `PSNR`
2. `SSIM`
3. `LPIPS`
4. `CLIP text similarity`
5. `quality_runtime`
6. `prompt_text_coverage_status`
7. `quality_readiness_status`

它是 `PW04` unified quality 流水线的底层计算器，不直接承担 release 决策。

### 2. notebook-only 辅助脚本

#### `pw04_notebook_common.py`

负责 3 个 `PW04` notebook 共用的 notebook-only 编排辅助，包括：

1. `PW04` CLI 命令构造。
2. stage output 路径解析。
3. `quality_shard` notebook 的 runtime 摘要与子进程环境绑定。
4. GPU peak summary 与 notebook 结果摘要读取。

它不改写 `PW04` 主脚本语义，只把同一套 `prepare / quality_shard / finalize` 主链语义映射到 3 个 notebook。

#### `pw_local_runtime.py`

负责 notebook 场景下的 local runtime 准备与归档，不承载新的 `PW` 阶段业务语义。

### 3. 阶段脚本

#### `pw00_build_family_manifest.py`

负责 family manifest、event grid 与 shard 计划。

#### `pw01_run_source_event_shard.py`

负责 source shard 级执行与 event 级落盘。

#### `pw02_merge_source_event_shards.py`

负责 source merge、threshold、clean evaluate、finalize manifest 与 clean pair manifest。

#### `pw02_metrics_extensions.py`

负责 `PW02` operating metrics、payload clean summary 与 analysis-only 绑定整理，不负责正式 clean quality 聚合。

#### `pw03_run_attack_event_shard.py`

负责 attack shard 级攻击与 detect。

#### `pw04_prepare_quality_pairs.py`

负责从 `PW02 clean pair manifest` 与 attacked positive rows 生成统一 quality plan。

#### `pw04_run_quality_shard.py`

负责一个 quality shard 的 clean/attack 局部计算。

#### `pw04_finalize_quality_metrics.py`

负责合并 quality shard，生成 canonical clean/attack quality 工件。

#### `pw04_merge_attack_event_shards.py`

负责 attack merge 主线、quality 主调度与 `PW04 summary` 写出。

#### `pw04_paper_exports.py`

负责主论文表格、主指标表与 figures 路径整理。

#### `pw04_metrics_extensions.py`

负责 supplemental metrics、tradeoff、clean imperceptibility、attack distortion、geometry/payload/wrong-event 等扩展导出。

#### `pw05_release_signoff.py`

负责最终 artifact-only release 与 signoff。

## 五、推荐运行顺序

### 步骤 1：单会话运行 PW00

作用：

1. 冻结 family 身份
2. 生成 event grid
3. 生成 shard plans

### 步骤 2：多会话并行运行 PW01

推荐参数维度：

1. `sample_role = positive_source | clean_negative`
2. `shard_index = 0..K-1`
3. `shard_count = K`

这个阶段允许多会话并行，但每个会话只负责本 shard。

### 步骤 3：单会话运行 PW02

作用：

1. 合并 source shards
2. 构建 thresholds
3. 生成 clean evaluate
4. 生成 finalize manifest
5. 生成 clean quality pair manifest

### 步骤 4：多会话并行运行 PW03

推荐参数维度：

1. `attack_shard_index = 0..A-1`
2. `attack_shard_count = A`

每个会话只负责本 attack shard。

### 步骤 5：单会话运行 PW04 Prepare

入口 notebook：`PW04_Attack_Merge_And_Metrics - 1.Prepare.ipynb`

作用：

1. 合并全部 attack shards。
2. 生成 canonical attack metrics 与相关 pool / manifest。
3. 生成 `quality_pair_plan.json`。
4. 冻结 `prepare manifest`、预期 quality shard 路径与 tail estimation 意图。

### 步骤 6：多会话并行运行 PW04 Quality

入口 notebook：`PW04_Attack_Merge_And_Metrics - 2.Quality.ipynb`

推荐参数维度：

1. `QUALITY_SHARD_INDEX = 0..Q-1`

这个阶段允许多会话并行，但每个会话只负责一个 `quality shard`，并且只能写 `exports/pw04/quality/shards/quality_shard_XXXX.json` 这类 shard 局部质量工件。

### 步骤 7：单会话运行 PW04 Finalize

入口 notebook：`PW04_Attack_Merge_And_Metrics - 3.Finalize.ipynb`

作用：

1. 校验全部计划内 `quality shard` 已完成。
2. 合并 quality shard，生成 canonical clean / attack quality 工件。
3. 生成 paper tables、supplemental metrics 与可选 tail estimation 导出。
4. 写出最终 `runtime_state/pw04_summary.json`。

### 步骤 8：单会话运行 PW05

作用：

1. 基于已落盘工件生成 readiness report
2. 生成 signoff 与 release manifest
3. 打包 source + artifacts
4. 输出最终 release zip

## 六、当前边界约束

为避免后续演化再次退回旧设计，当前文档明确以下约束：

1. `PW02` 不能重新承担 formal clean LPIPS/CLIP 聚合责任。
2. `PW04` 是 clean quality 与 attack quality 的统一 canonical 拥有者。
3. `pw04_run_quality_shard.py` 只能写 shard 级质量结果，不能写最终表格或图形。
4. `PW05` 不能依据当前会话的 `lpips` / `open_clip` import 成功与否决定 signoff。
5. 所有 notebook 下游只能读取已经落盘的工件，不允许依赖上游 runtime 暂存状态。
